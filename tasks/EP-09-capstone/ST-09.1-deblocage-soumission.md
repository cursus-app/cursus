## Contexte business

Le capstone est l'aboutissement du cursus. Il se débloque seulement quand les N-2 modules précédents sont validés. Soumissions intermédiaires possibles avec checks légers, soumission finale qui déclenche le harnais complet et bascule en attente de soutenance.

Valeur livrée : capstone structuré, déblocage automatique, possibilité de tester son work avec checks light avant la finale.

## Description fonctionnelle

Page dédiée capstone. Soumissions intermédiaires (checks repo + README + structure) sans bascule. Soumission finale : harnais complet → attente_soutenance. Date limite avec alerte automatique si dépassée.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Déblocage automatique**

```gherkin
Given un stagiaire a validé tous les modules N-2 précédents
When il consulte son parcours
Then le capstone est débloqué et visible
```

**Scénario 2 — Soumission intermédiaire**

```gherkin
Given le capstone est débloqué
When il fait une soumission intermédiaire
Then les checks "légers" tournent (repo existe, README présent, structure)
And le résultat est affiché sans bascule de statut
```

**Scénario 3 — Soumission finale**

```gherkin
Given il fait la soumission finale
When le harnais complet s'exécute et passe
Then le statut bascule à "en_attente_soutenance"
```

**Scénario 4 — Date limite dépassée**

```gherkin
Given la date limite arrive sans soumission finale
Then le stagiaire passe en "en_retard"
And une alerte est créée
```

## Cas limites à traiter

- **Date limite passée mais soumission finale validée** : OK avec mention "soumis en retard".
- **5 soumissions intermédiaires consommées** : message "Prêt pour la soumission finale ?".
- **Capstone soumis avant déblocage** : refus avec message clair.
- **Modification du cursus pendant capstone** : pas d'impact (version figée).

## Considérations sécurité
