## Contexte business

Beaucoup de cohortes vivent déjà sur Slack (canal cohorte, DMs formateurs). Aujourd'hui : un stagiaire fait sa soumission, le formateur la voit dans Cursus 12 h plus tard. Avec une intégration Slack, le canal de la cohorte reçoit en quasi-temps réel les événements clés : nouvelle soumission, alerte stagiaire, fin de module. Cohorte restée cool dans Slack, sans changement d'habitude.

Business value : c'est un Differentiator d'engagement — réduit la friction d'adoption. Critère décisif vs concurrents.

## Description fonctionnelle

- Connecteur Slack via Webhook URL (entrant Slack) configurable par cohorte (UI ST-22.6).
- Événements poussés : `submission.created`, `submission.passed`, `submission.failed`, `alert.created`, `cohort.module_completed`, `cohort.capstone_started`.
- Format de message : Block Kit Slack (cards riches avec titre, contexte, lien vers Cursus, icône).
- Throttling : max 1 message / sec / canal pour éviter spam.
- Retry exponentiel si Slack 5xx.
- Pause auto si Slack répond 429 (rate limit Slack).
- Logs systématiques (audit + obs).

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Soumission notifiée
Given la cohorte X est configurée avec un webhook Slack
When le stagiaire Y soumet le livrable M
Then un message Block Kit apparaît sur le canal Slack
And il contient : titre, nom du stagiaire (display_name), nom du module, lien vers le rapport Cursus
And le message arrive en < 5 s
```

```gherkin
Scénario 2 — Slack indisponible
Given le webhook Slack répond 500
When Cursus tente d'envoyer le message
Then retry exponentiel (3 tentatives sur 60 s)
And après 3 échecs, un log warn est créé
And la notification est marquée "failed" dans la table audit
```

```gherkin
Scénario 3 — Rate limit respecté
Given 100 events sont déclenchés en 10 s
When ils sont pushés vers Slack
Then le throttling cap à 1/s
And aucun message n'est perdu (queué dans Inngest)
```

## Cas limites à traiter

- **Webhook URL invalide** : vérification à la configuration (test message), refus si échec.
- **Cohorte désactive l'intégration** : stop des futurs envois (kill switch par cohorte).
- **Stagiaire sensible** : permettre opt-out par stagiaire ("Ne pas notifier mes soumissions sur Slack").
- **Slack répond 429** : pause 60 s puis ré-essayer.
- **PII** : ne jamais inclure emails dans le message (juste display_name).
- **Locale** : message dans la langue de la cohorte (FR/EN).

## Non-goals

- Pas de Slack OAuth app au MVP (juste webhook entrant), simple à configurer.
- Pas de commandes Slack (`/cursus status`) au MVP.
- Pas de threading des messages.
- Pas de bot bidirectionnel (Slack → Cursus).

## Tests à écrire

- **Unit Vitest** : `formatBlockKit` produit JSON valide selon event type, `verifySlackWebhook` gère 200 / 4xx / 5xx.
- **Integration** : event `submission.created` → job queué → Slack mock recoit le payload attendu.
- **Retry** : Slack 500 × 3 → marquage failed.
- **Rate limit** : 100 events 10 s → durée d'envoi réel \~ 100 s (cap 1/s).
- **Security** : webhook URL doit être HTTPS, hostname Slack uniquement (regex anti-SSRF).
- **A11y / i18n** : message en FR + EN.

## Observabilité

- **Logs** : event `integrations.slack.sent` (info) `{ cohort_id, event_type, status, latency_ms }`.
- **Métriques** : compteur `cursus.integrations.slack.sent.total` (par event_type), histogram latency.
- **Alerte** : taux d'échec > 10 % sur 1 h.
- **Dashboard** : page admin `/admin/integrations` avec health par cohorte.

## Considérations sécurité

- Webhook URL stockée chiffrée en DB (Supabase secrets ou colonne encryptée).
- Validation hostname : `*.slack.com` uniquement (évite SSRF).
- Rate limit serveur (1 message / s / canal).
- RLS strict sur table `integrations`.
- Audit log immuable.
- Opt-out stagiaire respecté.

## Considérations accessibilité

- Le message Slack est consommé par Slack — a11y gérée par eux.
- Cohérence visuelle (icône, couleur Cursus) sans dépendre uniquement de la couleur.

## Considérations performance

- Job async (pas blocage de l'UI Cursus).
- Throttling par canal (1/s) pour respecter rate limit Slack.
- Cible : message dans Slack en < 5 s p95 après event Cursus.
- Batch impossibles (Slack webhook = 1 message / call).
