## Contexte business

Symétrique à ST-22.1 (Slack), Discord est la deuxième grande plateforme communautaire (de loin la plus populaire chez les jeunes développeurs). Sans ce connecteur, beaucoup de cohortes nous rejetteraient ("on est sur Discord, pas Slack"). Cette story livre la même valeur via le système de webhooks Discord.

Business value : ouverture du produit au principal canal communautaire dev. Marqueur Differentiator d'acceptabilité.

## Description fonctionnelle

- Connecteur Discord via Webhook URL (entrant Discord) configurable par cohorte.
- Mêmes events que Slack (submission, alerts, module/capstone).
- Format : Discord Embeds (titre, description, color, fields, footer, timestamp). Plus simple que Block Kit, même logique.
- Throttling : Discord supporte \~5 messages / 2s ; cap 2/s pour sécurité.
- Retry exponentiel + audit log.
- Réutilise l'infrastructure de ST-22.1 (`integrations` table, job pattern).

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Embed Discord propre
Given une cohorte avec webhook Discord configuré
When un event submission.passed arrive
Then un embed Discord arrive sur le canal
And il a un titre, une couleur verte (success), un author Cursus, un footer timestamp
And un bouton "Voir le rapport" en lien externe
```

```gherkin
Scénario 2 — Discord 429 rate limit
Given Discord retourne 429 avec retry_after: 5
When le job reçoit la réponse
Then il pause 5 s et ré-essaie
And les jobs en file d'attente attendent aussi
```

```gherkin
Scénario 3 — Configuration test
Given l'admin configure une URL Discord invalide
When il clique "Tester l'intégration"
Then Cursus tente un POST de vérification
And retourne erreur claire "URL invalide ou canal inaccessible"
```

## Cas limites à traiter

- **Webhook supprimé du côté Discord** : POST renvoie 404 → marquer intégration `disabled_at` + notifier admin.
- **Embed > 6000 caractères** : tronquer + lien "voir plus" sur Cursus.
- **Plus de 10 embeds dans un message** : Discord limite à 10 — splitter.
- **Mention @everyone non désirée** : sanitize d'office, jamais inclure `@everyone` / `@here` / `<@&role>` dans le contenu.
- **Couleur color-blind safe** : utiliser palette accessible.

## Sous-tâches techniques

- **TT-22.2.1** — Service `server/integrations/discord.ts` : `formatEmbed(event, locale)`, `postToDiscord(url, payload)`.
- **TT-22.2.2** — Job Inngest `integrations.discord.send` (retry, throttling 2/s, respect retry_after).
- **TT-22.2.3** — Helper `verifyDiscordWebhook(url)` (validation hostname `discord.com` ou `discordapp.com`).
- **TT-22.2.4** — Helper `sanitizeMentions(text)` qui remplace `@everyone` → `everyone` (sans `@`).
- **TT-22.2.5** — Mapping color codes : success=green, warning=amber, danger=red, info=blue.
- **TT-22.2.6** — Tests intégration mock Discord.
- **TT-22.2.7** — Doc `docs/integrations/discord.md` (comment créer un webhook entrant).

## Dépendances

- **Bloqué par** : ST-22.1 (infrastructure posée), ST-12.1 (event bus), ST-22.6 (UI config).
- **Bloque** : aucune.

## Non-goals

- Pas d'OAuth app Discord (juste webhook).
- Pas de slash commands.
- Pas de bot full Discord.
- Pas d'interactions Discord (boutons retour).

## Tests à écrire

- **Unit Vitest** : `formatEmbed` génère structure valide, `sanitizeMentions` nettoie correctement.
- **Integration** : event → embed Discord mocké → payload validé par schema Discord.
- **Retry 429** : respect du `retry_after`.
- **Security** : URL non-Discord rejetée (anti-SSRF).
- **A11y** : couleurs avec symbole (pas seulement couleur dans embed).

## Observabilité

- **Logs** : event `integrations.discord.sent` (info) avec mesures.
- **Métriques** : compteur `cursus.integrations.discord.sent.total`, taux d'échec.
- **Alerte** : 404 répété → disable + admin notification.

## Considérations sécurité

- URL chiffrée en DB.
- Validation hostname strict (`discord.com`).
- Pas de leak PII.
- Sanitization @mentions.
- RLS strict.

## Considérations accessibilité

- Discord gère le rendu — nos embeds suivent leurs règles.
- Couleur avec icône associée (success = check, danger = x).

## Considérations performance

- Job async.
- Cap 2/s.
- Cible message < 5 s p95.
