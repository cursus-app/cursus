## Contexte business

Les connecteurs spécifiques (Slack, Discord, Teams) ne couvrent que 3 plateformes. Pour pouvoir intégrer avec n'importe quel système tiers (PMS interne, CRM, Notion, Airtable…), il faut un mécanisme générique : les webhooks sortants. C'est le « couteau suisse » des intégrations.

Business value : couvre les besoins long tail. Différenciateur fort ("intégrable partout").

## Description fonctionnelle

- L'admin (ou un formateur autorisé) déclare un webhook : `name`, `url`, `events_subscribed: [string]`, `secret` (généré par Cursus), `active: bool`.
- Pour chaque event abonné déclenché dans Cursus, Cursus POST le payload JSON à l'URL avec un header `X-Cursus-Signature: sha256=<HMAC>` (HMAC du body avec le secret).
- Le consommateur vérifie la signature (assurance que c'est bien Cursus + body non altéré).
- Retry exponentiel (1m, 5m, 30m, 2h, 1j) sur 5xx.
- Stockage des deliveries (succès / échec) avec replay possible depuis l'UI.
- Format payload stable, documenté.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Webhook reçu et vérifié
Given un webhook actif abonné à "submission.created"
When une soumission est créée
Then une requête POST est envoyée à l'URL avec header X-Cursus-Signature
And le payload est validé par le consommateur via HMAC
And le delivery est marqué success dans Cursus
```

```gherkin
Scénario 2 — Replay manuel
Given un delivery a échoué 5 fois
When l'admin clique "Replay"
Then une nouvelle tentative est faite
And le delivery_attempt_id est nouveau
```

```gherkin
Scénario 3 — Signature invalide côté client
Given le client reçoit le payload
When il calcule HMAC avec le secret
Then la signature match exactement
And toute altération du body change la signature (test négatif)
```

## Cas limites à traiter

- **URL défaillante répété** : après 5 retry, marquer webhook `unhealthy`, alerter admin.
- **Timeout** : 10 s max par tentative.
- **Payload très large** : limite 256 KB (sinon truncate + lien).
- **Boucle d'événements** (event A → webhook → crée event B → webhook…) : event versioning et dedup par `event_id`.
- **Rate limit cible** : respect optionnel `Retry-After` du consommateur.
- **Secret compromis** : rotation depuis UI (génére nouveau secret, ancien valide 24 h pour rolling).

## Non-goals

- Pas de webhook entrant (Cursus consume) au MVP.
- Pas de transformer / mapper de payload.
- Pas de filtre fin par contenu (juste par event_type).
- Pas de garantie ordering strict (deliveries indépendantes).

## Tests à écrire

- **Unit Vitest** : `signBody(body, secret)` génère HMAC stable, `verifyHmac` valide / rejette correctement.
- **Integration** : event → webhook actif → mock endpoint reçoit signed POST.
- **Retry** : endpoint 500 → 5 tentatives selon exponential backoff.
- **Replay** : marqué unhealthy puis replay → redelivery.
- **Security** : alteration body → signature mismatch.
- **A11y** : UI webhooks (table, modal) accessible.

## Observabilité

- **Logs** : `webhook.delivered` (info), `webhook.failed` (warn), `webhook.deadletter` (error).
- **Métriques** : compteur `cursus.webhooks.delivered.total`, gauge `cursus.webhooks.unhealthy.total`.
- **Alerte** : webhook unhealthy > 24 h → admin email.
- **Dashboard** : taux de delivery par webhook.

## Considérations sécurité

- HMAC SHA-256 obligatoire (jamais payload sans signature).
- Secret chiffré en DB.
- Validation URL HTTPS only.
- Anti-SSRF : refuser localhost / IPs internes (10.x, 172.16-31.x, 192.168.x).
- RLS strict.
- Rate limit création (max 10 webhooks / org).
- Audit log.

## Considérations accessibilité

- UI webhook table : sortable, scrolling navigable.
- Modal de création : form a11y standard.

## Considérations performance

- Dispatch async par event.
- Jobs concurrent par webhook (concurrency-limité 10).
- Cible delivery p95 < 3 s.
- Storage deliveries : 30 jours, purge auto.
