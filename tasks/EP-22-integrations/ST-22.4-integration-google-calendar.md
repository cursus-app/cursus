## Contexte business

Les soutenances de capstone (ST-09.3) doivent être planifiées entre stagiaires, formateurs et jurys externes. Aujourd'hui : email + Doodle. Avec Google Calendar, Cursus crée directement les événements (avec Meet link, invités, rappels) dans les agendas concernés. Réduit la friction de planification.

Business value : feature Differentiator concrète de la phase capstone. Accès amélioré à la rituel "jour de soutenance".

## Description fonctionnelle

- OAuth 2.0 Google avec scope `https://www.googleapis.com/auth/calendar.events`.
- Le formateur connecte son compte Google une fois (table `users.google_calendar_token`).
- UI "Planifier soutenance" sur la page capstone : choix date/heure, durée (30 min default), invités (stagiaire + jurys), description (générée par template).
- Cursus appelle `events.insert` Google Calendar API → événement créé dans le calendrier formateur, avec Meet link auto-généré + invités notifiés.
- Stockage du `event_id` Google côté Cursus pour synchronisation (update / cancel).
- Refresh token Google : refresh automatique.

## Cas limites à traiter

- **Token expiré** : refresh automatique avec refresh_token.
- **Refresh token révoqué** : forcer re-connexion, message clair.
- **Conflit de créneau** : pas de check côté Cursus, Google notifie l'invité.
- **Invité sans Google Account** : Google gère (invitation par email).
- **Quotas Google API** : suivre `quotaUser` + retry exponentiel.
- **Locale / timezone** : timezone du formateur (DST aware).
- **Suppression du compte Google** : détecter et alerter.

## Sous-tâches techniques

- **TT-22.4.1** — OAuth flow : route `GET /api/auth/google/connect` et callback `/api/auth/google/callback`.
- **TT-22.4.2** — Stockage tokens chiffrés en DB (`users.google_calendar_access_token`, `..refresh_token`, expires_at).
- **TT-22.4.3** — Service `server/integrations/google-calendar.ts` : `createEvent`, `updateEvent`, `deleteEvent`, `refreshTokenIfNeeded`.
- **TT-22.4.4** — Endpoint `POST /api/capstone/:id/schedule` (date, durée, invités).
- **TT-22.4.5** — UI sur page capstone : modal de planification, lien Meet visible après création.
- **TT-22.4.6** — Job Inngest `gcal.event.sync` (en cas de modification côté Google, webhook push notif).
- **TT-22.4.7** — Doc `docs/integrations/google-calendar.md`.

## Non-goals

- Pas de support Microsoft Outlook au MVP.
- Pas de planification automatique (juste manuelle).
- Pas de synchronisation bidirectionnelle complète (event modifié sur Google se sync mais pas update temps réel Cursus).
- Pas de gestion de disponibilités croisees.

## Tests à écrire

- **Unit Vitest** : `refreshTokenIfNeeded` logique d'expiration, payload events.insert valide.
- **Integration** : OAuth flow complet (mock Google).
- **E2E** : formateur connecte, planifie, voit le Meet link.
- **Security** : tokens chiffrés, scope minimal demandé.
- **Token expiré** : refresh transparent.

## Observabilité

- **Logs** : `gcal.event.created` (info), `gcal.token.refreshed` (debug), `gcal.api.error` (warn).
- **Métriques** : compteur events créés, taux d'échec.
- **Alerte** : > 5 % d'échec API → enquête.

## Considérations sécurité

- OAuth scope minimal (`calendar.events`).
- Tokens chiffrés (Supabase Vault ou colonne encryptée).
- Refresh token rotation.
- Possibilité de révoquer la connexion (UI profil).
- Audit log toutes opérations.
- Pas de leak du Meet link en public.

## Considérations accessibilité

- Modal de planification : focus trap, calendrier accessible (jour navigable).
- Confirmation événement créé avec `aria-live`.
- Lien Meet visible et copiable.

## Considérations performance

- API call < 2 s p95.
- Cache refresh token (évite refresh à chaque call).
- Batch impossible (1 event = 1 call API).
