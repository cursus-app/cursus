## Contexte business

Les connecteurs (Slack, Discord, Teams, Google Calendar, webhooks sortants) doivent être configurables par les utilisateurs (admin / formateur autorisé). Sans une UI claire, on est contraint d'éditer la DB à la main. Cette story livre l'écran centralisé de gestion des intégrations à l'échelle organisation et par cohorte.

Business value : expérience admin polie qui débloque l'adoption des features Differentiator EP-22. Sans cette UI, les autres stories EP-22 ne sont accessibles qu'aux développeurs.

## Description fonctionnelle

- Page `/admin/integrations` (admin) ou `/cohortes/:id/integrations` (formateur de la cohorte).
- Cards par intégration disponible : Slack, Discord, MS Teams, Google Calendar, Webhooks custom. Chaque card affiche : état (configuré / non), bouton "Configurer" / "Modifier".
- Modale de configuration spécifique par intégration : Slack/Discord/Teams = champ URL + test. Google Calendar = bouton "Connecter mon compte" → OAuth flow. Webhooks = form complet (name, url, events, secret généré).
- Liste des dernières délivrances par intégration (succès / échec / latence) avec drill-down.
- Toggle activer/désactiver sans supprimer la config.
- Documentation inline (lien "Comment configurer").

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Configurer Slack
Given la cohorte X n'a pas d'intégration Slack
When le formateur ouvre /cohortes/X/integrations
And clique "Configurer" sur la card Slack
And colle l'URL webhook + clique "Tester"
Then un message de test est envoyé dans Slack
And un badge vert apparaît "Test réussi"
And le formateur peut sauvegarder l'intégration
```

```gherkin
Scénario 2 — Désactiver intégration
Given Discord est configuré et actif
When le formateur clique le toggle "Actif"
Then l'intégration passe à inactive (sans suppression)
And aucun event ne sera envoyé vers Discord tant qu'inactif
```

```gherkin
Scénario 3 — Voir délivrances récentes
Given la cohorte a un webhook Slack actif
When le formateur clique "Voir les délivrances"
Then une table affiche les 20 derniers events envoyés (status, timestamp, latence)
And un filtre par status (success / failed) est dispo
```

## Cas limites à traiter

- **URL invalide** : test échoue → message clair, bouton sauvegarder désactivé tant que test pas validé.
- **Permission insuffisante** : un formateur ne peut configurer que les intégrations de SES cohortes (RLS).
- **Cohorte déjà lancée** : configurer une intégration en plein cours n'est pas bloqué, mais avertissement "L'historique ne sera pas réenvoyé".
- **Migration d'URL** : l'admin peut juste éditer, sans perdre l'historique des délivrances.
- **Suppression** : confirmation modale (action destructive).
- **Révocation OAuth Google** : détecter un 401 persistant et inviter à re-connecter.

## Non-goals

- Pas de marketplace d'intégrations.
- Pas de configurabilité du payload par intégration au MVP.
- Pas de gestion fine des permissions d'intégration par rôle (juste admin + formateur cohorte).
- Pas de monitoring temps réel des deliveries (juste history).

## Tests à écrire

- **Unit Vitest** : composables d'état d'intégration.
- **Integration** : `POST /api/integrations/slack/test` envoie message mocké.
- **A11y axe-core** : cards et modales 0 violation.
- **E2E Playwright** : parcours formateur configure Slack → voir test réussi → sauve.
- **RBAC** : formateur non-cohorte ne voit pas l'écran.
- **i18n** : tout traduit FR/EN.

## Observabilité

- **Logs** : event `integration.configured` (info) avec `{ kind, cohort_id, actor_id, status }`.
- **Métriques** : compteur `cursus.integrations.configured.total` par kind.
- **Insight** : taux d'utilisation de chaque type d'intégration.

## Considérations sécurité

- RLS strict (formateur cohorte ou admin uniquement).
- Secrets générés côté serveur, affichés 1 fois (pour webhooks).
- Pas de stockage des secrets dans le state Vuex/Pinia (juste fetch on-demand).
- Audit log de toutes config changes.

## Considérations accessibilité

- Toggle accessible (`role="switch"` + `aria-checked`).
- Modales avec focus trap + Esc.
- Cards : `<article>` avec titre h3.
- Liens d'aide annoncés.

## Considérations performance

- Lazy-load des modales (chacune un chunk).
- Skeleton durant fetch des deliveries.
- Pagination des deliveries (20 par page).
- Cible page interactive < 800 ms.
