## Contexte business

Le harnais est le cœur du produit (pilier #2). Pour qu'il reste fiable face aux évolutions (nouveaux types de checks, refonte des YAML), il faut une suite de tests d'intégration qui exécute le harnais réel contre des repos GitHub de référence curés. Le playbook (§4.4) impose une demi-douzaine de fixtures. Sans cette story, chaque modif harnais est un risque silencieux.

## Description fonctionnelle

Dossier `tests/harness-fixtures/` référençant 6-8 repos GitHub publics sur l'org `cursus-fixtures` : `pass-all-checks`, `missing-branch`, `missing-signed-commit`, `deploy-down`, `linter-fails`, `tests-fail`, `no-readme`, `lighthouse-low`. Tests Vitest qui déclenchent le harnais via l'API GH Actions, attendent le webhook callback, et vérifient le rapport produit. Mode "live" en CI nightly (les fixtures sont stables, donc ces tests passent).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Fixture pass-all-checks**

```gherkin
Given le repo "cursus-fixtures/pass-all-checks"
When le harnais est déclenché
Then tous les checks sont en vert
And le statut HarnessRun final est "validated"
And le test passe en moins de 6 min
```

**Scénario 2 — Fixture missing-branch**

```gherkin
Given le repo qui n'a pas la branche "feature/login"
When le harnais est déclenché
Then le check "branch_exists" est rouge
And le rapport explique "Branche feature/login manquante"
And le statut final est "failed"
```

**Scénario 3 — Fixture deploy-down**

```gherkin
Given un repo avec URL de deploy retournant 503
When le harnais ping
Then check "deploy_url_responds" rouge avec détail HTTP
And le rapport explique "URL de déploiement répond 503"
```

## Cas limites à traiter

- **GH Actions rate limit** : runs nocturnes (1×/jour), pas sur chaque PR.
- **Fixture supprimée accidentellement** : check pré-vol (existence repos) avant tests.
- **Latence GH réseau** : timeout 10 min, retry 1×.
- **Webhook callback raté** : timeout job + log + alerte.
- **Branch protection accidentelle sur fixtures** : doc d'opération de ces repos.
- **Cleanup** : aucune side-effect sur les fixtures, lecture seule via le harnais.

## Non-goals

- Pas de tests E2E UI pendant ces tests (purement intégration backend).
- Pas de generation dynamique des fixtures (curées manuellement).
- Pas de tests sur runners self-hosted (cloud GH suffit MVP).

## Tests à écrire

- **Integration** : 8 fixtures, 1 test par cas, cibles déterminées (vert / rouge).
- **Smoke** : check qu'au moins 1 fixture pass et 1 fail sur chaque PR (mode rapide).
- **Nightly** : suite complète.
- **Security** : vérifier que les checks ne leak pas de PII via les payloads webhook.

## Observabilité

- **Logs** : `harness.fixture.test.completed` (info, payload : fixture_name, result, duration_ms)
- **Métriques** : Compteur `cursus.ci.harness_fixtures.pass.total`, `.fail.total`
- **Alertes** : Slack si nightly échoue.

## Considérations sécurité

- Fixtures publiques (pas de secret dedans).
- Le harnais utilise une GitHub App dédiée (pas de PAT personnel).
- HMAC validation sur webhook (cf. ST-06.3).
- Pas de side-effect sur les fixtures (read-only).

## Considérations accessibilité

- Non applicable.

## Considérations performance

- Suite nightly : cible < 30 min total (8 fixtures × < 4 min chacun, parallélisation).
- Webhook timeout : 10 min par run.
- Pas d'impact sur les PR (suite hors-PR).
