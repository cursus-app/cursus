## Contexte business

Le playbook (§8) impose des perf budgets stricts (LCP < 2.5 s, INP < 200 ms, Lighthouse ≥ 90). Sans **Lighthouse CI bloquant**, ces budgets sont des bons souhaits qui dérivent silencieusement à chaque PR. Cette story automatise la mesure et la régression.

C'est aussi un signal de qualité : performance = expérience utilisateur perceptible (Premium 6).

## Description fonctionnelle

Lighthouse CI configuré dans `lighthouserc.json` avec 4-5 URLs clés (auth, dashboard stagiaire, dashboard formateur, profil public, page certif). Workflow CI `lighthouse.yml` qui execute sur chaque PR (preview deploy Vercel) et bloque si budget violé. Rapport HTML attaché à la PR. Persistance dans LHCI server (ou Vercel) pour comparer historiques.

## Critères d'acceptation (Gherkin)

**Scénario 1 — PR sans régression**

```gherkin
Given une PR avec changements UI mineurs
When Lighthouse CI s'exécute sur le preview
Then perf score ≥ 90, a11y ≥ 95
And la check PR "Lighthouse CI" passe vert
```

**Scénario 2 — Régression bloquante**

```gherkin
Given une PR introduit 500 KB JS supplémentaires
When Lighthouse CI passe
Then perf score chute à 78
And la check est rouge
And la PR ne peut pas être mergée
And le commentaire bot détaille les métriques en régression
```

**Scénario 3 — Compare historique**

```gherkin
Given le dashboard LHCI
When un développeur regarde les 30 dernières PRs
Then il voit la courbe d'évolution de chaque métrique
And il peut identifier la PR cause d'une chute
```

## Cas limites à traiter

- **Preview Vercel indisponible** : Lighthouse skip avec warning (non-bloquant).
- **Flaky Lighthouse runs** : 3 runs par URL, médiane retenue.
- **Page derrière auth** : test depuis session login dédiée (fixture user).
- **Nouvelle page non listée dans config** : warning + ajout demandé dans la PR.
- **Optimisation délibérée qui dépasse budget** : override possible via label `perf-budget-override` (audité).

## Non-goals

- Pas de WebPageTest premium (LHCI suffit).
- Pas de Real User Monitoring (RUM) au MVP (Plausible vital metrics suffit).
- Pas de budgets différents par device au MVP (mobile + desktop globaux).

## Tests à écrire

- **CI verification** : intentionnellement régresser une PR de test (ajouter 1MB JS) → LHCI bloque.
- **Smoke** : configurer un nouveau budget plus strict → PR fail.
- **Performance** : measure of Lighthouse CI itself (< 5 min total).

## Observabilité

- **CI logs** : Lighthouse score / metrics par URL.
- **Métriques** : Histogram `ci.lighthouse.score`, gauge `ci.lighthouse.budget_violations`.
- **Alertes** : Slack si main dépasse budget (ne devrait pas, vu blocage PR).

## Considérations sécurité

- Token d'auth fixture : courte durée (1h), rotation hebdomadaire.
- Pages privées : pas de credentials user réel dans la config Lighthouse.
- Rapports Lighthouse peuvent contenir des screenshots : vérifier pas de PII (sur fixtures = OK).

## Considérations accessibilité

- A11y score bloquant aussi (≥ 95).
- Pages clavier-only et lecteur d'écran couvertes par audit Lighthouse.

## Considérations performance

- Budgets stricts cf. playbook §8.1.
- Lighthouse CI lui-même : < 5 min par PR.
- Cache des artifacts Lighthouse.
