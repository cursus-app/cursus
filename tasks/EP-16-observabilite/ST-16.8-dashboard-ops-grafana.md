## Contexte business

Les alertes (ST-16.7) traitent les incidents qui dépassent un seuil. Le **dashboard ops** est la vue panoramique 24/7 pour observer en continu et identifier des tendances : latences cumulées, distribution erreurs, pics d'usage, comparaisons week-over-week. Sans ce dashboard, on est aveugle entre les incidents.

Grafana (cloud free ou self-hosted) lit depuis Postgres (métriques tables) + Sentry API + Plausible API.

## Description fonctionnelle

Grafana Cloud avec 3 datasources :

- Postgres (Supabase réplique read-only) pour métriques applicatives
- Sentry API pour erreurs / perf
- Plausible API pour metrics produit

Dashboards livrés :

1. **Dashboard Ops global** : santé système (uptime, latences API, erreurs, queue jobs)
2. **Dashboard Harnais** : latence p50/p95, taux succès, distribution par cursus
3. **Dashboard Auth & RGPD** : tentatives login, lockouts, exports GDPR, suppressions
4. **Dashboard Produit (Mohamed)** : DAU, cohortes actives, soumissions, capstones (différent du dashboard cohorte formateur)

Versionnés en JSON dans `infra/grafana/dashboards/`.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Dashboard Ops charge en < 5 s**

```gherkin
Given un admin ouvre Grafana cloud

```

**Scénario 2 — Latence harnais (1h)**

```gherkin
Given le dashboard Harnais

```

**Scénario 3 — Versionning JSON**

```gherkin
Given les dashboards sont définis dans infra/grafana/
When une PR modifie un dashboard
Then le diff JSON est reviewé
And l'import dans Grafana se fait via API call automatique
```

## Cas limites à traiter

- **Panne Grafana Cloud** : dashboards locaux fallback (Supabase queries directes).
- **Datasource lente (Sentry rate limited)** : timeout 10 s, message dégradé.
- **Time range trop large (90 jours)** : warning sur perf query.
- **Accès délégué** : SSO Grafana via Auth0 ou équivalent (pas de password partagé).
- **Configuration drift** : CI vérifie que les dashboards en prod = ceux du repo.
