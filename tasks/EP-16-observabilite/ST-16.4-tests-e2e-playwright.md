## Contexte business

Le playbook (§4.5) impose 5 parcours E2E obligatoires (onboarding, soumission/harnais, blocage, création cursus, capstone). Ces tests forment le filet de sécurité anti-régression : ils sont les seuls qui valident que l'app fonctionne du clic jusqu'à la base de données. Sans eux, chaque PR risque de casser un parcours business critique sans qu'on le sache avant production.

Playwright est l'outil retenu (cross-browser, instrumentation excellente).

## Description fonctionnelle

Suite Playwright dans `tests/e2e/` avec 5 specs principaux + helpers communs (login as user, seed DB, etc.). Exécuté sur chaque PR en CI (parallélisé, max 10 min). Configuration : 3 navigateurs (Chromium, Firefox, WebKit), captures vidéo sur échec, traces exportées. Résultats publiés comme artifact.

Parcours implémentés :

1. **Onboarding stagiaire** : invitation → compte créé → GitHub lié → semaine 1 visible
2. **Soumission + harnais validé** : page module → soumission → harnais réussi → portfolio MAJ
3. **Stagiaire bloqué** : bouton → modal → notif formateur → résolution
4. **Création cursus + cohorte + invitation** : formateur crée cursus → démarre cohorte → invite stagiaires
5. **Capstone complet** : soumission → soutenance → évaluation → certificat émis

## Critères d'acceptation (Gherkin)

**Scénario 1 — Onboarding stagiaire**

```gherkin
Given un environnement de test propre seedé
When Playwright joue "01-onboarding.e2e.ts"
Then le test crée une invitation, simule magic link, lie GitHub, voit "Semaine 1"
And le test passe en moins de 90 s
```

**Scénario 2 — Soumission + harnais**

```gherkin
Given un stagiaire est en semaine 2
When le test soumet un repo "fixture-pass"
Then le harnais retourne "validated" sous 5 min (mock GH Actions pour rapidité test)
And la page portfolio affiche le livrable validé
```

**Scénario 3 — Capstone complet**

```gherkin
Given un stagiaire en dernière semaine
When il soumet son capstone et le formateur soutient avec note
Then le certificat est généré en PDF
And le profil public devient accessible
```

## Cas limites à traiter

- **Flaky test (timing)** : retry 2× automatique, marque "flaky" si retry réussit.
- **Test cassant en CI mais pas en local** : artifacts trace + vidéo pour debug.
- **Database state pollué** : reset complet (truncate + seed) avant chaque suite.
- **API externe (GitHub Actions)** : mockée via fixtures de webhook.
- **Resend** : email mocké + capture du mock pour assertions.
- **Race condition entre tests** : parallélisation par workspace isolé (workers Playwright).

## Tests à écrire

- **E2E (Playwright)** : les 5 specs ci-dessus.
- **CI** : workflow bloquant sur main pour merger.
- **A11y** : axe-core intégré dans Playwright (test smoke a11y sur chaque page visitée).

## Observabilité

- **CI** : rapport Playwright HTML exposé comme artifact.
- **Métriques** :

  - Compteur `ci.e2e.pass`, `ci.e2e.fail` (CI metrics).
  - Histogram `ci.e2e.duration.seconds`.

- **Alertes** : Slack si E2E suite échoue sur `main`.

## Considérations sécurité

- Tests utilisent un Supabase de test isolé (jamais prod).
- Pas de credentials de prod dans le code de test.
- Reset DB complet entre suites pour pas de fuite.
- Mock email/GitHub (pas d'appel externe en CI).

## Considérations accessibilité

- Axe-core intégré dans Playwright via `@axe-core/playwright`.
- Chaque page visitée → audit a11y rapide en arrière-plan.
- Test échec si critical a11y violation.

## Considérations performance

- Total suite < 10 min en CI (parallélisation 4 workers).
- Trace + vidéo seulement sur échec.
- Caching browsers Playwright dans CI cache.
