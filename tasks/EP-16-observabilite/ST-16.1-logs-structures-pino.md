## Contexte business

Les logs sont la matière première du debugging en production. Sans logs structurés (JSON avec niveau, timestamp, traceId, userId), aucune corrélation possible, aucune query exploitable. Le playbook (§5.3) impose Pino comme logger et un format JSON pour ingestion par Vercel logs puis BetterStack/Logtail. Cette story finalise et généralise la pratique déjà amorcée dans ST-01.7.

## Description fonctionnelle

Logger Pino configuré globalement (`server/utils/logger.ts`). Output JSON en prod, pretty en dev (via `pino-pretty`). Tous les endpoints Nitro injectent un `traceId` (généré par middleware ou reçu depuis l'en-tête `x-request-id`) et un `userId` si auth. Bibliothèque de helpers : `logger.event('cursus.submission.created', payload)`. Mode lint : aucun `console.log` ne passe en CI (eslint rule).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Logs en prod au format JSON**

```gherkin
Given l'app tourne en production
When un endpoint /api/submissions répond
Then un log JSON est émis avec {level, time, msg, traceId, userId, method, path, status, duration_ms}
And il est ingestible par Vercel logs / Logtail sans transformation
```

**Scénario 2 — Pas de PII dans les logs**

```gherkin
Given un user se connecte avec son email
When le login est tracé
Then aucun log ne contient l'email en clair
And seulement un hash SHA256 "emailHash" apparaît
```

**Scénario 3 — TraceId corrélé entre client et serveur**

```gherkin
Given un appel /api/X depuis le client avec x-request-id=abc123
When le serveur traite
Then tous les logs liés contiennent traceId=abc123
And le client peut chercher ce traceId dans Sentry pour corréler erreur ↔ logs
```

## Cas limites à traiter

- **Logs trop verbeux en debug** : niveau `debug` jamais en prod (configurable par env).
- **Burst de logs (boucle infinie)** : rate limit interne du logger (max 1000/s avec sampling au-delà).
- **Log d'une erreur avec stack trace cyclique** : `pino` gère native.
- **Migration de niveau** : passage de `info` à `error` sans redéploy via feature flag.
- **Données non-sérialisables (BigInt, function)** : custom serializer.

## Non-goals

- Pas d'agrégateur de logs custom (BetterStack/Logtail font le job).
- Pas de tracing distribué complet à ce stade (story OpenTelemetry séparée v1.x si besoin).
- Pas de full event sourcing (logs ≠ store de vérité événementiel).

## Tests à écrire

- **Unit** :

  - `redactPii()` : email retiré, hashEmail présent
  - `attachRequestContext` : traceId stable sur une requête

- **Integration** :

  - Appel `/api/submissions` → log produit conforme à schéma JSON prédéfini
  - Burst : 5000 logs en 1s → sampling sous 1000

- **CI Lint** :

  - `no-console` rule échoue si commit avec `console.log`

- **Security** :

  - Pas de logs de mot de passe, secrets, tokens (test grep pattern)

- **Performance** :

  - Overhead du log < 1 ms par appel.

## Observabilité

- **Logs** : ce sont les logs eux-mêmes.
- **Métriques** :

  - Compteur `cursus.logs.total` labellisé par `level`
  - Histogram `cursus.logs.payload_bytes`

- **Alertes** : pic de logs `error` > 10/min → enquête.

## Considérations sécurité

- Pas de PII (email en hash, jamais en clair).
- Pas de secrets (whitelist serializer).
- Logs persistants dans solution conforme RGPD (Logtail EU).
- Rétention 30 jours par défaut (cf. ST-15.8).

## Considérations accessibilité

- Non applicable (logs backend).

## Considérations performance

- Logger async / non-blocking (pino transport).
- Buffer batch 100 logs / 250 ms vers Vercel.
- Overhead par log < 1 ms.
- Bandwidth sortant : < 10 KB/s en moyenne.
