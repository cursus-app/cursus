## Contexte business

Quand un incident production survient (down, latence, bug critique), l'équipe doit suivre une procédure connue. Le runbook incident centralise les commandes utiles (rollback, kill-switch, status page update), les accès requis, et les escalades. C'est cité par playbook §18 et complémente le runbook sécurité (ST-15.7).

## Description fonctionnelle

Fichier `docs/runbooks/incident-response.md` couvrant 4 scénarios courants : (1) app down, (2) latence élevée, (3) data corruption suspectée, (4) clé tierce compromise. Chaque scénario : symptômes, première action, commandes prêtes à copier-coller, contacts, postmortem template. Sous-runbooks par alerte (lien depuis ST-16.7).

## Critères d'acceptation (Gherkin)

**Scénario 1 — App down**

```gherkin
Given la page d'accueil retourne 500
When l'on-call ouvre le runbook "app-down"
Then il voit en haut : "Rollback dernier deploy" avec commande prête
And les contacts ops sont listés
And un postmortem template est linké
```

**Scénario 2 — Latence élevée**

```gherkin
Given latence p95 > 2 s sur les endpoints
When on suit le runbook
Then on a 5 actions de diagnostic : Vercel logs, Sentry tx, Supabase metrics, Inngest queue, Lighthouse on prod
```

**Scénario 3 — Postmortem**

```gherkin
Given un incident résolu
When on copie le template
Then on a sections : Quoi, Quand, Pourquoi, Action items
And un délai cible 48h après résolution est noté
```

## Cas limites à traiter

- **On-call indisponible** : escalade documenterée.
- **Runbook obsolète** : revue trimestrielle obligatoire.
- **Outils tiers indisponibles** : fallback path.
- **Incident hors-scope** : escalade au founder.
- **Communication utilisateurs** : template d'email + post Slack.

## Non-goals

- Pas d'auto-remediation runbook (humain dans la boucle).
- Pas de chaos engineering au MVP.
- Pas de simulation production.

## Tests à écrire

- **Tabletop** : 1 scénario par trimestre.
- **Scripts dry-run** : vérifier que les scripts marchent sans side-effect.
- **Doc lint** : vérifier que les liens internes au runbook restent valides (CI markdown link check).

## Observabilité

- **Logs** : `incident.runbook.opened` (info, payload : `runbook_name`, `actor`)
- **Métriques** : Compteur `cursus.incidents.runbook_used.total`
- **Alertes** : aucune.

## Considérations sécurité

- Runbook public (pas de secret dedans).
- Scripts en repo, pas de credentials hardcoded.
- Tabletop exercises : pas d'exposition données prod.

## Considérations accessibilité

- Runbook structuré (sections numérotées, table of contents, code blocks copiables).

## Considérations performance

- Rollback Vercel : < 2 min.
- Kill-switch propagation : < 30 s.
