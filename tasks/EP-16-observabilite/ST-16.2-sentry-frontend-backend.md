## Contexte business

Sentry est l'outil de capture d'erreurs choisi (playbook §9). Couvrir front (Vue/Nuxt) + back (Nitro) avec déduplication, breadcrumbs, source maps source de l'analyse de tout incident utilisateur. Sans cette story, les erreurs en prod restent muettes et la résolution prend des jours au lieu de minutes.

## Description fonctionnelle

SDK Sentry initialisé côté client (`@sentry/vue` + `@sentry/nuxt`) et serveur (`@sentry/node`). Captures automatiques : erreurs non gérées Vue, erreurs Nitro/Nuxt, promesses non handlées. Tagging par release (commit SHA), environment, userId (sans PII), feature flags actifs. Source maps uploadées à Sentry pendant le build CI. Filtrage des erreurs bruit (ResizeObserver loop, network aborted). UI : un widget "Report a problem" optionnel.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Erreur Vue capturée**

```gherkin
Given un composant Vue throw une erreur
When elle remonte au top
Then Sentry capture avec stack trace minifiée
And source maps résolvent vers source TS lisible
And la release et userId tagés
```

**Scénario 2 — Erreur API Nitro capturée**

```gherkin
Given un endpoint throw
When erreur loggée
Then Sentry reçoit l'erreur avec traceId, route, status
And l'utilisateur voit un message générique "Quelque chose s'est mal passé" + ID Sentry abrégé
```

**Scénario 3 — Erreur bruit filtrée**

```gherkin
Given une erreur "ResizeObserver loop limit exceeded"
When elle remonte
Then Sentry ne la stocke pas (beforeSend retourne null)
```

## Cas limites à traiter

- **Rate limit Sentry atteint (free tier)** : sampling 50 % au-delà.
- **Erreurs en boucle** : dédoublonnage par fingerprint.
- **PII dans error message** : scrubbing automatique (emails, IPs).
- **Erreurs CORS** : filtrées ou tagged distinctement.
- **Erreurs ancien navigateur** : tagged `browser_unsupported` pour ne pas alerter inutilement.
- **Test Sentry en local** : `SENTRY_DSN` absent → capture mockée en console seulement.

## Non-goals

- Pas de Session Replay au MVP (cost + privacy).
- Pas de profiling continue (sur-dimensionné).
- Pas d'alerting Sentry custom au MVP (utiliser presets Sentry).
- Pas de bug report widget intrusif (uniquement bouton discret).

## Considérations sécurité

- DSN Sentry à révéler côté client (nécessaire pour capture front) mais limité à ingestion (pas API admin).
- Scrubbing PII server-side configuré.
- Source maps privées (uploadées à Sentry mais non accessibles publiquement).
- Rétention 30 jours (cf. ST-15.8).

## Considérations accessibilité

- ErrorBoundary : message clair + bouton "Réessayer" accessible clavier.
- Pas de stack trace en clair pour l'utilisateur.

## Considérations performance

- SDK Sentry chargement lazy (après interactive).
- Sampling traces 10 % (free tier friendly).
- Overhead négligeable (< 100ms init).
