## Contexte business

Une cohorte est un groupe de stagiaires qui suit un cursus avec des dates et un rythme communs. Sans cycle de vie clair (brouillon → active → terminée → archivée), on ne peut pas gérer simultanément plusieurs promotions sans confusion.

Valeur livrée : gestion structurée des promos avec garde-fous (impossible de démarrer sans stagiaires + formateur, impossible de supprimer une cohorte active).

## Description fonctionnelle

Page `/cohortes` (list + filtres). Pages `/cohortes/new`, `/cohortes/:id/edit`, `/cohortes/:id` (vue détaillée + dashboard). Statuts : `brouillon`, `active`, `terminée`, `archivée`. Job nocturne d'archivage automatique après 30j terminée.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Création cohorte**

```gherkin
Given un formateur principal crée une cohorte
When il sélectionne un cursus (publié) et configure dates + rythme
Then la cohorte est créée en statut brouillon
And elle référence une cursus_version figée
```

**Scénario 2 — Démarrage requiert pré-requis**

```gherkin
Given une cohorte est en brouillon
When le formateur clique "Démarrer la cohorte"
Then la cohorte passe à active SI : ≥1 stagiaire invité + ≥1 formateur principal
Else un message indique les pré-requis manquants
```

**Scénario 3 — Archivage automatique**

```gherkin
Given une cohorte est terminée
When 30 jours s'écoulent
Then elle passe automatiquement en archivée (job Inngest)
And les stagiaires gardent accès lecture seule
```

**Scénario 4 — Suppression bloquée si active**

```gherkin
Given une cohorte est active
When le formateur essaie de supprimer
Then l'action est refusée avec un message clair
And l'option "Archiver" est proposée
```

## Cas limites à traiter

- **Date de fin avant date de début** : refus validation Zod.
- **Modification du cursus après démarrage** : interdite (verrouillé sur `cursus_version`).
- **Stagiaires en cours quand cohorte archivée** : maintenir l'accès lecture seule.
- **Cursus publié modifié** : cohorte reste sur version originelle (ST-03.5).

## Non-goals (hors scope)

- Pas de duplication de cohorte au MVP.
- Pas de fusion de cohortes.
- Pas de cohorte multi-cursus.

## Tests à écrire

- **Unit** : transitions de statut (state machine).
- **E2E (Playwright)** : créer cohorte, ajouter stagiaire, démarrer, archiver.
- **Security** : non-formateur ne peut pas créer.
- **Integration** : job d'archivage tourne correctement.

## Observabilité

- **Logs** : `cohorte.created`, `cohorte.started`, `cohorte.archived`.
- **Métriques** : `cohortes.created.total`, `cohortes.active.gauge`.

## Considérations sécurité

- RLS strict : un formateur ne voit que ses cohortes.
- Audit log sur transitions (start, archive, delete).
- Pas de leak de PII stagiaires dans les logs.

## Considérations accessibilité

- Filtres + sélecteurs accessibles clavier.
- Date pickers : label clair, format ISO.

## Considérations performance

- Liste : pagination + index sur `status`, `formateur_id`.
- Page detail : SSR pour LCP < 1.5s.

## Definition of Done spécifique

- Cycle de vie complet testé E2E.
- Job archivage tourne sur fixtures.
- RLS matrix validée.
