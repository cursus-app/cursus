## Contexte business

Imprévu (jour férié, événement collectif, pandémie) : le formateur veut décaler le planning de N jours sans casser les statuts ni les soumissions déjà faites. Sans cette feature, il devrait éditer manuellement chaque échéance — erreur garantie.

Valeur livrée : décalage 1-clic d'une cohorte entière avec notification automatique aux stagiaires.

## Description fonctionnelle

Bouton "Décaler la cohorte" qui décale toutes les échéances aval d'un nombre de jours, en préservant les semaines passées. Modal avec preview du nouveau planning. Notification à tous les stagiaires.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Décalage aval**

```gherkin
Given une cohorte active est en semaine 3
When le formateur décale de 7 jours à partir de la semaine 4
Then les échéances des semaines 4 à N sont décalées de 7 jours
And les semaines 1-3 ne sont pas touchées
And une notif est envoyée à tous les stagiaires
```

**Scénario 2 — Preview avant confirmation**

```gherkin
Given le formateur clique "Décaler la cohorte"
Then une modal s'ouvre avec preview du nouveau planning
And il peut confirmer ou annuler
```

**Scénario 3 — Décalage en arrière (avance)**

```gherkin
Given le formateur décale en arrière (avance de 3 jours)
When il confirme
Then warning fort "Cela peut surprendre les stagiaires"
And les échéances sont effectivement avancées
```

## Cas limites à traiter

- **Décalage en arrière** : autorisé avec warning fort.
- **Décalage qui chevauche le capstone** : avertir.
- **Décalage qui dépasse la date de fin de cohorte** : ajuster la fin automatiquement (avec confirmation).
- **Décalage rapide multiple** : combinable (delta cumulatif).

## Non-goals (hors scope)

- Pas de décalage individuel d'un seul stagiaire (sauf override formateur — voir ST-08.3).
- Pas de re-planification par module spécifique.
- Pas de cal endrier interactif drag-and-drop.

## Tests à écrire

- **Unit** : algo de décalage (positif, négatif, semaines passées).
- **E2E (Playwright)** : décalage + vérification que les dates sont mises à jour.
- **Edge** : décalage qui passerait le capstone après la date de fin.
- **Integration** : notif envoyée à tous les stagiaires.

## Observabilité

- **Logs** : `cohorte.schedule_shifted` (avec delta_days, week_from).
- **Métriques** : `cohortes.shifted.total`.

## Considérations sécurité

- RLS : formateur principal uniquement.
- Audit log obligatoire.
- Validation Zod du delta (entre -30 et +90 jours).

## Considérations accessibilité

- Preview du nouveau planning : table accessible (caption + scope).
- Warning fort : `role="alert"`.

## Considérations performance

- Bulk update : transaction unique, < 1s pour 30 stagiaires × N modules.
- Notifications : enqueue async (pas bloquant).

## Definition of Done spécifique

- E2E décalage +7 jours vérifié.
- Décalage qui dépasse la fin = ajustement automatique.
- Notif reçue par tous les stagiaires test.
