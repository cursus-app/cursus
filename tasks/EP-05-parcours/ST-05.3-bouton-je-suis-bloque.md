## Contexte business

Le stagiaire bloqué doit pouvoir demander de l'aide _sans hésitation_. Sans Cursus, il hésite à déranger son formateur pour une question qu'il pense triviale, et finit par abandonner ou rester bloqué plusieurs jours en silence. Cette story matérialise l'un des 5 piliers du produit : **"suivi formateur asynchrone"**.

Valeur livrée : escalade en 1 clic + 1 phrase, friction zéro, message positif pour ne pas culpabiliser le stagiaire.

## Description fonctionnelle

Bouton "Je suis bloqué" visible en permanence sur la page du livrable. Modal avec textarea ("Décris ton blocage en 3 phrases", min 20 char, max 500). Crée une `Alert` kind="blocked", notifie le formateur principal (in-app + email), passe le statut du module en "bloqué". Toast positif au stagiaire.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Demande d'aide**

```gherkin
Given un stagiaire est sur son livrable
When il clique "Je suis bloqué"
Then une modal s'ouvre demandant "Décris ton blocage en 3 phrases"
And il doit saisir minimum 20 caractères
And le bouton "Envoyer" est désactivé tant que <20 char
```

**Scénario 2 — Soumission valide**

```gherkin
Given la modal est ouverte et le stagiaire a saisi son blocage
When il clique "Envoyer"
Then une Alert est créée en DB (kind="blocked")
And une notif in-app + email est envoyée au formateur principal
And le statut du module passe en "bloqué"
And un toast positif s'affiche "Mohamed a été prévenu, il va te répondre rapidement"
```

**Scénario 3 — Alerte déjà ouverte**

```gherkin
Given le stagiaire a déjà une alerte ouverte sur ce module
When il reclique "Je suis bloqué"
Then un message "Tu as déjà demandé de l'aide, attends une réponse" s'affiche
And un lien "Voir ma demande en cours" est offert
```

## Cas limites à traiter

- **Multi-clic rapide** : debounce 2s côté serveur (idempotence par stagiaire + module + 5 dernières minutes).
- **Texte vide ou < 20 caractères** : bouton désactivé + refus serveur 400.
- **Texte > 500 caractères** : refus avec "Garde-le court".
- **Stagiaire qui se débloque seul** : bouton "C'est bon, j'ai trouvé" pour auto-résolution.
- **Formateur principal indisponible** : au MVP, alerte tombe quand même chez lui.

## Non-goals (hors scope)

- Pas de système de tickets / threads au MVP (l'alerte est résolue par un commentaire formateur).
- Pas de chat en temps réel avec le formateur.
- Pas d'upload de capture d'écran dans la modal au MVP.
- Pas de catégorisation (technique/conceptuel) — texte libre uniquement.

## Tests à écrire

- **Unit** : `alertService.createBlockedAlert()` cas idempotent (2 appels → 1 seule Alert).
- **E2E (Playwright)** : clic → modal → saisie → envoi → toast → notif visible côté formateur.
- **E2E** : re-clic après alerte ouverte = message + lien.
- **Security** : tentative d'injection HTML → échappé.
- **A11y** : modal `role="dialog"`, focus piege, ESC ferme.

## Observabilité

- **Logs** : `alert.created` (info), `alert.duplicate_blocked` (warn).
- **Métriques** : `alerts.created.kind=blocked.total`, median time-to-resolution.
- **Alertes** : pic d'alertes blocked sur un module = problème pédagogique.

## Considérations sécurité

- Validation Zod stricte (moduleId UUID, message 20-500).
- RLS : un stagiaire ne peut créer une Alert que pour un module qu'il suit.
- Rate limit : 5 alertes / h / stagiaire.
- Échappement HTML dans le rendu email.

## Considérations accessibilité

- Bouton "Je suis bloqué" : taille touch target ≥ 44px, contraste 4.5:1, label aria.
- Modal : `role="dialog"`, `aria-modal="true"`, focus piege, ESC ferme.
- Toast : `aria-live="polite"`, durée 4s.
- Respect `prefers-reduced-motion`.

## Considérations performance

- Endpoint : p95 < 300ms (validation + 2 inserts DB + 1 notif + 1 email enqueue).

## Definition of Done spécifique

- Test idempotence avec 10 requêtes parallèles passe.
- Rendu email vérifié sur Gmail + Outlook + Apple Mail.
- Latence p95 < 300ms sur staging.
