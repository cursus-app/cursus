## Contexte business

Action principale du stagiaire : soumettre son repo + URL de deploy. Déclenche le harnais. Le résultat doit revenir vite (< 5 min) et de manière lisible. **C'est le moment clé de l'expérience produit** : un harnais lent ou un rapport illisible casse la promesse de feedback rapide.

Valeur livrée : soumission fluide, statut temps réel via WebSocket, rapport humain lisible, blocage anti-spam au-delà de 10 échecs.

## Description fonctionnelle

Bouton "Soumettre" sur la page du module. Modal avec champs URL repo (validé GitHub + compte lié) et URL deploy (si requise). HarnessRun créé en statut "running". Affichage temps réel via Supabase Realtime. Rapport en cartes par check.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Soumission valide**

```gherkin
Given un stagiaire est sur son livrable
When il clique "Soumettre" et saisit son URL de repo
Then la modal valide que c'est bien un repo GitHub
And valide que le repo appartient à son compte GitHub lié
And un HarnessRun est créé en statut "running"
```

**Scénario 2 — Réception temps réel du résultat**

```gherkin
Given le HarnessRun est en cours
When le stagiaire reste sur la page
Then il voit un indicateur "Analyse en cours..." avec progression par étape
And le résultat arrive en push via Supabase Realtime (sans rafraîchir)
And les checks apparaissent un par un dans le rapport
```

**Scénario 3 — Validation réussie**

```gherkin
Given le harnais a terminé
When le statut est "validated"
Then les 7+ checks sont affichés en vert (avec messages)
And un toast "Bien joué !" apparaît
And le portfolio est mis à jour (XP attribués)
```

**Scénario 4 — Échec avec retry**

```gherkin
Given le harnais a échoué
When le statut est "failed"
Then les checks en échec sont en rouge avec messages clairs
And un bouton "Soumettre une nouvelle version" est disponible
```

**Scénario 5 — Blocage anti-spam**

```gherkin
Given le stagiaire a déjà 10 soumissions échouées
When il essaie une 11ᵉ
Then le bouton est désactivé avec message "Demande l'aide de ton formateur"
And une alerte est créée pour le formateur
```

## Cas limites à traiter

- **Repo non public** : message clair "Rend ton repo public ou demande l'aide".
- **URL malformée** : validation client + server (regex GitHub repo).
- **Timeout du harnais** (>10 min) : statut "timeout", possibilité de relancer.
- **Soumission pendant qu'une analyse est en cours** : refus avec message.
- **Repo qui n'est pas du stagiaire** : refus (compare avec github_handle stocké).

## Tests à écrire

- **E2E (Playwright)** : soumission complète happy path.
- **E2E** : 10 échecs successifs → blocage.
- **Unit** : validation URL GitHub.
- **Integration** : Realtime push fonctionne (WebSocket mocké).
- **Security** : un stagiaire ne peut pas soumettre pour un autre.

## Observabilité

- **Logs** : `submission.created`, `submission.validated`, `submission.failed`.
- **Métriques** : `submissions.created.total`, `submissions.validated.total`, `submissions.failed.total`.
- **Latence** : p50/p95 entre submission et résultat affiché (cible p95 < 5 min).
- **Alertes** : si latence p95 > 7 min sustained.

## Considérations sécurité

- Validation que le repo appartient au stagiaire (`github_handle` match).
- RLS : stagiaire crée une submission pour lui uniquement.
- Rate limit : 5 submissions / module / h.
- Sanitize URL (pas d'XSS dans le rapport).
- Anti-spam : 10 échecs → alerte automatique.

## Considérations accessibilité

- Modal : `role="dialog"`, focus piege, ESC ferme.
- Indicateur progress : `aria-live="polite"`, `aria-valuenow`.
- Rapport : structure héadings, navigation clavier.
- État success/fail : icône + texte (pas seulement couleur).

## Considérations performance

- **Chemin critique** : page module + soumission < 1.5s LCP.
- Realtime : reconnexion automatique sur perte WS.
- Rapport : virtualization si > 30 checks.

## Definition of Done spécifique

- E2E happy path < 5 min total.
- Latence p95 mesurée sur staging < 5 min.
- Test blocage anti-spam après 10 échecs.
