## Contexte business

Un admin doit pouvoir vérifier "tout va bien" en un coup d'œil sans connecter sur Grafana/Sentry (ST-16.8 fait l'observabilité ops complète). Cette page synthétise les indicateurs clés directement dans Cursus : statut Supabase, Resend, Inngest, GitHub Actions ; latence harnais ; taux d'erreur récent ; jobs en file. C'est un confort opérationnel pour Mohamed qui est admin produit et "premier ops".

C'est aussi rassurant pour la direction qui veut un dashboard ultra-simple "vert/rouge".

## Description fonctionnelle

Page "Administration > Système". 6 cartes :

1. **Statut services** : 4 pastilles vert/orange/rouge pour Supabase, Resend, Inngest, GitHub Actions (basées sur ping + status pages externes).
2. **Latence harnais (24h)** : p50 et p95, courbe simple.
3. **Erreurs (24h)** : nombre d'erreurs 5xx, taux d'échec login.
4. **Jobs Inngest** : nb en file, nb échoués retry-able.
5. **Volumétrie** : DAU, cohortes actives, submissions du jour.
6. **Dernier déploiement** : version + commit SHA + date + lien GitHub.

Page mise à jour toutes les 30 s (polling). Aucun appel direct à des services tiers depuis le front.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Vue d'ensemble santé**

```gherkin
Given un admin connecté
When il navigue vers "Administration > Système"
Then la page charge en moins de 2 s
And il voit les 6 cartes avec données fraîches (<60 s)
And les pastilles statut sont vert/orange/rouge selon les pings
And la latence harnais affiche p50 et p95 sur 24h
```

**Scénario 2 — Service en panne**

```gherkin
Given le ping HEAD vers `api.resend.com` échoue 3× consécutivement
When l'admin charge la page
Then la pastille Resend est rouge
And un message "Resend semble indisponible. Les emails sont en file d'attente."
And l'événement "system.service.down" est tracé (warn)
```

**Scénario 3 — Refresh automatique**

```gherkin
Given un admin reste sur la page 60 secondes
When 30 s s'écoulent
Then les données se rafraîchissent silencieusement sans clignotement
And l'horodatage "Dernière mise à jour : il y a 5 s" se met à jour
```

## Cas limites à traiter

- **Tous services down** : grand bandeau rouge "Plusieurs services semblent indisponibles, regarde #ops".
- **Onglet inactif** : pause du polling après 5 min (Page Visibility API).
- **Métriques absentes (dev)** : skeleton + message "Pas encore de donnée" plutôt qu'erreur.
- **Très haute latence harnais** : orange à p95 > 5 min, rouge à > 7 min (cohérent avec alertes ST-16.7).
- **Polling après laptop sleep** : reprend proprement sans burst de requêtes.

## Sous-tâches techniques

- **TT-14.5.1** — Endpoint `GET /api/admin/system/health` qui agrège tous les indicateurs (Promise.all parallèle)
- **TT-14.5.2** — Service `healthCheckService.pingExternal(url)` avec timeout 3 s et retry 1×
- **TT-14.5.3** — Cache Upstash Redis 30 s sur les pings (évite hammer)
- **TT-14.5.4** — Query Prisma : latence harnais sur 24h via `prisma.harnessRun.aggregate({ _avg, _max })` ou `$queryRaw` avec `percentile_cont`
- **TT-14.5.5** — Query Prisma : erreurs (table `logs` ou Sentry API)
- **TT-14.5.6** — Endpoint Inngest `GET /v1/queue/stats` pour jobs en file
- **TT-14.5.7** — Page `pages/admin/system.vue` avec `<HealthCard>` + polling 30 s via `useIntervalFn` (`@vueuse/nuxt` ^12)
- **TT-14.5.8** — Composant `<MetricCard>` réutilisable (UCard de @nuxt/ui : titre, valeur principale, sparkline SVG)
- **TT-14.5.9** — Build-time injection : `__GIT_COMMIT_SHA__` injecté par Nuxt 4 config

- Pas d'historique long terme (> 7 jours) : page = "santé maintenant".

- Pas de SLI/SLO formels — réservé v1.x.

## Tests à écrire

- **Unit** :

  - `healthCheckService.pingExternal()` : timeout, retry, cache
  - Reducer "statut global" : rouge si ≥ 1 rouge, orange si ≥ 1 orange, vert sinon

- **Integration** :

  - `GET /api/admin/system/health` répond < 1 s même si 1 service externe time out
  - Cache Redis : 2ᵉ appel dans les 30 s renvoie cached

- **E2E (Playwright)** :

  - Admin charge la page → 6 cartes apparaissent
  - Polling : avancer le temps, vérifier rafraîchissement

- **A11y (axe-core)** :

  - Pastilles : icône + texte de statut (jamais couleur seule)
  - Sparklines : description textuelle alternative

- **Security** :

  - RBAC : non-admin → 403
  - Pas d'exposition de tokens externes dans la réponse

- **Performance** :

  - p95 endpoint < 800 ms (parallel + cache)

## Observabilité

- **Logs** :

  - `system.health.checked` (info, périodique)
  - `system.service.down` (warn, payload : `service`, `attempts`)

- **Métriques** :

  - Gauge `cursus.system.service.up` labellisé par `service`
  - Histogram `cursus.system.health.duration.seconds`

- **Alertes** : alerte si Cursus lui-même cesse de répondre à ce healthcheck (canary externe).

## Considérations sécurité

- Aucun secret en réponse JSON (latences, statuts, compteurs uniquement).
- Endpoint cache + rate limit (Upstash, 60 req/min/admin).
- Pas de leak de versions de dépendances (uniquement version Cursus + commit SHA).
- Pings externes faits par le serveur (pas le browser) pour éviter CORS et CSRF.

## Considérations accessibilité

- Cartes avec titres `<h2>`, statut annoncé via `aria-label` (`Supabase : opérationnel`).
- Polling rafraîchissement annoncé discrètement via `aria-live="polite"`.
- Sparkline : `<svg role="img" aria-label="Latence p95 sur 24h, dernière valeur 2.3 min">`.

## Considérations performance

- Polling 30 s pause si onglet inactif (Page Visibility API).
- Cache Redis 30 s côté serveur sur requêtes externes.
- Cible : LCP < 1.5 s sur la page (peu de DOM, charts SVG inline).
