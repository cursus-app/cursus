## Contexte business

L'audit log est déjà écrit en DB (chaque action sensible est tracée : suppression de compte, override de statut, génération de certificat, etc.). Mais sans une UI, ces données sont accessibles uniquement via SQL — donc inaccessibles à un admin non-technique. Cette story expose l'audit log dans l'app sous forme d'explorer filtrable, permettant à l'admin de répondre rapidement à des questions du type "qui a forcé la validation de ce capstone ?" ou "quand le compte X a-t-il été désactivé ?".

C'est aussi un prérequis fonctionnel des conformités à venir (ISO 27001, SOC 2-like) si Cursus se commercialise. Sans UI, le contrôle interne reste théorique.

## Description fonctionnelle

Page "Administration > Audit log" affichant un tableau paginé des événements (30 derniers jours par défaut). Colonnes : timestamp, acteur, type d'événement, cible (ID + nom résolus), contexte (JSON résumé). Filtres : période (date range), acteur (autocomplete users), type d'événement (multiselect), ID de cible (recherche). Une ligne cliquée ouvre un panel de droite avec le JSON complet et le diff "avant / après" si l'action est une modification.

Export CSV de la sélection courante (réutilise l'infra de ST-14.2). Pas d'édition possible — l'audit log est immuable.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Recherche par utilisateur**

```gherkin
Given un admin connecté sur "Audit log"
When il sélectionne dans le filtre "Acteur" l'utilisateur "Mohamed S."
And il sélectionne la période "30 derniers jours"
Then la table affiche toutes les actions effectuées par Mohamed S. sur cette période
And chaque ligne affiche le timestamp UTC + timezone locale au survol
And le compteur "423 événements" est visible en haut
```

**Scénario 2 — Détail d'un événement**

```gherkin
Given un événement "submission.overridden" est listé
When l'admin clique sur la ligne
Then un panel s'ouvre à droite
And il affiche le JSON complet du contexte
And il affiche le diff "ancien statut: failed → nouveau statut: validated"
And il affiche la raison saisie par le formateur
```

**Scénario 3 — Filtres multiples**

```gherkin
Given l'admin sélectionne les types ["user.disabled", "user.created"]
When il applique les filtres
Then seules ces deux catégories d'événements apparaissent
And la pagination cursor se reset au début
And l'URL contient les filtres (deeplinkable)
```

## Cas limites à traiter

- **Audit log volumineux (> 100k events)** : pagination cursor-based stricte (Prisma `cursor`), pas de count total exact.
- **Utilisateur cible supprimé RGPD** : ID UUID affiché + label "(compte supprimé)".
- **Acteur système (job cron, webhook)** : affiché comme "Système" avec icône robot.
- **Période > 12 mois** : avertissement "Au-delà de 12 mois, les événements peuvent être archivés (ST-15.8)" + bouton "Rechercher dans l'archive" (job async).
- **JSON contexte > 10 Ko** : tronqué en preview avec bouton "Voir le contexte complet".
- **Filtres vides** : empty state "Aucun événement ne correspond. Élargis la période."

## Tests à écrire

- **Unit** :

  - Validation Zod des filtres (date range valide, event_types whitelistés)
  - `resolveActorAndTarget()` sur événements avec acteurs supprimés
  - `EventDiffView` : rendu pour absent → présent

- **Integration** :

  - Query indexée : `EXPLAIN` confirme l'usage d'index composites
  - Pagination cursor Prisma : 3 pages successives sans doublon ni saut
  - Filtres combinés : `actorId` + `eventType` + période → résultats intersectés

- **E2E (Playwright)** :

  - Admin filtre par utilisateur et type → résultats affichés
  - Clic sur ligne → panel ouvre → JSON visible
  - Deeplink URL avec filtres → page chargée avec mêmes filtres

- **A11y (axe-core)** :

  - UTable navigable clavier, ligne sélectionnée annoncée
  - UDrawer détail : focus déplacé à l'ouverture, Esc ferme

- **Security** :

  - RBAC : non-admin → 403
  - SQL injection via `q` : impossible (Prisma `where` paramétré)
  - Pas de fuite d'audit log d'autres organisations

## Considérations sécurité

- Lecture seule strict : pas d'endpoint PATCH/DELETE sur audit_log.
- RBAC `admin` strict.
- Pas d'API publique de l'audit log.
- Toute consultation est elle-même tracée (`audit.viewed`) — meta-audit.

## Considérations accessibilité

- UTable : navigation clavier (Tab/Arrow), Enter/Space pour sélection, focus dans le détail.
- Filtres : labels explicites, états de chargement annoncés via `aria-live`.
- Diff view : sémantique `<ins>` / `<del>` (en plus de la couleur).

## Considérations performance

- Cible : recherche p95 < 800 ms même sur 1M lignes (index composites Prisma).
- Partitionnement DB par mois (Postgres `PARTITION BY RANGE (created_at)`) v1.x si nécessaire — géré via SQL brut dans migration Prisma dédiée.
- Pas de count(\*) global (utilisation `pg_class.reltuples`).
- Cache court (60s) sur autocomplete acteurs/event_types.
