## Contexte business

Le certificat Cursus est l'**objet final** que le stagiaire reçoit et présente publiquement (LinkedIn, CV). Pour qu'il porte la marque de l'organisation qui forme (et non "Cursus" génériquement), l'admin doit pouvoir injecter son logo, sa couleur d'accent, son texte d'accroche et choisir un template visuel parmi 2-3 styles. C'est ce qui distingue un certificat anonyme d'un certificat marqué.

Au-delà du certificat, ce branding s'applique aussi aux emails transactionnels critiques (invitation, validation capstone, émission certificat) — l'utilisateur reçoit ces emails au nom de l'organisation qui forme.

## Description fonctionnelle

Page "Administration > Branding". Formulaire avec : upload de logo (PNG/SVG, max 2 Mo, 200×200 px recommandé), color picker pour la couleur d'accent (avec aperçu live), champ "Nom officiel de l'organisation" (pied de certificat), choix d'un template parmi 3 visuels prédéfinis (`classic`, `modern`, `minimal`). Un panneau de droite affiche en temps réel un aperçu du certificat.

Au save, les paramètres sont persistés dans `organization_settings` (préparation multi-tenant). Tous les certificats émis ultérieurement utilisent ces paramètres. Les certificats déjà émis ne sont pas re-générés (immutabilité).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Mise à jour du logo et couleur**

```gherkin
Given un admin connecté sur la page "Branding"
When il upload un logo PNG 512×512 et choisit la couleur "#7c3aed"
Then l'aperçu du certificat se met à jour en moins de 500 ms
And le logo est validé côté serveur (mime type, dimensions, taille)
When il clique sur "Enregistrer"
Then les paramètres sont persistés en DB
And l'événement "branding.updated" est tracé
And un toast "Branding mis à jour" s'affiche
```

**Scénario 2 — Choix de template**

```gherkin
Given l'admin a configuré son logo et sa couleur
When il sélectionne le template "modern"
Then l'aperçu se rafraîchit avec la nouvelle mise en page
And il peut basculer entre les 3 templates sans recharger
```

**Scénario 3 — Validation format**

```gherkin
Given l'admin tente d'uploader "logo.gif" de 5 Mo
When le fichier est sélectionné
Then une erreur s'affiche : "Format non supporté (PNG ou SVG uniquement) et taille max 2 Mo"
And le fichier n'est pas uploadé
And aucun appel serveur n'est fait
```

## Cas limites à traiter

- **Logo trop petit (< 100×100)** : warning non bloquant "Le logo risque d'être pixelisé sur le PDF".
- **SVG avec script embarqué** : nettoyage côté serveur via DOMPurify, refus si `<script>` ou `<foreignObject>`.
- **Couleur d'accent à contraste insuffisant** : warning "Le contraste avec le fond blanc est de 2.1, lisibilité réduite" (WCAG ratio).
- **Rollback** : bouton "Restaurer les paramètres précédents" (table d'historique).
- **Concurrence admin** : 2 admins éditent en même temps → optimistic locking (`updated_at`), 409 si conflit.

## Tests à écrire

- **Unit** :

  - `sanitizeSvg()` : rejets de SVG malicieux (script, foreignObject, onload)
  - `validateImage()` : rejet de fichiers mime-spoofed (gif renommé en png)
  - `getContrastRatio()` : conformité formule WCAG

- **Integration** :

  - `PUT /api/admin/branding` met à jour la row + historique
  - Optimistic locking : 2 PUT concurrents → 1 succès, 1 conflit 409
  - Génération de certificat utilise bien le logo et la couleur configurés

- **E2E (Playwright)** :

  - Upload logo → aperçu mis à jour → save → recharge page → settings persistés
  - Émettre un certificat → vérifier que le PDF contient le bon logo

- **A11y (axe-core)** :

  - Color picker accessible clavier (sliders ARIA + saisie hex)
  - Aperçu certificat : description textuelle alternative

- **Security** :

  - SVG avec `<script>` rejeté
  - PNG avec extension `.svg` rejeté (mime sniffing)
  - Path traversal bloqué (bucket scoping)
  - Écriture sans rôle admin → 403

## Observabilité

- **Logs** :

  - `branding.updated` (info, payload : `actor_id`, `changed_fields`)
  - `branding.logo.upload.rejected` (warn, payload : raison)

- **Métriques** :

  - Compteur `cursus.branding.updates.total`
  - Compteur `cursus.branding.logo.rejected.total` labellisé par `reason`

## Considérations sécurité

- Upload : limite 2 Mo, mime check côté serveur (sharp images bitmap, DOMPurify SVG).
- SVG sanitization stricte (xss-vector connu).
- Audit log de tout changement de branding.
- URL publique du logo OK (déjà destiné à apparaître sur le certificat public).

## Considérations accessibilité

- Aperçu certificat : `role="img" aria-label="Aperçu de ton certificat"`.
- Color picker : alternative saisie hex texte, focus visible.
- Logo : `alt` cohérent dans tous les rendus (certificat, emails, header).

## Considérations performance

- Aperçu live : debounce 300 ms sur changements de couleur.
- Logo PNG : pré-calcul des variantes via `@nuxt/image`.
- Cible : page de branding charge en < 1.5 s, save < 500 ms.
