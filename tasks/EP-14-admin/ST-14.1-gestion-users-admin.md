## Contexte business

Une fois le pilote livré, Cursus va héberger plusieurs cohortes successives, plusieurs formateurs, et probablement quelques co-formateurs invités ponctuellement. L'admin (initialement Mohamed lui-même) a besoin d'un panneau pour créer, désactiver et gérer ces comptes formateurs sans passer par la console Supabase. Cette story incarne le pilier "industrialisation" de la v1.1 : pouvoir grandir d'une à plusieurs cohortes sans dépendance technique.

Sans cet écran, chaque ajout de formateur passe par une intervention manuelle dans la base, ce qui freine l'adoption interne et expose à des erreurs (rôle mal attribué, oubli de désactivation post-départ).

## Description fonctionnelle

Section "Administration > Utilisateurs" accessible uniquement aux comptes de rôle `admin`. On y voit la liste paginée de tous les comptes (stagiaires, formateurs, admins) avec filtres par rôle, statut (`actif`, `désactivé`), date de création et dernière connexion. Trois actions sont disponibles : **Créer un compte formateur** (modal avec email, nom, rôle, envoi automatique d'une invitation magic-link), **Désactiver un compte** (soft-delete : compte non supprimé mais incapable de se reconnecter, sessions invalidées), **Réactiver un compte** désactivé.

Chaque action est tracée dans l'audit log avec l'admin qui l'a faite. L'UI affiche un badge `désactivé` distinct (gris barré) et un bandeau d'info pour les comptes désactivés.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Création d'un compte formateur par l'admin**

```gherkin
Given un utilisateur connecté avec le rôle "admin"
When il navigue vers "Administration > Utilisateurs"
And il clique sur "Créer un formateur"
And il saisit email "amina@example.com", nom "Amina K.", rôle "formateur_principal"
And il soumet le formulaire
Then un compte est créé avec le statut "en attente d'activation"
And un email d'invitation magic-link est envoyé via Resend
And l'événement "admin.user.created" est tracé dans l'audit log
And un toast "Invitation envoyée à amina@example.com" s'affiche
```

**Scénario 2 — Désactivation d'un compte**

```gherkin
Given un compte formateur existant et actif
When l'admin clique sur "Désactiver" depuis la ligne
And il confirme dans la modal "Cette action coupera l'accès immédiatement"
Then la colonne `disabled_at` est renseignée en DB
And toutes les sessions actives sont invalidées (revoke)
And le compte ne peut plus se connecter (auth rejette avec message "Compte désactivé, contacte l'admin")
And l'événement "admin.user.disabled" est tracé dans l'audit log
And la ligne affiche un badge "Désactivé" et une date
```

**Scénario 3 — Accès refusé pour non-admin**

```gherkin
Given un utilisateur avec le rôle "formateur_principal"
When il tente d'accéder à "/admin/users" directement
Then il reçoit un statut HTTP 403
And il est redirigé vers son dashboard formateur
And l'événement "admin.access.denied" est tracé (warn)
```

## Cas limites à traiter

- **Tentative de désactivation de soi-même** : l'admin ne peut pas se désactiver — bouton masqué + 400 côté serveur (`cannot disable self`).
- **Dernier admin** : interdit de désactiver le dernier compte admin actif (sinon plus aucun accès admin) — 400 côté serveur avec message clair.
- **Email déjà existant** : si l'email est déjà rattaché à un compte (même désactivé), refus 409 avec message "Un compte existe déjà avec cet email".
- **Formateur en cours d'activité** : si le compte à désactiver est `formateur_principal` actif d'une cohorte, afficher une confirmation renforcée listant les cohortes impactées et imposer une réassignation préalable.
- **Réactivation après désactivation longue** : un compte désactivé depuis > 12 mois est considéré purgé conformément à la politique de rétention (ST-15.8) — réactivation impossible, l'admin doit créer un nouveau compte.

## Non-goals

- Pas d'édition fine du profil utilisateur (avatar, bio, etc.) — l'utilisateur le fait depuis son propre profil.
- Pas d'import CSV en masse de comptes au MVP v1.1 — ajouter en v1.2 si volume justifie.
- Pas de gestion multi-organisation — single-tenant assumé jusqu'à la v2.
- Pas de transfert de propriété d'une cohorte d'un formateur à un autre (story dédiée v1.2 si besoin).

## Tests à écrire

- **Unit** :

  - Validation Zod des payloads create/disable
  - Helper `canDisableUser(actor, target)` : règles "pas soi-même", "pas le dernier admin"

- **Integration** :

  - `POST /api/admin/users` → compte créé + email mis en queue Resend
  - `POST /api/admin/users/:id/disable` → `disabled_at` renseigné + tentative login refusée
  - Cas négatif RLS : un `formateur_principal` qui appelle `/api/admin/users` reçoit 403

- **E2E (Playwright)** :

  - Parcours admin : créer un formateur → vérifier réception email d'invitation (mock Resend) → désactiver → vérifier login impossible
  - Tentative de désactivation de soi-même : bouton absent

- **A11y (axe-core)** :

  - UTable navigable au clavier, tri annoncé via `aria-sort`
  - UModal confirm avec focus piégé, Esc ferme

- **Security** :

  - Bypass tentatif : appel direct `/api/admin/users` avec token formateur → 403
  - Injection dans `q` (recherche) → `prisma.user.findMany({ where: { OR: [...] } })` paramétré, pas de risque

## Observabilité

- **Logs** :

  - `admin.user.created` (info, payload : `target_user_id`, `actor_id`, `role`)
  - `admin.user.disabled` (info, payload : `target_user_id`, `actor_id`, `reason`)
  - `admin.access.denied` (warn, payload : `actor_id`, `path`)

- **Métriques** :

  - Compteur `cursus.admin.users.created.total`
  - Compteur `cursus.admin.users.disabled.total`
  - Compteur `cursus.admin.access.denied.total` labellisé par `path`

- **Alertes** : alerte si `admin.access.denied.total` > 20/heure (tentative d'escalade de privilèges).

## Considérations sécurité

- RBAC strict : middleware `requireRole(["admin"])` sur toutes les routes `/api/admin/*` et `/admin/*` pages.
- RLS Supabase : table `users` accessible en lecture/écriture seulement par les admins via policy `user_role = 'admin'` (effective grâce au rôle Postgres `cursus_app` utilisé par Prisma).
- Audit log immuable : insertions seulement (pas d'update/delete), via table partitionnée.
- Rate limit : 30 créations/heure/admin pour éviter un spam d'invitations.
- Invalidation de sessions : appel `supabase.auth.admin.signOut(userId)` à la désactivation.

## Considérations accessibilité

- UTable : `<table>` sémantique avec `<th scope="col">`, tri annoncé via `aria-sort`.
- Boutons d'action de ligne : `aria-label="Désactiver le compte de Amina K."` (texte dynamique).
- Badges de statut : couleur + icone Tabler + texte (jamais couleur seule).
- Toasts annoncés via `aria-live="polite"`.

## Considérations performance

- Pagination cursor-based (Prisma `cursor`/`take`/`skip`) pour scale > 1000 utilisateurs.
- Index DB : `@@index([globalRole, status])`, `@@index([email])` dans `prisma/schema.prisma`.
- Cible p95 du `GET /api/admin/users` : < 200 ms pour 500 utilisateurs.
