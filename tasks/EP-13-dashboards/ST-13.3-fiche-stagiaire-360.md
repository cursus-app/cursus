## Contexte business

Quand un formateur détecte qu'un stagiaire a besoin d'attention (via la heatmap CUR-85, une alerte, un message), il doit pouvoir basculer en moins de 3 secondes sur une vue complète de ce stagiaire : tout son historique, toutes ses actions possibles. Sans cette vue, le formateur jongle entre 5+ écrans (livrables, harnais, commentaires, audit, planning) ce qui rend les interventions lentes et incomplètes.

La fiche stagiaire 360 matérialise le pilier « le formateur intervient avec contexte complet » et complète la heatmap : la heatmap dit « qui » a besoin d'aide, la fiche dit « pourquoi » et offre « quoi faire ». C'est la vue de résolution d'incident pédagogique.

## Description fonctionnelle

Page `/cohortes/:cohorteId/stagiaires/:stagiaireId` réservée formateurs/co-formateurs. Structure : (1) entête avec avatar, nom, statut global, KPIs personnels (% cursus, XP, badges, alertes actives, dernière activité) ; (2) timeline chronologique unifiée (livrables soumis, runs harnais, quiz, commentaires formateur, alertes déclenchées/résolues, badges obtenus, mentions feed) avec filtres par type ; (3) panneau d'actions contextuelles : commenter le stagiaire, override un livrable, planifier un appel (lien Calendly externe), étendre une échéance, marquer une alerte traitée, envoyer un message ; (4) onglet « audit log » avec l'historique technique complet.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Vue 360 complète**

```gherkin
Given un formateur clique sur un stagiaire depuis la heatmap
When la fiche /cohortes/:cohorteId/stagiaires/:stagiaireId s'ouvre
Then il voit l'entête avec les KPIs personnels
And la timeline chronologique inverse (plus récent en premier) de tous les événements
And des filtres par type (livrables / quiz / alertes / commentaires / audit)
And un panneau d'actions toujours visible à droite (desktop) ou en bas de page (mobile)
```

**Scénario 2 — Action override de livrable**

```gherkin
Given un formateur consulte la fiche d'un stagiaire bloqué sur le livrable semaine 5
When il clique sur l'événement "livrable S5" puis sur "Override"
Then une modale propose "valider manuellement" ou "étendre l'échéance de N jours"
And il doit fournir un motif (texte libre obligatoire ≥5 caractères)
When il valide
Then l'override est enregistré avec son user_id, le motif, le timestamp
And l'événement apparaît dans la timeline et dans l'audit log
And le stagiaire reçoit une notification "votre livrable S5 a été validé par <formateur>"
```

**Scénario 3 — Audit log filtrable**

```gherkin
Given un formateur ouvre l'onglet "Audit log" de la fiche
When il filtre par type d'action "override" sur les 30 derniers jours
Then il voit uniquement les overrides effectués par n'importe quel formateur
And chaque entrée affiche : qui, quand, quoi, motif, lien vers l'événement source
```

## Cas limites à traiter

- Stagiaire avec 200+ événements timeline : pagination cursor-based (50 par page) ou virtualisation
- Stagiaire désactivé : entête avec badge « désactivé », actions destructives grisées, lecture seule
- Stagiaire transféré entre cohortes : timeline de l'ancienne cohorte conservée mais avec marqueur visuel
- Pas d'activité récente (>14j) : bannière « inactif » avec CTA « planifier un appel » mis en avant
- Override d'urgence en mode hors-ligne : message clair « connexion requise pour cette action »
- Conflit d'édition (deux formateurs commentent en même temps) : optimistic UI avec résolution last-write-wins + toast info

## Dépendances

- **Bloqué par** : CUR-85 (ST-13.2 heatmap) — point d'entrée principal ; CUR-67 (ST-08.4 audit log unifié) — source vérité historique
- **Bloque** : Aucun — vue terminale formateur

## Non-goals

- Pas d'édition profil stagiaire par le formateur (le stagiaire gère son profil via CUR-37)
- Pas de visio intégrée ; lien externe Calendly/Meet uniquement
- Pas de chat synchrone (post-MVP) ; commentaires asynchrones uniquement
- Pas de modification rétroactive des XP/badges (intentionnellement immuables pour l'intégrité du système)

## Tests à écrire

- **Unit** : composable `useStagiaireTimeline` avec merge trié d'événements multi-sources ; validation form override (motif obligatoire)
- **Integration** : endpoint timeline retourne union correcte triée, pagination cursor stable ; override crée une entrée audit log ; RLS bloque accès formateur d'une autre cohorte
- **E2E (Playwright)** : navigation heatmap → fiche → timeline filtrée → override avec motif → vérification audit log et notification stagiaire
- **A11y (axe-core)** : timeline avec `<ol>` sémantique, modale override avec focus trap et aria-labelledby, actions clavier-accessibles, contraste
- **Security** : tentative d'override sans être formateur de la cohorte = 403 ; motif XSS (sanitization) ; rate limit sur les overrides (max 20/heure/formateur)

## Observabilité

- **Logs** : `stagiaire.fiche.viewed` (level=info, payload={formateurId, stagiaireId, cohorteId}) ; `stagiaire.override.applied` (level=warn, payload={action, motif, target_event_id}) ; `stagiaire.action.failed` (level=error)
- **Métriques** : compteur `stagiaire_actions_total{action_type}` ; histogram `timeline_load_duration_ms` ; compteur `stagiaire_overrides_total{type='validate'|'extend'}`
- **Traces** : span `stagiaire.timeline.fetch` autour de la requête union
- **Alertes** : si `stagiaire_overrides_total` augmente >3 sigmas vs baseline sur 24h, alerte (potentiel abus ou bug système déclenchant override en cascade)

## Considérations sécurité

- RLS strict : seul un formateur de la cohorte du stagiaire peut accéder à la fiche
- Override require POST + CSRF token + motif non vide ; chaque override est immuable dans l'audit log
- Rate limit : 20 actions destructives par heure par formateur pour limiter le blast radius en cas de compte compromis
- Sanitization du motif (DOMPurify) ; jamais render en innerHTML
- Audit log inaltérable (table append-only avec triggers anti-update/delete)

## Considérations accessibilité

- Timeline en `<ol>` sémantique avec dates en `<time>` machine-readable
- Filtres comme groupe radio/checkbox accessible (fieldset + legend)
- Modale override avec focus trap, fermable Escape, aria-modal="true", aria-labelledby pointant sur titre
- Annonce aria-live des actions réussies (« override appliqué »)
- Contraste AAA sur les badges de statut (alertes, retards) pour visibilité prioritaire

## Considérations performance

- Timeline : pagination cursor-based 50 événements par page, infinite scroll
- Requête union pré-triée côté DB (UNION ALL + ORDER BY index) plutôt que côté app
- KPIs personnels en vue matérialisée rafraîchie event-driven
- Budget LCP 2s sur stagiaire avec 200 événements
- Lazy-load des détails d'événements (le contenu d'un livrable n'est chargé qu'au clic)
