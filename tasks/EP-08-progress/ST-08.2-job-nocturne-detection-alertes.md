## Contexte business

Détection automatique de dérives sans intervention manuelle. Sans ce job, le formateur découvre les retards quand c'est trop tard. C'est le pilier "suivi formateur asynchrone" matérialisé.

Valeur livrée : 3 règles d'alertes automatiques (pas de soumission 48h, quiz ratés, submissions répétées), idémpotence sur jour, ajout au digest formateur.

## Description fonctionnelle

Cron job (Inngest scheduled function) qui tourne 1× par jour à 6h. Scanne les progressions et applique des règles. Crée des `Alert` idémpotentes (même source = update timestamp, pas duplication).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Règles d'alertes**

```gherkin
Given le job tourne à 6h chaque matin
When il scanne les progressions
Then pour chaque règle, il crée une alerte si déclenchée :
  - Pas de soumission depuis 48h sur livrable en cours → kind="no_submission"
  - 2 quiz consécutifs ratés → kind="quiz_failures"
  - 5+ submissions échouées sur même livrable → kind="repeated_failures"
And les alertes sont ajoutées au digest formateur du lendemain
```

**Scénario 2 — Idémpotence**

```gherkin
Given une alerte existe déjà (même kind, même source) ouverte
When le job re-tourne
Then on ne duplique pas, on update le timestamp
```

**Scénario 3 — Job en panne**

```gherkin
Given le job échoue (DB indisponible)
Then une alerte ops est levée
And le job retry exponentiellement
```

## Cas limites à traiter

- **Stagiaire en vacances déclarées** : skip les règles (à intégrer si feature dispo).
- **Job qui tombe en panne** : monitor + retry exponential.
- **Job qui prend trop de temps** (> 30 min) : alert ops.
- **Cohortes archivées** : skip.

## Non-goals (hors scope)

- Pas de règles configurables par cohorte au MVP.
- Pas de notification temps réel (digest journalier suffit).
- Pas de ML pour prédiction.

## Tests à écrire

- **Unit** : chaque règle déclenche au bon seuil.
- **Unit** : idémpotence (2 runs → 1 seule alerte).
- **Integration** : job complet sur des données fixtures.
- **Resilience** : DB indisponible → retry.

## Observabilité

- **Métriques** : `alerts.created.total` par kind.
- **Logs** : `cron.detect_alerts.started`, `cron.detect_alerts.completed` (avec count).
- **Alertes** : si job échoue 2j consécutifs = critical.

## Considérations sécurité

- Job run avec service_role (pas user context).
- Pas de PII dans les logs.
- Audit log des alertes créées.

## Considérations performance

- Scan : batch par cohorte (pas N+1).
- Cible : < 5 min pour 100 cohortes × 30 stagiaires.

## Definition of Done spécifique

- 3 règles testées.
- Idémpotence vérifiée (2 runs consécutifs).
- Job tourne sur 10 cohortes fixtures.
