## Contexte business

Toute action sensible doit laisser une trace. Audit log = source de vérité légale (RGPD), debug, conformité. Sans audit, impossibilité de prouver qui a fait quoi.

Valeur livrée : trace immuable de toutes les actions sensibles (override, création cursus, suppression compte, etc.) avec before/after, contexte (IP, user agent), rétention 12 mois.

## Description fonctionnelle

Table `audit_log` unifiée pour tous les événements importants. Format event sourcing léger : actor_id, action_type, target_type, target_id, before_json, after_json, ip, user_agent, timestamp.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Trace d'action sensible**

```gherkin
Given une action sensible est effectuée
When elle complète
Then une entrée audit est créée avec : actor_id, action_type, target_type, target_id, before_json, after_json, ip, user_agent, timestamp
And l'entrée est immuable (RLS deny UPDATE/DELETE)
```

**Scénario 2 — Consultation admin**

```gherkin
Given un admin consulte l'audit log via SQL
When il filtre par utilisateur ou par action
Then les entrées correspondantes s'affichent dans un format lisible
```

**Scénario 3 — Rétention 12 mois**

```gherkin
Given le job de rétention tourne
When une entrée a > 12 mois
Then elle est archivée ou supprimée selon politique RGPD
```

## Actions tracées

- Création/édition/suppression cursus, cohorte, module
- Création/désactivation utilisateur
- Override harnais (ST-06.5)
- Émission certificat (ST-10.2)
- Décalage planning (ST-04.4)
- Modification rôles
- Suppression compte
- Login échoué (5+ tentatives)
- Désactivation 2FA

## Cas limites à traiter

- **before_json contient PII** : appliquer un mask (email hashé, password jamais).
- **Volume audit log** : partitionner par mois en v1.x.
- **Audit log lui-même manipulé** : RLS `false` pour UPDATE/DELETE — immuable.
- **Job rétention en panne** : alerte critique.
