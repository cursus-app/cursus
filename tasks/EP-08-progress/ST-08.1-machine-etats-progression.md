## Contexte business

État central du système, source de vérité de "où en est le stagiaire". 7 statuts : `à_venir`, `en_cours`, `soumis`, `validé`, `bloqué`, `en_alerte`, `en_retard`. Sans state machine claire, des bugs subtils minent la confiance.

Valeur livrée : machine à états documentée + testée, transitions invalides bloquées, traces complètes via audit log.

## Description fonctionnelle

Table `progressions` (user_id, module_id, status, due_date, ...). State machine documentée dans `docs/state-machines/progression.md`. Module `progressionStateMachine.ts` (xstate ou maison).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Statuts valides**

```gherkin
Given un stagiaire est sur le module M
When la machine à états calcule son statut
Then status ∈ { à_venir, en_cours, soumis, validé, bloqué, en_alerte, en_retard }
```

**Scénario 2 — Transition soumis → validé**

```gherkin
Given un livrable est soumis
When la submission est créée
Then status passe de "en_cours" à "soumis"

Given le harnais valide
When le webhook arrive
Then status passe de "soumis" à "validé"
```

**Scénario 3 — Détection retard**

```gherkin
Given un livrable est en retard de 48h
When le job nocturne tourne
Then status passe à "en_alerte"
```

**Scénario 4 — État terminal**

```gherkin
Given le statut atteint un état terminal (validé, validé_override)
When les conditions changent
Then plus de transition possible (immuable)
```

## Cas limites à traiter

- **Transition invalide tentée** : refus + log security.
- **Race condition** (2 updates simultanés) : transaction + version optimiste.
- **Restart d'un module validé** : impossible sauf admin avec audit log.

## Non-goals (hors scope)

- Pas d'historique fin des transitions au MVP (audit log fait office).

- Pas de rollback automatique.

## Tests à écrire

- **Unit** : toutes transitions valides (matrice).
- **Unit** : toutes transitions invalides rejetées.
- **Concurrency** : 2 updates simultanés → 1 seul gagne.
- **Integration** : transitions persistées cohéremment.

## Observabilité

- **Métriques** : distribution des statuts par cohorte.
- **Logs** : `progression.transitioned` (avec from → to).
- **Alertes** : pic de "bloqué" ou "en_alerte" sur une cohorte.

## Considérations sécurité

- RLS : stagiaire voit sa progression, formateur voit ses cohortes.
- Audit log sur chaque transition.
- Pas de bypass coté client (transitions server-side).

## Considérations performance

- Index sur (user_id, module_id, status).
- State check : < 10ms.
