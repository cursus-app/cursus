## Contexte business

L'utilisateur doit pouvoir gérer son profil : avatar, nom, bio courte, préférences de notification, langue, thème, suppression de compte. Story de base, RGPD-compliant (droit à l'oubli).

Valeur livrée : auto-administration sans support ticket. Conformité RGPD avec suppression de compte en self-service.

## Description fonctionnelle

Page `/profil` avec édition inline. Stockage avatar dans Supabase Storage avec redimensionnement côté serveur. Préférences thème/langue/notifs. Flow de suppression avec confirmation forte (email + password), purge sous 30 jours.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Édition inline avec autosave**

```gherkin
Given un user est sur /profil
When il édite son nom et clique en dehors
Then la modification est sauvegardée automatiquement (debounce 500ms)
And un toast "Modifié" apparaît brièvement (2s)
And la valeur est persistée en DB
```

**Scénario 2 — Upload avatar valide**

```gherkin
Given il upload un avatar
When le fichier est valide (PNG/JPG, < 2 Mo, < 2000×2000 px)
Then l'avatar est uploadé sur Supabase Storage
And remplace l'ancien (qui est supprimé par job de purge)
And l'image est servie via Supabase Image Transform (resize à 256px)
```

**Scénario 3 — Suppression de compte**

```gherkin
Given il clique "Supprimer mon compte"
When la confirmation est validée (saisie email + mot de passe)
Then le compte est marqué pour suppression sous 30 jours (soft-delete avec `deletion_scheduled_at`)
And il est déconnecté immédiatement
And un email "Suppression programmée" est envoyé avec lien d'annulation
```

## Cas limites à traiter

- **Avatar trop gros / format invalide** : message clair côté client + refus serveur.
- **Suppression d'un compte admin unique** : refus avec message "Au moins 1 admin requis".
- **Tentative de suppression d'un compte avec capstone non validé** : message d'avertissement (perte des progrès).
- **Annulation de suppression** dans les 30j : possible via email.
- **Avatar avec EXIF localisation** : stripé côté serveur (privacy).

## Observabilité

- **Logs** : `user.profile.updated`, `user.avatar.uploaded`, `user.deletion.scheduled`.
- **Métriques** : `users.avatars.uploaded.total`, `users.deleted.total`.
- **Alertes** : pic de suppressions = potential issue.

## Considérations sécurité

- **RLS** : `users` policy = user peut modifier uniquement sa propre row.
- **Upload** : validation MIME réelle (pas extension), max 2 Mo.
- **EXIF stripping** : obligatoire (privacy : pas de géoloc dans les avatars).
- **Suppression** : double confirmation (password + email), audit log.
- **Pas d'XSS** sur bio : sanitize HTML (DOMPurify ou Markdown safe).

## Considérations accessibilité

- Tous les labels associés aux inputs.
- Avatar : alt text "Photo de profil de \[Nom\]".
- Erreurs annoncées via `aria-live`.
- Toast "Modifié" : `aria-live="polite"`.

## Considérations performance

- Upload avatar : multipart, progress visible.
- Page profil : LCP < 1s (données déjà dans store Pinia).
