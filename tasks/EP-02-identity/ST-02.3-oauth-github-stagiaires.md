## Contexte business

Le harnais s'appuie sur les repos GitHub des stagiaires. Connecter GitHub est donc **obligatoire** pour un stagiaire. Sans lien GitHub, pas de soumission possible et pas de validation automatique. C'est une étape clé de l'onboarding (étape 3 sur 4).

Valeur livrée : connexion GitHub en 2 clics via OAuth Supabase, `github_handle` stocké sur le profil, bouton de reconnexion si token expiré.

## Description fonctionnelle

OAuth GitHub via Supabase Auth (Provider GitHub). Stockage du `github_handle` sur le profil utilisateur. Vérification que le compte reste connecté (token valide). Page d'onboarding étape 3 : "Connecter mon GitHub". Middleware serveur : bloque `/api/submissions` si pas de github_handle.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Connexion GitHub réussie**

```gherkin
Given un stagiaire en cours d'onboarding atteint l'étape 3
When il clique sur "Connecter mon GitHub"
Then il est redirigé vers GitHub OAuth
And il accorde les permissions (read:user, read:public_repo)
And il revient sur Cursus avec son github_handle stocké
And un toast "GitHub connecté ✅" s'affiche
```

**Scénario 2 — Soumission bloquée sans GitHub**

```gherkin
Given un stagiaire n'a pas connecté son GitHub
When il essaie de soumettre un livrable
Then une modale s'affiche "Connecte ton GitHub pour pouvoir soumettre"
And il est redirigé vers la page de connexion GitHub
```

**Scénario 3 — Changement de compte GitHub**

```gherkin
Given un stagiaire a son GitHub connecté (handle: "karim-d")
When il reconnecte avec un autre compte (handle: "karim-dev")
Then l'ancien handle est remplacé
And ses anciennes submissions restent attachées à lui (par user_id, pas par handle)
And un log audit trace le changement
```

## Cas limites à traiter

- **Refus des permissions OAuth** : retour avec erreur lisible "GitHub est requis pour soumettre tes livrables".
- **Compte GitHub privé / suspendu** : message clair, pas de crash.
- **Compte GitHub déjà utilisé par un autre stagiaire** : refus avec message "Ce GitHub est déjà lié à un autre compte Cursus".
- **Token GitHub expiré** : bouton "Reconnecter" sur le profil.
- **Compte GitHub avec handle qui change** (rare) : webhook GitHub non utilisé au MVP — vérification à chaque soumission.

## Sous-tâches techniques

- **TT-02.3.1** — Créer OAuth App GitHub (org `cursus-app`) + callback URLs
- **TT-02.3.2** — Configurer le provider GitHub dans Supabase Auth dashboard
- **TT-02.3.3** — Page d'onboarding étape 3 avec bouton "Connecter GitHub"
- **TT-02.3.4** — Stockage `github_handle` sur table `users` (UNIQUE constraint)
- **TT-02.3.5** — Middleware côté serveur : bloquer `/api/submissions` si pas connecté
- **TT-02.3.6** — Bouton "Reconnecter" en cas de token expiré (visible sur /profil)

## Non-goals (hors scope)

- Pas d'accès aux repos privés du stagiaire (lecture publics suffit pour MVP).
- Pas d'OAuth GitHub pour les formateurs au MVP (utile mais pas critique).
- Pas de sync automatique des repos de l'utilisateur.

## Tests à écrire

- **Integration** : OAuth callback crée bien le lien et stocke `github_handle`.
- **E2E (Playwright)** : parcours onboarding complet avec GitHub mocké (MSW).
- **Security** : un user ne peut pas s'attribuer le handle d'un autre (DB UNIQUE + tentative manuelle).
- **Unit** : middleware refuse submission si pas de handle.

## Observabilité

- **Logs** : `auth.github.connected`, `auth.github.disconnected`, `auth.github.expired`.
- **Métriques** : `auth.github.connected.total`, taux d'abandon sur l'étape OAuth (cible < 10%).
- **Alertes** : si taux d'abandon étape OAuth > 20%, investiguer UX.

## Considérations sécurité

- Scopes minimaux : `read:user`, `read:public_repo`.
- Pas de `repo` (accès privé) ni `write:*` au MVP.
- Token OAuth stocké chiffré côté Supabase (Vault).
- UNIQUE constraint sur `users.github_handle` : impossibilité d'usurpation.
- Validation du handle au callback (regex GitHub valid).

## Considérations accessibilité

- Bouton "Connecter GitHub" : label clair, icône + texte, focus visible.
- État connecté/déconnecté annoncé (`aria-live`).

## Considérations performance

- Page onboarding étape 3 : LCP < 1.5s.
- Callback OAuth : redirection sous 300ms après GitHub return.

## Definition of Done spécifique

- Au moins 3 stagiaires test ont fait l'OAuth end-to-end.
- Test "compte GitHub déjà utilisé" → refus clair.
- Middleware bloque bien `/api/submissions` sans handle.
