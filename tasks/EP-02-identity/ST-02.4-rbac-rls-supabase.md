## Contexte business

Pilier sécurité du produit. 4 rôles : `stagiaire`, `formateur_principal`, `co_formateur`, `admin`. Chaque rôle a des permissions strictes. Sans RBAC + RLS, une fuite de données inter-cohortes serait catastrophique pour la confiance.

Valeur livrée : isolation forte entre cohortes (un stagiaire ne voit que sa cohorte), permissions claires par rôle, double couche de sécurité (middleware Nuxt 4 + RLS Supabase = dernière ligne de défense).

## Description fonctionnelle

Le rôle est stocké dans `memberships` (par utilisateur × cohorte) + `users.global_role` pour admin. Toutes les tables sensibles ont RLS active avec policies par rôle. Composable `usePermission()` côté client pour masquer les actions interdites (UX). Middleware Nuxt `auth.global.ts` qui charge le rôle au login. Tests RLS exhaustifs.

## Coexistence Prisma 7 ↔ RLS Supabase

Prisma 7 utilise le rôle Postgres applicatif `cursus_app` (non-superuser) → la RLS reste effective comme défense en profondeur. Les middlewares Nuxt 4 appliquent en parallèle la logique RBAC côté application. Le client Supabase JS (Auth, Storage, Realtime) reste utilisé pour les flux où la RLS est naturellement appliquée via JWT utilisateur.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Isolation inter-cohortes (RLS test négatif)**

```gherkin
Given un stagiaire de la cohorte A essaie d'accéder à une submission de la cohorte B
When il fait la requête (même en bypass du middleware)
Then la RLS Supabase bloque (zéro ligne renvoyée)
And un log security est émis
```

**Scénario 2 — Formateur principal voit ses cohortes**

```gherkin
Given un formateur principal de la cohorte A consulte ses stagiaires
When il fait la requête
Then il voit uniquement les stagiaires de SES cohortes
And les autres cohortes sont invisibles
```

**Scénario 3 — Co-formateur limité par modules**

```gherkin
Given un co-formateur est rattaché uniquement au module M2
When il consulte la liste des livrables
Then il voit uniquement les livrables du module M2
And il ne voit pas les alertes des autres modules
```

**Scénario 4 — Admin accès global**

```gherkin
Given un admin se connecte
When il accède à n'importe quelle page
Then il voit tout (à l'exception des données ultra-sensibles : voir audit log immutable)
```

**Scénario 5 — Tentative d'élévation de privilège**

```gherkin
Given un utilisateur tente d'élever ses privilèges en modifiant son rôle dans le payload
When il fait la requête
Then la RLS sur la table users bloque toute modification de role par lui-même
And un log security critique est émis
```

## Cas limites à traiter

- **Utilisateur multi-cohortes** : peut être stagiaire dans A et co-formateur dans B simultanément.
- **Désactivation d'un membership** : pas de suppression, soft-delete avec `deactivated_at` ; RLS exclut les memberships désactivés.
- **Membership révoqué pendant qu'une requête est en cours** : RLS check à chaque requête (pas de cache côté serveur des permissions).
- **Admin unique désactivé** : refus avec message "Au moins 1 admin actif requis".
- **Rôle** `global_role` admin vs `memberships.role` stagiaire : admin gagne.

## Non-goals (hors scope)

- Pas de rôles custom au MVP (juste les 4 fixés).
- Pas de scope plus fin (lecture seule sur tel objet) au MVP.
- Pas de délégation temporaire de permissions.
- Pas de visualisation graphique des permissions (admin dashboard).

## Tests à écrire

- **Tests RLS** : pour chaque table sensible, vérifier qu'un user d'une autre cohorte ne peut PAS lire / écrire (matrix table×rôle).
- **Tests middleware** : route `/admin` redirige si pas admin.
- **Security** : modification du rôle dans le JWT côté client → rejet serveur.
- **Integration** : composable `usePermission()` retourne les bonnes réponses pour chaque rôle.
- **E2E** : un stagiaire ne voit jamais le menu admin.
- **Prisma** : test que les requêtes via `cursus_app` respectent les RLS (ne bypass pas).

## Observabilité

- **Logs** : tentatives d'accès non autorisé (`auth.permission.denied` level warn).
- **Métriques** : taux de 403 par endpoint (alert si > 5%).
- **Alertes security** : tentative explicite d'élévation de privilège → alert critique.

## Considérations sécurité

- **RLS = dernière ligne de défense** : même si bug app, RLS bloque.
- **Policies "deny by default"** : explicit grants seulement.
- **Pas de** `service_role` côté client (jamais).
- **JWT claims** validés côté serveur (pas de confiance dans le token coté client).
- **Audit log immuable** : RLS policy `false` pour UPDATE/DELETE même pour admin.
- **Rôles stockés côté serveur uniquement**, jamais dans localStorage.
- **Rôle Prisma** `cursus_app` : non-superuser, RLS active sur ses requêtes.

## Considérations accessibilité

- Menu et actions masquées selon rôle (pas "disabled" mais "hidden" si vraiment non accessible).
- Messages d'erreur 403 clairs et actionnables.

## Considérations performance

- Policies RLS indexées (index sur `user_id`, `cohorte_id`, `role`).
- Permission check côté serveur : < 10ms.
- Composable `usePermission()` : pure function, pas de roundtrip.

## Definition of Done spécifique

- Matrix RLS testée exhaustivement (au moins 20 tests négatifs).
- Audit sécurité manuel par un dev externe (pair review).
- Documentation des rôles dans `docs/security/rbac.md`.
