## Contexte business

Le mode d'inscription est **par invitation uniquement** (anti-fraude, anti-spam). L'utilisateur invité crée son compte avec un mot de passe fort, confirme son email, et peut se connecter. C'est le premier rempart de sécurité du produit — sans cela, n'importe qui pourrait s'inscrire et noyer la base.

Valeur livrée : un parcours signup/login professionnel, sécurisé par construction (validation forte de mot de passe, captcha après tentatives, anti-bruteforce), et UX-friendly (force du mot de passe en temps réel).

## Description fonctionnelle

Pages `/login`, `/signup?token=...`, `/forgot-password`, `/reset-password?token=...`. Backed par Supabase Auth. Pas de signup libre sans token d'invitation. Validation Zod côté client (vee-validate) et serveur. Indicateurs de force de mot de passe (5 critères visuels). Rate limiting + captcha après 5 échecs.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Pas de signup libre**

```gherkin
Given un utilisateur arrive sur /signup sans token
When il accède à la page
Then il est redirigé vers /login avec un message "L'inscription est sur invitation uniquement"
```

**Scénario 2 — Signup avec token valide + mdp fort**

```gherkin
Given un utilisateur arrive sur /signup avec un token valide
When il soumet email + mot de passe (12+ char, majuscule, minuscule, chiffre, symbole)
Then son compte est créé en statut "non confirmé"
And un email de confirmation est envoyé
And il est redirigé vers une page "Vérifie ton email"
```

**Scénario 3 — Mot de passe faible refusé en temps réel**

```gherkin
Given le mot de passe ne respecte pas les règles
When l'utilisateur tape "abc"
Then les indicateurs de force du mdp s'affichent en temps réel (5 critères, code couleur)
And le bouton "Créer mon compte" reste désactivé tant que tous les critères ne passent pas
```

**Scénario 4 — Email déjà utilisé**

```gherkin
Given l'email est déjà utilisé pour un compte existant
When l'utilisateur soumet
Then une erreur claire s'affiche : "Cette adresse est déjà associée à un compte. Connecte-toi ou récupère ton mot de passe."
```

**Scénario 5 — Confirmation par email**

```gherkin
Given l'utilisateur clique sur le lien de confirmation
When le token est valide et non expiré
Then son compte passe en "confirmé"
And il est redirigé vers la page d'onboarding
```

**Scénario 6 — Captcha après 5 échecs**

```gherkin
Given l'utilisateur a 5 tentatives de connexion échouées en 5 min
When il essaie la 6ᵉ
Then un captcha s'affiche (hCaptcha ou Cloudflare Turnstile)
And la 6ᵉ tentative n'est validée que si captcha OK
```

## Cas limites à traiter

- **Email avec accents ou caractères spéciaux** : autorisé, normalisation Unicode NFC.
- **Mot de passe avec espaces** : autorisé (pas de trim).
- **Token de confirmation expiré** (>24h) : message + bouton "renvoyer".
- **Réinscription après suppression de compte** : autorisée (après délai purge 30j).
- **Tentative SQL injection** : Zod valide strictement.
- **Email avec +alias** (`user+test@example.com`) : autorisé.

## Tests à écrire

- **Unit** : validation Zod mot de passe (cas 100 % couverts : trop court, sans majuscule, sans chiffre, etc.).
- **Unit** : composable `useAuth()` mocké (login success, login fail, reset).
- **Integration** : appel API signup → user créé en DB + email enqueued.
- **E2E (Playwright)** : parcours signup complet de A à Z.
- **Security** : tentative SQL injection sur email field → bloqué par Zod.
- **Security** : 6 tentatives login échouées → captcha apparaît.
- **A11y** : tab order cohérent, labels associés, erreurs annoncées aux lecteurs d'écran.

## Observabilité

- **Logs** : `auth.login.success`, `auth.login.failed`, `auth.signup.success`, `auth.password.reset_requested` (level info/warn).
- **Métriques** : taux d'échec login, latence p95 du login, taux de signup complétés vs abandonnés.
- **Sentry** : erreurs sur l'auth taggees `auth_flow`.
- **Alertes** : taux d'échec login > 30% en 5 min = potential attack.

## Considérations sécurité

- **Mot de passe** : 12+ char, majuscule, minuscule, chiffre, symbole (politique stricte).
- **Hash** : Supabase Auth utilise bcrypt nativement.
- **Rate limiting** : 5 tentatives / 5 min / IP + email.
- **Captcha** : Turnstile après 5 échecs.
- **No user enumeration** : messages génériques sur login fail ("Email ou mot de passe invalide", jamais "User non trouvé").
- **No timing attack** : comparaison bcrypt constante.
- **Reset token** : signed JWT, single-use, expiré à 24h.

## Considérations accessibilité

- Labels explicites sur tous les champs.
- Force du mot de passe annoncée via `aria-live="polite"`.
- Erreurs associées au champ via `aria-describedby`.
- Focus management : auto-focus sur email au load.
- Pas de contraste insuffisant sur les indicateurs de force.

## Considérations performance

- Endpoint login : p95 < 200ms hors bcrypt (bcrypt = ~100ms incompressible).
- Page login : LCP < 1.5s (chemin critique).

## Definition of Done spécifique

- Test de pénétration manuel : SQL injection, XSS sur email, enum users — tous bloqués.
- Audit a11y axe-core sur les 4 pages : 0 erreur critique.
- E2E parcours complet (signup → confirm → login) passe.
