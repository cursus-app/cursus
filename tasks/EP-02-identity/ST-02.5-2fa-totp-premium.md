## Contexte business

2FA TOTP **au MVP plutôt qu'en v1.1**, parce qu'on émet des certificats vérifiables : l'usurpation de compte formateur compromettrait l'intégrité du système (un attaquant pourrait valider un capstone et émettre un faux certificat). Premium feature qui améliore la confiance.

Valeur livrée : protection forte des comptes à risque élevé (formateurs, admins), codes de backup pour résilience, compatibilité avec apps standards (Google Auth, 1Password, Authy).

## Description fonctionnelle

2FA TOTP optionnel pour stagiaires, **fortement encouragé** pour formateurs et admin. App authenticator standard (Google Authenticator, 1Password, Authy). Activation depuis `/settings/2fa`. QR code scan + validation. 8 codes de backup générés à l'activation, chiffrés en DB. Step de saisie TOTP après email/mot de passe.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Activation 2FA**

```gherkin
Given un utilisateur active la 2FA depuis son profil
When il scanne le QR code avec son authenticator
Then les codes générés sont reconnus
And des codes de backup (8 codes à usage unique) sont affichés et téléchargeables
And la 2FA est activée pour son compte
And un email de confirmation est envoyé
```

**Scénario 2 — Login avec 2FA**

```gherkin
Given la 2FA est active
When l'utilisateur se connecte
Then après email/mot de passe, il doit saisir un code TOTP
And le code de 6 chiffres est validé (fenêtre 30s ± 1)
And le compteur d'usage du code est incrémenté (anti-replay)
```

**Scénario 3 — Récupération via code de backup**

```gherkin
Given un utilisateur a perdu son téléphone
When il utilise un code de backup
Then le code est consommé (jamais réutilisable)
And il est invité à réenrôler son authenticator immédiatement
And un email d'alerte est envoyé ("Code backup utilisé")
```

**Scénario 4 — Désactivation 2FA protégée**

```gherkin
Given un formateur essaie de désactiver la 2FA
When il clique sur "Désactiver"
Then il doit ressaisir son mot de passe + un code TOTP
And un email "2FA désactivée" est envoyé (alerte sécurité)
```

## Cas limites à traiter

- **Décalage horloge serveur/client** : fenêtre tolérante ± 30s (1 step).
- **Tous les backup codes consommés** : workflow de récupération via support / admin (manuel au MVP).
- **Code TOTP invalide 5x** : rate limit (lock 5 min).
- **Réutilisation d'un code TOTP encore valide** : refusé (anti-replay via counter).
- **Désactivation forcée par admin** : possible avec audit log + email user.

## Non-goals (hors scope)

- Pas de WebAuthn / Passkeys au MVP (v1.x).
- Pas de SMS 2FA (mauvaise pratique sécurité).
- Pas de notification push 2FA (Authy app).
- Pas d'enforcement 2FA obligatoire (juste recommandé) au MVP.

## Tests à écrire

- **Unit** : génération + validation TOTP (cas valide, expiré, replay).
- **Unit** : génération backup codes (8, uniques, format lisible).
- **E2E (Playwright)** : enrôlement 2FA + login avec 2FA + utilisation backup code.
- **Security** : tentative de bypass 2FA (skip step) → refus serveur.
- **Security** : code TOTP éculé (5 min après génération) → refus.

## Observabilité

- **Logs** : `auth.2fa.enrolled`, `auth.2fa.login_success`, `auth.2fa.backup_used` (warn), `auth.2fa.disabled` (warn).
- **Métriques** : `auth.2fa.enrolled.total`, taux d'utilisation 2FA chez formateurs (cible 80%).
- **Alertes** : formateur sans 2FA = rappel dans onboarding et alerte admin mensuelle.

## Considérations sécurité

- Backup codes : chiffrés en DB (pas hash, car affichage one-time à l'activation seulement).
- TOTP secret : stocké chiffré côté serveur, jamais exposé client après setup.
- Anti-replay : compteur d'usage stocké (rejet si re-soumission du même code).
- QR code : généré côté serveur, jamais logé ou conservé (one-time render).
- Désactivation : requiert mot de passe + code TOTP (double-check).

## Considérations accessibilité

- QR code accompagné du secret en clair (pour saisie manuelle si pas de caméra).
- Champ TOTP : autocomplete="one-time-code", inputmode="numeric".
- Codes backup : téléchargeables en TXT (lisible par lecteur d'écran).

## Considérations performance

- Validation TOTP : < 50ms (calcul HOTP).
- Setup 2FA : full flow < 1 min.

## Definition of Done spécifique

- Test réel avec Google Authenticator + 1Password + Authy : tous fonctionnent.
- Backup code utilisé puis bloqué (anti-replay).
- Mot de passe + TOTP requis pour désactiver.
