## Contexte business

Le formateur invite un stagiaire par email. Le mail contient un lien unique avec un token JWT signé qui pré-remplit le compte (email + cohorte + rôle). L'utilisateur n'a qu'à choisir un mot de passe. Sans cela, on devrait imposer un système de codes manuels, friction insupportable pour le formateur.

Valeur livrée : invitation en 1 clic côté formateur, onboarding en 30s côté stagiaire. Token signé = pas de risque d'usurpation d'identité.

## Description fonctionnelle

Endpoint `POST /api/invitations` qui génère le token et envoie l'email. Endpoint `GET /api/invitations/:token` qui valide le token et retourne le contexte (sans consommer). Endpoint `POST /api/invitations/:token/accept` (consommation usage unique). Page `/invite/:token` qui propose la création du compte. Token JWT signé avec `jose`, payload : `{email, cohorteId, role, expiresAt}`.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Génération d'invitation**

```gherkin
Given un formateur invite "karim@ex.com" à une cohorte
When la requête POST /api/invitations est faite
Then un token JWT signé est généré avec payload : { email, cohorteId, role, expiresAt }
And l'expiration est de 7 jours par défaut
And un email d'invitation est envoyé via EP-12
And l'invitation est stockée en DB (statut: pending)
```

**Scénario 2 — Accès à l'invitation**

```gherkin
Given Karim clique sur le lien dans l'email
When il arrive sur /invite/:token
Then le token est vérifié (signature + expiration)
And la page affiche "Bienvenue dans la cohorte X, créons ton compte"
And l'email est pré-rempli et non modifiable
```

**Scénario 3 — Consommation du token (usage unique)**

```gherkin
Given Karim soumet son mot de passe
When le compte est créé
Then le token est marqué consommé (usage unique)
And Karim est membre de la cohorte avec le rôle "stagiaire"
And il est redirigé vers l'onboarding
```

**Scénario 4 — Token expiré**

```gherkin
Given le token est expiré
When Karim arrive sur la page
Then un message "Cette invitation a expiré" s'affiche
And un bouton "Demander un nouveau lien" notifie le formateur
```

**Scénario 5 — Token déjà consommé**

```gherkin
Given le token est déjà consommé
When quelqu'un l'utilise une 2ᵉ fois
Then un message d'erreur clair + redirection vers /login
```

## Cas limites à traiter

- **Token signé avec une clé qui change** : tous les tokens existants invalidés (rotation gérée par version dans header).
- **Email du token diffère de l'email saisi** : erreur, token rejeté (email pré-rempli + non modifiable de toute façon).
- **Cohorte supprimée entre l'invitation et l'inscription** : message clair "Cohorte plus disponible, contacte ton formateur".
- **Invitation cliquée 2x simultanément** : seul le premier réussit (transaction).
- **Email non délivré** : bouton "Renvoyer" disponible 1h après envoi.

## Tests à écrire

- **Unit** : génération + vérification JWT (cas valide, cas signature invalide, cas expiré).
- **Unit** : transitions de statut invitation (state machine).
- **Integration** : accept invitation crée bien le `membership` et user.
- **Integration** : tentative de re-consommer un token = 410 Gone.
- **E2E (Playwright)** : parcours invitation → email → compte créé.
- **Security** : token forgé (clé incorrecte) → rejeté.
- **Security** : tentative de modifier `cohorteId` dans le payload côté client → détectée par signature.

## Observabilité

- **Logs** : `invitation.sent`, `invitation.accepted`, `invitation.expired`, `invitation.token_invalid` (warn).
- **Métriques** : `invitations.sent.total`, `invitations.accepted.total`, `invitations.expired.total`.
- **Time-to-accept (median)** : pour mesurer l'engagement (cible : < 4h).
- **Alertes** : taux d'expiration > 30% = problème délivrabilité email.

## Considérations sécurité

- **JWT signé** avec clé secrete forte (ES256 ou HS256, 256 bits min).
- **Rotation de clé** supportée via `kid` dans le header.
- **Single-use** : DB enforce (UNIQUE constraint sur token + statut).
- **Expiration** : 7 jours par défaut, configurable par formateur.
- **Pas de PII dans le token** au-delà de l'email (pas de nom, pas de téléphone).
- **Rate limit** : 100 invitations / formateur / h.

## Considérations accessibilité

- Page `/invite/:token` : labels clairs, focus auto sur password.
- États d'erreur (expiré, consommé) : message clair + alternative actionnable.

## Considérations performance

- Vérification JWT : < 10ms (jose est rapide).
- Endpoint accept : < 300ms p95 (création user + membership + log audit).

## Definition of Done spécifique

- Test de pénétration : token forgé rejeté, replay attack bloqué.
- Parcours E2E complet en < 2 min.
- Template email valide HTML + accessible.
