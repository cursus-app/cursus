## Contexte business

Le dashboard formateur (EP-13) montre des KPI ponctuels. Pour piloter véritablement une cohorte sur la durée (10-12 semaines), il faut des graphiques d'évolution : avancement moyen vs temps, taux de validation par module, drop-off cumulatif. Cette story livre les graphes premium qui élèvent le reporting de "chiffres" à "narratif visuel".

Business value : Differentiator fort, le formateur expert dispose des mêmes outils de pilotage qu'un produit data-driven (Mixpanel, Amplitude). Convaincant pour cycle de vente B2B.

## Description fonctionnelle

- Page `/cohortes/:id/reporting` (en plus du dashboard).
- Graphique 1 : line chart "Avancement moyen cohorte vs jalons théoriques" (axe X = jours, axe Y = % progression).
- Graphique 2 : stacked area "Répartition statuts soumissions / jour" (passed / failed / pending).
- Graphique 3 : sparkline par stagiaire (mini-line dans un tableau).
- Filtres : date range, modules.
- Toggle d'affichage : moyen / médian / quartiles.
- Annotations : événements clés (début capstone, fin semaine 4) marqués sur les graphes.
- Données : agrégation pré-calculée nuit (cron) + cache.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Avancement moyen
Given une cohorte avec 12 stagiaires en semaine 5
When le formateur ouvre la page reporting
Then un line chart montre l'avancement moyen (52 %) vs jalon théorique (50 %)
And un sparkline par stagiaire est dans le tableau
And les valeurs hover correctement (tooltip date + valeur)
```

```gherkin
Scénario 2 — Répartition statuts
Given 15 jours d'activité cohorte
When je consulte le stacked area
Then je vois les couches passed/failed/pending par jour
And cliquer sur une couche filtre la table en dessous
```

```gherkin
Scénario 3 — Filtre par module
Given la cohorte a 8 modules
When je filtre sur "Module 3 - Routing"
Then les graphes recalculent uniquement pour ce module
And les annotations re-positionnent
```

## Cas limites à traiter

- **Cohorte juste créée (pas de données)** : empty state avec illustration + "Pas encore de données — les graphes se rempliront dans 24 h".
- **Petites cohortes (< 3 stagiaires)** : afficher les valeurs individuelles plutôt que moyennes.
- **Dropouts** : continuer à afficher avec marqueur stoppage.
- **Date range très large (> 6 mois)** : agrégation hebdo plutôt que quotidienne.
- **Browser sans JS** : page degrade gracefully (table HTML server-rendered).
- **prefers-reduced-motion** : pas d'animation d'apparition des graphes.

## Tests à écrire

- **Unit Vitest** : agrégation SQL retourne bonnes valeurs sur fixture.
- **Integration** : endpoint retourne payload structuré correct.
- **A11y axe-core** : graphes ont une table équivalente accessible.
- **E2E Playwright** : naviguer, filtrer, voir mise à jour.
- **Visual Chromatic** : snapshots des 3 graphes (dark + light).
- **Performance** : page interactive < 1 s sur cohorte 50 stagiaires.
- **Security** : RLS, formateur d'une autre cohorte ne voit pas.

## Observabilité

- **Logs** : event `reporting.viewed` (info) avec `{ cohort_id, view: 'progress'|'submissions', user_id }`.
- **Métriques** : compteur `cursus.reporting.views.total`, p95 latency endpoint.
- **Insight** : pages reporting consultées > X fois / formateur → indicateur de valeur perçue.

## Considérations sécurité

- RLS strict.
- Pas d'agrégation cross-orgs.
- CSP : pas de scripts externes (charts en JS local).

## Considérations accessibilité

- Chaque graphe a un titre `<h3>` et un fallback `<table>` avec données accessibles.
- Couleurs avec palette accessible + symboles (pas couleur seule).
- Tooltip annoncé par `aria-live`.
- Navigation clavier des graphes (focus sur points clés).
- Annonce des changements de filtres.
- Respect `prefers-reduced-motion` (pas d'anim).

## Considérations performance

- Vue matérialisée → lectures rapides.
- Cache côté client (vue-query stale 5 min).
- Chart libs lazy-loaded.
- Cible : page interactive < 1 s pour cohorte de 50 stagiaires.
- Pas de re-render à chaque hover (debounce 50 ms).
