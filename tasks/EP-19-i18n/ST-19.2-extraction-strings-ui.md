## Contexte business

Une fois `@nuxtjs/i18n` installé (ST-19.1), la dette s'accumule à chaque composant qui code des strings en dur. Cette story est l'audit + extraction systématique : passer tous les composants existants et tous les emails/PDF/server messages en mode 100 % clé i18n. C'est la condition matérielle pour que ST-19.3 (traduction EN) ait quelque chose à traduire.

Business value : un produit non-extrait est non-traduisible. Sans cette story, EN reste théorique et la prochaine fonctionnalité ajoutera de la dette plus vite qu'on ne la rembourse.

## Description fonctionnelle

Parcours de tous les fichiers `pages/**`, `components/**`, `layouts/**`, `composables/**`, `server/**` à la recherche de strings affichables. Pour chaque string : créer une clé hiérarchique (`module.section.action`, ex: `cursus.builder.module.add`), remplacer la string par `{{ t('cursus.builder.module.add') }}` ou `$t(...)`, ajouter la traduction FR dans `locales/fr.json` (EN reste vide, sera comblé par ST-19.3).

Livrable : `locales/fr.json` complet (estimation 400-800 clés au MVP), `locales/en.json` avec les mêmes clés en placeholders (`__TODO__`), zéro string visible non-traduite dans l'UI.

## Cas limites à traiter

- **Strings concaténées** : refactoriser en interpolation (`t('greet', { name })`) plutôt que `t('hello') + name`.
- **Pluralisation** : utiliser `t('items.count', count)` avec règle CLDR (zéro, un, autre).
- **Strings dans du JS pur** (toast, alert) : passer par le composable, pas du `console.log` brut.
- **Texte riche (avec balises)** : utiliser `<i18n-t keypath="...">` + slots pour éviter des balises HTML dans les strings.
- **Date / nombre / monnaie** : utiliser `$d()` et `$n()`, pas du `toLocaleString` direct.
- **Strings dynamiques (générées)** : si la source est interne (ex. status enum), créer une clé par valeur ; si externe (entrée utilisateur), pas de traduction.

## Sous-tâches techniques

- **TT-19.2.1** — Audit automatisé : script Node qui scanne `**/*.vue`, `**/*.ts` et liste les strings suspectes (heuristique : caractères accentués, > 3 mots).
- **TT-19.2.2** — Refactoriser composants atomes/molécules d'abord (Button, FormField, EmptyState).
- **TT-19.2.3** — Refactoriser pages MVP critique (dashboard stagiaire, dashboard formateur, builder cursus).
- **TT-19.2.4** — Refactoriser pages secondaires (admin, profil, notifications).
- **TT-19.2.5** — Refactoriser server (emails templates, PDF certificat, error messages JSON API).
- **TT-19.2.6** — Compléter `locales/fr.json` (400-800 clés organisées par namespace).
- **TT-19.2.7** — Compléter `locales/en.json` avec mêmes clés (valeurs `__TODO__`).
- **TT-19.2.8** — Test snapshot des deux fichiers et diff CI : toute clé ajoutée FR doit l'être EN.

## Dépendances

- **Bloqué par** : ST-19.1 (setup i18n).
- **Bloque** : ST-19.3 (traduction EN), ST-19.6 (lint rule), toute évolution UI future.

## Non-goals

- Pas d'extraction automatique 100 % (humain dans la boucle pour bonnes clés).
- Pas de gestion de TMS (Translation Management System) au MVP.
- Pas de support de plus de 2 locales.
- Pas de traduction EN à cette étape (story dédiée ST-19.3).

## Tests à écrire

- **Script CI** : `check-i18n.ts` qui parse les `t('...')` du code et vérifie présence dans `fr.json` + `en.json`. Échoue si une clé manque.
- **Unit Vitest** : composant clé `<Button>` rend bonne string en FR et en EN (vue de test sous chaque locale).
- **Visual Chromatic** : snapshots des écrans clés sous chaque locale.
- **E2E Playwright** : naviguer en FR et en EN, vérifier qu'aucun `__TODO__` n'apparaît jamais en FR.
- **Lighthouse** : pas de surcharge i18n sur les bundles principaux.

## Observabilité

- **Métriques** : taille des bundles locale (cible : `fr.json` < 50 KB gz, `en.json` < 50 KB gz).
- **CI** : nb de clés non utilisées (dead keys) reporté hebdo, à nettoyer.
- **Logs** : event `i18n.missing_key` (warn) si en runtime une clé manque (ne doit jamais arriver en prod après cette story).

## Considérations sécurité

- Pas d'injection : les paramètres d'interpolation sont des strings échappées par Vue.
- Pas de strings sensibles (tokens, secrets) dans les fichiers locales.

## Considérations accessibilité

- `aria-label`, `aria-describedby`, `placeholder`, `title` extraits aussi.
- Vérifier que la longueur EN n'écrase pas les layouts (textes EN parfois 30 % plus longs).
- Lecteurs d'écran annoncent bien la langue par bloc si une zone change de locale (rare, mais possible).

## Considérations performance

- `fr.json` et `en.json` split par namespace via Vite (chunk par route si lazy-load configuré).
- Cible bundle additionnel par locale : < 50 KB gz.
- Pas de re-render inutile : `t()` utilise reactivity Vue uniquement quand la locale change.
