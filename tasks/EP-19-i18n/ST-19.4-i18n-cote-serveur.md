## Contexte business

La locale UI seule ne suffit pas : un stagiaire FR doit recevoir ses emails Resend, son certificat PDF et les messages d'erreur d'API dans sa langue. Cette story livre l'i18n côté serveur (Nitro endpoints + jobs Inngest + emails + PDF) pour que la locale de l'utilisateur (stockée en DB sur `users.preferred_locale`) soit propagée à chaque rendu serveur.

Business value : cohérence totale FR/EN. Sans cette story, un user EN voit l'UI en EN mais reçoit un email "Bienvenue sur Cursus !" — friction et perte de crédibilité.

## Description fonctionnelle

Chaque rendu serveur qui produit du texte utilisateur (email Resend, PDF certificat, JSON error response) prend une locale en paramètre et utilise un instance i18n serveur (`@intlify/core` ou similaire). La locale est résolue dans l'ordre : (1) param explicite de l'API, (2) `users.preferred_locale` en DB, (3) fallback `fr`.

Livrables :

- Service `server/utils/i18n.ts` qui expose `tServer(locale, key, params)`.
- Refactor de tous les templates email (`emails/welcome.vue`, `emails/cohorte-start.vue`, etc.) pour accepter une `locale` prop et tirer leur copy depuis `useI18n` serveur.
- Refactor du template PDF certificat (`server/pdf/certificate.ts`) avec locale.
- Wrapper d'erreurs API : `respondError(event, 'auth.invalid_credentials')` qui produit `{ error: { code, message_fr, message_en } }` ou choisit selon locale du request.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Email en EN pour user EN
Given un stagiaire EN s'inscrit
When un email "welcome" est envoyé via Resend
Then le sujet est en EN ("Welcome to Cursus")
And le corps de l'email est en EN
And les CTAs sont en EN
```

```gherkin
Scénario 2 — Certificat PDF localisé
Given un stagiaire EN achève son cursus
When le PDF certificat est généré
Then tout le texte du PDF est en EN
And la date est au format M/D/YYYY (ou locale-appropriate)
And la signature reste typographique cohérente
```

```gherkin
Scénario 3 — Erreur API localisée
Given un client envoie POST /api/login avec mauvais identifiants depuis une session locale=en
When le serveur répond
Then { error: { code: "auth.invalid_credentials", message: "Invalid email or password" } }
And le code reste stable (api stable contract)
And un user FR aurait "Identifiants invalides"
```

## Cas limites à traiter

- **Email envoyé par job (sans contexte utilisateur)** : payload du job inclut `recipient_locale` pour éviter une lecture DB de plus.
- **PDF de certificat de cohorte mixte** (FR + EN stagiaires) : générer un PDF par stagiaire, dans sa locale.
- **Erreur générique 500** : message générique "Une erreur est survenue / An error occurred" + code de corrélation. Pas de traduction tardive d'une exception.
- **Webhook payload** (ST-22.5) : pas traduit, anglais structuré (consommateur machine).
- **Locale absente en DB** : fallback `fr`, pas de crash.

## Sous-tâches techniques

- **TT-19.4.1** — `server/utils/i18n.ts` : instance i18n serveur chargeant `locales/*.json`, expose `tServer(locale, key, params?)`.
- **TT-19.4.2** — Refactor templates emails (`emails/*.vue` via `@vue-email`) : props `locale` + clés i18n.
- **TT-19.4.3** — Refactor PDF certificat : passer `locale` au moteur de rendu, utiliser `tServer`.
- **TT-19.4.4** — Helper `respondError(event, errorKey, params?)` : retourne JSON avec message localisé selon `event.context.locale`.
- **TT-19.4.5** — Middleware Nitro `resolveLocale` : pose `event.context.locale` depuis (header `accept-language` ou `users.preferred_locale`).
- **TT-19.4.6** — Migration DB : colonne `users.preferred_locale` (default 'fr', check IN ('fr','en')).
- **TT-19.4.7** — Jobs Inngest : inclure `recipient_locale` dans le payload.
- **TT-19.4.8** — Tests d'intégration : render un email + un PDF + une erreur en FR et EN.

## Non-goals

- Pas d'i18n des logs serveur (toujours en EN structuré, machine-readable).
- Pas de traduction des audit log entries (timestamp + code event, pas de texte UI).
- Pas de support de date locale fantaisiste (toujours via `Intl.DateTimeFormat`).

## Tests à écrire

- **Unit Vitest** : `tServer('fr', 'common.save')` → "Enregistrer" ; `tServer('en', 'common.save')` → "Save".
- **Integration Nitro** : appel d'une route qui produit une erreur → message dans la locale du request.
- **Integration Resend** : render template `welcome` en FR puis EN → comparer snapshots HTML.
- **Integration PDF** : render certificat en FR puis EN → comparer texte extrait.
- **Migration** : `users.preferred_locale` défaut 'fr' pour anciens users.
- **Security** : tentative d'injection dans le param locale (`../`) → 400 (validation Zod stricte de l'enum).

## Observabilité

- **Logs** : event `i18n.server.render` (debug) avec `{ surface: 'email'|'pdf'|'api_error', locale, key_namespace }`.
- **Métriques** : compteur `cursus.i18n.server.renders.total` labellisé par `surface` et `locale`.
- **Alerte** : `i18n.missing_key` côté serveur (warn) → blocant en CI.

## Considérations sécurité

- Validation Zod du paramètre `locale` (enum strict).
- Pas d'injection dans les params d'interpolation (échappement Vue / `@intlify`).
- Pas de PII dans les clés (les valeurs OK car contenu user déjà connu).

## Considérations accessibilité

- Emails HTML : attribut `lang` sur `<html>` correct (FR ou EN).
- PDF : metadata PDF `Language` correcte (`/Lang (fr-FR)` ou `/Lang (en-US)`).
- Pas de mélange de langues dans un même paragraphe.

## Considérations performance

- Instance i18n serveur singleton (chargée 1 fois au boot, pas par requête).
- Pas d'I/O JSON par requête (locales en mémoire).
- Rendering PDF EN ≤ 1.5× FR (textes plus longs OK, mais on garde le PDF en 2-3 s p95).
