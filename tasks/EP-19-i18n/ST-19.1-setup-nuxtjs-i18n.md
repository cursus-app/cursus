## Contexte business

Cursus est conçu FR-first (audience initiale francophone, Mohamed pilote en France) mais l'ambition produit (01-vision §6) inclut une internationalisation propre : pas de hack `if (lang === 'en')` éparpillé, mais une architecture i18n posée dès le départ. Cette story installe `@nuxtjs/i18n` avec FR par défaut + EN comme deuxième locale, prête à recevoir les traductions (ST-19.2) et les locales serveur (ST-19.4).

Sans cette story, les locales restent codées en dur dans les templates, et chaque page ajoutée alourdit la dette i18n. La poser tôt évite une migration douloureuse 6 mois plus tard.

## Description fonctionnelle

Installer `@nuxtjs/i18n` (>= v8), configurer `nuxt.config.ts` avec `defaultLocale: 'fr'`, `locales: [{ code: 'fr', file: 'fr.json' }, { code: 'en', file: 'en.json' }]`, `strategy: 'prefix_except_default'` (URLs : `/cursus` pour FR, `/en/cursus` pour EN). Helper composable `useT()` (wrapper sur `$t`) avec auto-import. Fichiers `locales/fr.json` + `locales/en.json` initialisés avec un namespace minimum (`common`, `nav`, `errors`).

Intégration côté Nuxt : SSR i18n actif, hydration sans flash de langue (FOUL — Flash Of Untranslated Locale). Détection initiale via Accept-Language puis cookie `i18n_redirected`.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Pages FR par défaut
Given un visiteur arrive sur https://cursus.app/
When le navigateur n'envoie aucun Accept-Language
Then la page se charge en FR
And l'URL reste sans préfixe (/cursus, /login, etc.)
And le lang attribute du <html> est "fr"
```

```gherkin
Scénario 2 — Pages EN avec préfixe
Given un visiteur clique sur le sélecteur de langue → EN
When la page se recharge
Then l'URL est préfixée /en (ex: /en/cursus)
And le lang attribute du <html> est "en"
And un cookie i18n_redirected=en est posé
```

```gherkin
Scénario 3 — Use composable useT
Given un composant fait const { t } = useT()
When il rend t('common.save')
Then en FR il affiche "Enregistrer"
And en EN il affiche "Save"
And aucune string hardcodée n'apparaît
```

## Cas limites à traiter

- **Locale inconnue dans l'URL** (`/de/cursus`) : 404 propre (pas de fallback silencieux qui masquerait un bug).
- **Clé i18n manquante** : warn en dev, fallback sur la clé brute en prod (jamais d'erreur runtime).
- **Hydration mismatch** : forcer la locale serveur dans le HTML initial pour éviter FOUL.
- **Switch de locale en cours de session** : conserver la route équivalente (ex `/cursus/123` → `/en/cursus/123`).
- **Locale sur sous-domaine vs path** : décision actée → path (`strategy: prefix_except_default`), pas de sous-domaine.

## Non-goals

- Pas de gestion de pluralisation complexe au-delà du built-in (CLDR rules suffisent).
- Pas de namespace dynamique chargé à la demande (tous les bundles en build).
- Pas de support RTL au MVP (ar, he) — story dédiée si nécessaire.
- Pas d'outil de traduction managé (Crowdin, Lokalise) au MVP.

## Tests à écrire

- **Unit Vitest** : `useT('common.save')` retourne la bonne valeur FR/EN, fallback sur la clé si absente.
- **Integration Nitro** : `GET /` charge FR, `GET /en` charge EN.
- **E2E Playwright** : changer la langue via le sélecteur → URL et `<html lang>` changent, recharger conserve la langue.
- **A11y** : attribut `lang` correct sur `<html>` (axe-core check).
- **CI Lint (à venir ST-19.6)** : aucune string codée en dur dans les composants.

## Observabilité

- **Logs** : event `i18n.locale.switched` (info) avec payload `{ from, to, user_id, route }`.
- **Métriques** : compteur `cursus.i18n.locale.active.total` labellisé par `locale` (échantillonné sur première vue d'une session).
- **Alerte** : si > 5 % des sessions sont en EN sans qu'on l'ait promu, c'est un signal de demande utilisateur.

## Considérations sécurité

- Le cookie `i18n_redirected` est `SameSite=Lax`, `Secure`, pas de PII.
- Pas d'injection : les valeurs des locales sont des strings sérialisées, pas d'HTML.
- Validation Zod des clés i18n au build pour éviter accès à des chemins arbitraires.

## Considérations accessibilité

- Attribut `<html lang>` mis à jour à chaque switch (critère WCAG 3.1.1).
- Pages préfixées correctement par les lecteurs d'écran (voix change).
- Sélecteur de langue (ST-19.5) doit afficher l'endonyme ("Français", "English") + code ISO.
- Pas d'autoplay de switch de locale sans action utilisateur.

## Considérations performance

- Locales JSON chargées en lazy par route (pas tout en bundle initial).
- Bundle EN absent du chunk FR par défaut (tree-shaken).
- Cible : surcoût bundle initial < 5 KB gz pour le module i18n + dictionnaire `common`.
- Pas de re-rendu inutile au switch (Vue reactivity ciblée).
