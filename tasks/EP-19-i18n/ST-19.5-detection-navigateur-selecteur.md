## Contexte business

Les mécaniques i18n existent (ST-19.1 à 19.4) mais l'utilisateur a besoin d'un point d'entrée explicite pour choisir sa langue, et d'une première détection intelligente à l'arrivée. Cette story livre la détection navigateur (via `Accept-Language`) et le sélecteur UI (header/footer) pour basculer manuellement.

Business value : un visiteur anglophone tombe sur la home FR → si on lit son `Accept-Language: en-US`, on propose un switch (banner discret) → conversion EN +30 % vs forcer FR.

## Description fonctionnelle

**Détection navigateur** : à la première visite (pas de cookie `i18n_redirected`), serveur lit `Accept-Language`, match contre les locales supportées (`fr`, `en`), pose un cookie + redirige si nécessaire.

**Sélecteur UI** : composant `<LocaleSwitcher />` placé dans le header (desktop) et le menu mobile. Affiche l'endonyme ("Français" / "English") + drapeau optionnel décoratif. Au clic, switch la locale courante, repose le cookie, conserve la route équivalente.

**Banner d'invitation** : si la locale détectée diffère de la locale courante (ex: navigateur EN, page FR), afficher un banner top "View this page in English?" avec actions "Switch" / "Keep French". Dismiss persistant en cookie.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Détection EN au premier visit
Given un visiteur a Accept-Language: en-US
And il n'a pas de cookie i18n_redirected
When il arrive sur https://cursus.app/
Then un banner top apparaît "View in English?" avec actions Switch / Keep FR
And le banner peut être dismiss définitivement
```

```gherkin
Scénario 2 — Switch manuel via sélecteur
Given je suis sur /cursus/123 en FR
When je clique sur le sélecteur de langue → English
Then la page se recharge sur /en/cursus/123
And le cookie i18n_redirected=en est posé
And toute l'UI bascule en EN sans recharge supplémentaire si possible
```

```gherkin
Scénario 3 — Persistance entre sessions
Given j'ai déjà choisi EN dans une session précédente
When je reviens demain
Then la home s'affiche directement en EN
And aucun banner ne s'affiche
```

## Cas limites à traiter

- **Accept-Language complexe** (`fr-CA,en;q=0.9,es;q=0.5`) : parser correctement, prendre la première locale supportée.
- **User loggué avec preferred_locale en DB** : prime sur cookie et Accept-Language.
- **Switch sur page sans équivalent** : conserver la racine, ne pas crasher.
- **Banner conflictuel avec cookies bar** (ST-15.5) : empiler proprement, banner langue en-dessous du cookies banner.
- **Bots / SEO** : ne pas servir EN à un bot qui demande FR (canonical URL = locale demandée).
- **Cookie bloqué** : fallback sur localStorage, et si bloqué aussi → re-prompt au prochain visit (acceptable).

## Sous-tâches techniques

- **TT-19.5.1** — Middleware Nitro `detectLocaleOnFirstVisit` : si pas de cookie, parse `Accept-Language`, pose cookie.
- **TT-19.5.2** — Composant `<LocaleSwitcher />` (header + footer + mobile menu) avec dropdown accessible.
- **TT-19.5.3** — Composant `<LocaleSuggestionBanner />` (top of page, dismissible).
- **TT-19.5.4** — Logic switch route équivalente : helper `switchLocalePath(locale)` (built-in `@nuxtjs/i18n`).
- **TT-19.5.5** — Sync `preferred_locale` en DB si user loggué et change la locale (PATCH `/api/me`).
- **TT-19.5.6** — Test du parsing `Accept-Language` (lib `accept-language-parser`).
- **TT-19.5.7** — Animation de switch (fade 150ms) respectant `prefers-reduced-motion`.

## Dépendances

- **Bloqué par** : ST-19.1 à ST-19.4 (i18n posé), ST-18.4 (Banner, Dropdown).
- **Bloque** : aucune.

## Non-goals

- Pas de géo-IP detection (Accept-Language suffit).
- Pas de redirection automatique sans demande (toujours opt-in via banner).
- Pas de support de plus de 2 locales.
- Pas de mise à jour de `preferred_locale` côté admin (l'user le fait lui-même).

## Tests à écrire

- **Unit Vitest** : parser `Accept-Language` retourne bonne locale ; helper `switchLocalePath` conserve la query.
- **Integration Nitro** : `GET /` avec `Accept-Language: en` et pas de cookie → 200 + banner suggéré (pas redirect immédiat).
- **E2E Playwright** : (1) visiter en FR, switch EN, vérifier URL `/en` + cookie. (2) Recharger : pas de banner.
- **A11y axe-core** : sélecteur de langue navigable au clavier, banner avec rôle `region` et dismiss focusable.
- **Security** : tentative d'injection dans le cookie i18n_redirected → enum strict côté serveur.

## Observabilité

- **Logs** : event `i18n.banner.shown` (info), `i18n.banner.accepted` (info), `i18n.banner.dismissed` (info).
- **Métriques** : taux d'acceptation du banner (acceptés / shown) — cible > 30 % parmi les users dont la locale navigateur diffère de la page.
- **Funnel** : combien de switch EN vs total sessions EN.

## Considérations sécurité

- Cookie `i18n_redirected` : `SameSite=Lax`, `Secure`, `HttpOnly` non requis (lecture côté client OK).
- Validation Zod du payload PATCH `/api/me.preferred_locale` (enum `fr`|`en`).
- Pas de XSS via dropdown (Vue rend les strings échappées).

## Considérations accessibilité

- `<LocaleSwitcher />` : `<button aria-haspopup="listbox">`, navigation flèches haut/bas, Esc pour fermer.
- Banner : rôle `region` + `aria-label="Language suggestion"`, dismiss focusable.
- Drapeaux : `aria-hidden="true"` (décoratifs), endonyme reste la source de vérité.
- Annonce du switch via `aria-live="polite"` ("Language changed to English").

## Considérations performance

- Détection serveur : 1 ms supplémentaire par requête (parse cheap).
- Dropdown : pas de JS initial si pas ouvert (lazy).
- Banner : composant léger, mount conditionnel.
- Pas de re-render full page au switch — Vue gère via reactivity du store i18n.
