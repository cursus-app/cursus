## Contexte business

Mettre l'i18n en place (ST-19.1 à 19.5) ne suffit pas pour la durée : sans garde-fou automatique, la prochaine PR ajoutera des strings codées en dur et la dette repartira. Cette story installe la règle de lint qui détecte tout texte visible non extrait. C'est l'assurance long terme du pilier i18n.

Business value : on "verrouille" la qualité i18n du produit. Toute régression est bloquée en CI — pas découverte 3 mois plus tard sur un report bug client EN.

## Description fonctionnelle

Règle ESLint custom `no-hardcoded-strings` (ou réutilisation de `@intlify/eslint-plugin-vue-i18n`) qui détecte :

- Texte direct dans un template Vue (entre balises).
- Attributs `placeholder`, `title`, `alt`, `aria-label` codés en dur.
- Strings dans `defineProps`, `defineEmits`, `useToast`, alerts JS.

Exception : les valeurs techniques (clés objet, ID, classes CSS), les strings courtes en `[a-z0-9-_]+` (probablement techniques), et les fichiers `tests/`, `*.test.ts`. Niveau d'erreur : `error` en CI, `warn` en local pour ne pas bloquer le dev rapide.

Livrables : configuration ESLint + 1 pass de fix sur l'existant + doc des conventions.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — String hardcodée détectée en CI
Given une PR qui ajoute <button>Enregistrer</button> sans $t()
When la CI s'exécute
Then ESLint échoue avec error "Hardcoded string detected, use $t() or t()"
And la PR ne peut pas être mergée tant que la règle n'est pas respectée
```

```gherkin
Scénario 2 — Attribut hardcodé détecté
Given <input placeholder="Ton email" />
When le lint passe
Then erreur "placeholder must use $t()"
```

```gherkin
Scénario 3 — Faux positifs gérés
Given <div data-testid="submit-button">
When le lint passe
Then aucune erreur (attribut technique, pas affiché)
And les class CSS, id, slot names sont aussi exemptés
```

## Cas limites à traiter

- **Texte conditionnel (**`{{ count > 1 ? 'items' : 'item' }}`) : forcer `$t('items', count)` avec CLDR.
- **Strings dynamiques avec template literals** : si concaténation de strings UI, nécessite refactor en `$t('key', params)`.
- **Composants tiers (nécessitent littéraux)** : opt-out avec `// eslint-disable-next-line no-hardcoded-strings` + justification en commentaire.
- **Storybook stories** : exempté (les stories montrent des exemples avec strings).
- **Tests** : exempté.
- **Audit log messages** (techniques) : exempté (logs serveur en EN structuré).

## Sous-tâches techniques

- **TT-19.6.1** — Évaluer `@intlify/eslint-plugin-vue-i18n` (règles `no-raw-text`, `no-missing-keys-in-other-locales`).
- **TT-19.6.2** — Configurer `.eslintrc` avec niveau `error` pour PR + ignoré sur `tests/**`, `stories/**`.
- **TT-19.6.3** — Premier pass `eslint --fix` (semi-automatique) sur l'existant.
- **TT-19.6.4** — Documentation `docs/i18n/lint-rules.md` (règles, exemptions, justification d'un disable).
- **TT-19.6.5** — Pre-commit hook Lefthook : ESLint en `--max-warnings 0` sur les fichiers stagés i18n-sensibles.
- **TT-19.6.6** — GitHub Actions : workflow PR avec annotations ESLint en commentaire de PR.

## Dépendances

- **Bloqué par** : ST-19.1 (i18n installé), ST-19.2 (extraction faite, sinon règle inutilisable).
- **Bloque** : aucune fonctionnelle, mais améliore la qualité de toute story UI future.

## Non-goals

- Pas de règle exigeant la présence systématique d'une clé EN traduite (c'est ST-19.3 le check).
- Pas de vérification d'orthographe sur les valeurs des locales.
- Pas d'analyse stylistique des strings (juste "existe / n'existe pas").

## Tests à écrire

- **Unit (sur la règle)** : composer un fixture Vue avec et sans string hardcodée, vérifier que la règle déclenche bien.
- **CI** : lancer ESLint sur tout le repo, fail si erreur.
- **Integration** : PR de test qui ajoute une string hardcodée → CI échoue bien.
- **Documentation** : vérifier que les exemptions documentées marchent.

## Observabilité

- **CI** : compteur d'avertissements / erreurs ESLint i18n par PR — cible 0 erreurs main.
- **Métriques repo** : nombre d'`eslint-disable no-hardcoded-strings` (cible : < 10 sur tout le repo, justifiés).

## Considérations sécurité

- Pas de surface sécurité directe.
- Indirectement : pas de strings UI accidentellement sensibles (mots de passe en placeholder, etc.).

## Considérations accessibilité

- Encourage indirectement à traduire les attributs ARIA (`aria-label`, `aria-describedby`).
- Aucune régression a11y attendue.

## Considérations performance

- ESLint local : ajout négligeable (< 100 ms par fichier).
- CI : 1-2 s de plus sur la step lint (acceptable).
