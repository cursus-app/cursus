## Contexte business

Lors de la création d'un module (ST-03.x), le formateur doit rédiger ressources, exercices, etc. L'IA peut suggérer des ressources externes pertinentes (articles, vidéos, repos GitHub) basées sur le titre + objectifs du module. Le formateur les revue, conserve ou écarte. Cette story réduit le temps de création d'un module de 50 à 20 min.

Business value : améliore la qualité et la diversité du contenu pédagogique, baisse drastiquement le coup de création. Différenciateur fort vs LMS classiques.

## Description fonctionnelle

- CTA "Suggérer des ressources" sur l'éditeur de module (input : titre + objectifs + tags).
- Backend appelle un LLM qui retourne 5-10 candidats `{ title, url, kind: 'article'|'video'|'repo'|'doc', rationale, confidence }`.
- Validation Zod stricte, vérification que les URL sont absolues + accessibles HEAD (probabilité de non-hallucination).
- Le formateur voit la liste, peut "Ajouter" / "Ignorer" / "Modifier" chaque candidat. Conservation des ajoutés en DB.
- Métriques d'acceptation pour calibrer le prompt.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Suggestion sur un module "React Hooks"
Given un module avec titre "Les Hooks React", objectifs renseignés
When le formateur clique "Suggérer des ressources"
Then 5-10 ressources apparaissent en moins de 8 s
And chaque ressource a un titre, URL, kind, rationale
And les URLs ont été vérifiées HEAD (200 OK)
```

```gherkin
Scénario 2 — Ajout d'une suggestion
Given une liste de suggestions est affichée
When le formateur clique "Ajouter" sur "Article : React Hooks Deep Dive"
Then la ressource est ajoutée au module avec son kind/url/titre
And un event ai.resource.suggestion_accepted est tracé
```

```gherkin
Scénario 3 — URL inaccessible
Given le LLM propose une URL qui retourne 404 lors du HEAD check
When la validation passe
Then cette ressource est exclue du résultat affiché (ou marquée en "à vérifier")
```

## Cas limites à traiter

- **URL hallucinée** : HEAD check (5 s timeout) avant affichage — si 404, exclure.
- **Trop de suggestions identiques** (mêmes sites) : dédoublonner par domaine.
- **Quota** : limiter à N suggestions / formateur / jour (cf. ST-21.5).
- **Module en langue X** : suggérer dans la même langue.
- **Ressources payantes** : warning si domaine connu (medium.com, udemy.com, etc.).
- **Bias** : diversifier auteurs / domaines pour éviter monopole d'une source.

## Non-goals

- Pas de génération de ressource (article écrit par IA) au MVP.
- Pas de recommendation continue / push.
- Pas de scraping du contenu de l'URL au MVP (juste HEAD check).
- Pas de vérification de la licence du contenu.

## Tests à écrire

- **Unit Vitest** : Zod schema valide les URLs (https only), checkUrlReachable timeout.
- **Integration** : mock LLM → suggestions valides + HEAD checks parallel.
- **Integration** : URL 404 → exclue.
- **A11y axe-core** : liste navigable au clavier, icônes kind étiquetées.
- **E2E Playwright** : formateur clique "Suggérer" → voit liste → accepte 3 → vérifie persistance.
- **Quotas** : 11e génération bloquée.
- **Security** : URL injection (`javascript:`, `data:`) bloquée.

## Observabilité

- **Logs** : event `ai.resource.suggestion_generated` (info) avec `{ module_id, n_suggestions, latency_ms, tokens, cost_usd }`.
- **Logs** : `ai.resource.suggestion_accepted` (info), `ai.resource.suggestion_ignored` (info).
- **Métriques** : taux d'acceptation par suggestion — cible > 30 %.
- **Insight** : kind de ressources les plus acceptées (calibrer le prompt).

## Considérations sécurité

- Validation Zod stricte URL (https, pas javascript:, pas data:).
- HEAD check avec timeout court (5 s) pour éviter SSRF.
- Rate limit serveur.
- Pas de PII utilisateur dans le prompt (juste titre + objectifs du module).

## Considérations accessibilité

- Liste card-based : `role="list"` + `<article>` par card.
- Icônes kind : avec `aria-label` ("Article", "Vidéo", "Dépôt GitHub").
- Boutons accept/ignore : `aria-label="Ajouter la ressource [titre]"`.
- Loading state annoncé.

## Considérations performance

- HEAD checks en parallel (Promise.all) — cible 5 s total.
- Cap input/output LLM : input 1k tokens, output 800 tokens.
- Latence p95 cible : 8 s (LLM + HEAD checks).
- Cache : pas de cache (peut produire des doublons sur le même module en re-cliquant, OK).
