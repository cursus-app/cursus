## Contexte business

Le rapport du harnais (ST-06.4) liste les checks passés et échoués avec logs bruts. Le stagiaire comprend le format mais perd du temps à décoder "pourquoi mon test échoue". Cette story livre un résumé intelligent IA : 3 raisons probables du rejet + 3 pistes d'action concrètes. Le stagiaire gagne 10-15 min de débug par soumission échouée.

Business value : améliore drastiquement la boucle apprentissage (l'erreur devient feedback exploitable). Marqueur fort Differentiator.

## Description fonctionnelle

- Déclenché automatiquement quand une soumission est `rejected` ou `failed` (job Inngest).
- Input : payload `submission` (id, repo URL, output checks, stderr essentiel tronqué, langue de l'énoncé).
- Prompt LLM structuré : "Tu es un mentor dev. Voici le rapport. Donne 3 raisons probables (concises) et 3 actions concrètes (implémentables en < 30 min)."
- Output validation Zod : `{ reasons: [string;3], actions: [string;3], confidence: 'low'|'medium'|'high' }`.
- Affichage dans le rapport (encadré "Éclairage IA") avec disclaimer ("Cette analyse peut se tromper, vérifie").
- Stocker la synthèse en DB (liée à la soumission) — évite régénération coupûteuse.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Résumé généré sur rejet
Given une soumission ST-K passe en "rejected"
When le job ai.summarize.report s'exécute
Then un résumé IA est créé et lié à la soumission
And le rapport affiche 3 raisons + 3 actions + disclaimer
And le tout en moins de 10 s
```

```gherkin
Scénario 2 — Résumé déjà généré
Given un stagiaire re-visite un rapport ancien
When la page se charge
Then le résumé existant est affiché immediatement (pas de re-call LLM)
```

```gherkin
Scénario 3 — LLM down ou réponse invalide
Given le LLM échoue 2 fois consécutivement
When le job retry
Then après 2 retry, l'encadré affiche "L'éclairage IA est indisponible pour le moment"
And le rapport reste lisible sans le résumé
```

## Cas limites à traiter

- **Logs sensibles** (secrets fuités en stderr) : `redactSensitive(stderr)` avant prompt (regex sur tokens, mots de passe).
- **Logs très longs** : tronquer aux 5 premières erreurs distinctes + résumé stats.
- **Soumission réussie** : pas de génération (économie coût).
- **Ré-soumission** : nouveau résumé généré pour la nouvelle soumission.
- **Multilingue** : résumé généré dans la langue du stagiaire (preferred_locale).
- **Hallucination** : disclaimer toujours visible, garde-fous (ST-21.5).

## Tests à écrire

- **Unit Vitest** : `redactSensitive` masque tokens/mots-de-passe, parseur logs tronque correctement.
- **Integration** : mock LLM → génération valide → stockage DB.
- **Integration retry** : LLM fail 2x → job retry 1 fois puis marque comme `failed`.
- **A11y** : disclaimer textuel (pas juste icône) + `aria-label`.
- **Security** : test que aucun secret fuit dans le payload envoyé au LLM (snapshot).
- **E2E** : soumission rejetée → rapport affiche résumé < 15 s.

## Observabilité

- **Logs** : event `ai.summary.generated` (info) avec `{ submission_id, latency_ms, tokens, cost_usd, confidence }`.
- **Métriques** : compteur `cursus.ai.summary.generated.total`, gauge `cursus.ai.summary.daily_cost_usd`.
- **Alerte** : p95 latency > 30 s ou cost > seuil.
- **Insight** : taux de soumissions rejetées qui reçoivent résumé (cible > 95 %).

## Considérations sécurité

- Redaction systématique des stderr avant prompt.
- RLS : seul le propriétaire de la soumission (+ formateur) voit le résumé.
- Pas de PII envoyée au LLM (uniquement texte technique).
- Clé LLM en secret env.
- Quotas par cohorte (cost guard).

## Considérations accessibilité

- Encadré IA : `<section aria-label="Éclairage IA">`.
- Disclaimer visible (pas seulement icône).
- Liste numérotée pour les 3 raisons et 3 actions.
- Contraste respecté.

## Considérations performance

- Stocker la synthèse en DB évite re-call (cost économisé).
- Job async : pas de blocage UI.
- Cap input : 4k tokens, output : 500 tokens (concis).
- Latence cible : génération en < 10 s.
