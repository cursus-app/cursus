## Contexte business

Les formateurs passent un temps disproportionné à rédiger des quiz alors que la matière du module est déjà écrite (ressources, vidéos, exos). L'IA peut générer un brouillon de quiz cohérent en quelques secondes, qu'il leur reste à valider / ajuster. Cette story est la pierre angulaire du tier Differentiator (01-vision "AI assist") et son objectif est de réduire le temps de création d'un quiz de 30 min à 5 min.

Business value : c'est l'un des deux ou trois usages IA qui justifient seuls le tier Differentiator dans la roadmap v1.1.

## Description fonctionnelle

- Action "Générer un quiz" sur la page d'un module (nécessite ressources existantes).
- Backend Inngest job : prend `module_id`, fetch les ressources (titre + résumé + transcripts si dispo), construit un prompt structuré, appelle un LLM (Claude / OpenAI), valide la sortie via Zod (schema `Quiz { title, questions: [{ text, type, options, correct_index, explanation }] }`).
- UI : modale "Générer un quiz" → choix de difficulté (facile / moyen / dur), nb de questions (5/10/15), focus optionnel (mots-clés). Loading état. Résultat affiché en mode édition (pas push direct), le formateur ajuste, sauve.
- Cost guard : limite N générations / formateur / jour (configurable, voir ST-21.5 garde-fous + ST-21.6 métriques).

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Génération réussie
Given un module avec 3 ressources (article 1, vidéo 1, exo 1)
When le formateur clique "Générer un quiz" et choisit 5 questions, difficulté moyen
Then un brouillon de quiz apparaît en édition après 5-15 s
And les 5 questions sont cohérentes avec le contenu du module
And chaque question a 4 options + bonne réponse + explication
And le formateur peut accepter / éditer / régénérer
```

```gherkin
Scénario 2 — Limite de générations
Given le formateur a déjà généré 10 quiz dans la journée
When il tente une 11e génération
Then le serveur renvoie 429 avec message "Limite atteinte, réessayer demain"
And une métrique est incrémentée
```

```gherkin
Scénario 3 — Sortie invalide
Given le LLM renvoie un JSON malformé
When la validation Zod échoue
Then le job retry 1 fois avec un prompt plus strict
And si l'échec persiste, le formateur voit une erreur claire ("Réessayer")
And l'incident est tracé dans Sentry
```

## Cas limites à traiter

- **Pas de ressources** : bouton désactivé + tooltip "Ajoute au moins une ressource pour générer un quiz".
- **Ressources très longues** : tronquer / chunker, prioriser les titres et premiers paragraphes (tokens limit).
- **LLM down** : circuit breaker après 3 échecs consécutifs, fallback message UI.
- **Hallucination factuelle** : la story dédiée ST-21.5 gère les garde-fous ("3 raisons de revoir cette question"). Ici on indique que le formateur reste responsable.
- **PII dans les ressources** : ne pas envoyer le contenu user (juste les ressources pédagogiques).
- **Locale** : générer dans la langue du module (front-matter ou setting de la cohorte).

## Sous-tâches techniques

- **TT-21.1.1** — Schema Zod `GeneratedQuiz` strict (questions, options, correct_index, explanation).
- **TT-21.1.2** — Prompt template `prompts/generate-quiz.tpl` versionné.
- **TT-21.1.3** — Service `server/ai/generateQuiz.ts` : fetch ressources, call LLM via API résilient (retry + timeout 30 s), validation Zod.
- **TT-21.1.4** — Job Inngest `quiz.generate` (async, idempotent par `generation_id`).
- **TT-21.1.5** — Endpoint `POST /api/modules/:id/quiz/generate` qui déclenche le job et renvoie un `generation_id`.
- **TT-21.1.6** — Endpoint `GET /api/quiz-generations/:id` (polling ou SSE) pour récupérer le statut.
- **TT-21.1.7** — Page modal `<GenerateQuizModal />` avec form (difficulté, nb questions) + loading + résultat éditable.
- **TT-21.1.8** — Limite quotas par rôle (RLS + middleware), stockage compteur quotidien.
- **TT-21.1.9** — Persistance du quiz généré en mode brouillon (statut `draft`) jusqu'à la validation du formateur.

## Non-goals

- Pas de quiz multimédia généré (images, vidéos).
- Pas de questions ouvertes à correction libre (juste QCM).
- Pas de fine-tuning du LLM sur les données Cursus.
- Pas de stream de la sortie LLM mot par mot (juste loading puis résultat).
- Pas de génération cross-modules.

## Tests à écrire

- **Unit Vitest** : validation Zod du schema GeneratedQuiz, helper de tronquage de ressources.
- **Integration** : mock LLM → génération valide → persistance brouillon.
- **Integration LLM réel** : test smoke (1 / jour en CI) avec budget contrôlé.
- **Quotas** : 10 générations → 11e bloquée (429).
- **E2E Playwright** : parcours formateur génère → édite → publie.
- **Security** : pas d'injection prompt user (séparateurs structurés).

## Observabilité

- **Logs** : event `ai.quiz.generated` (info) avec `{ module_id, difficulty, n_questions, latency_ms, tokens_in, tokens_out, cost_usd }`.
- **Métriques** : histogram `cursus.ai.quiz.latency_ms`, compteur `cursus.ai.quiz.generations.total`, gauge `cursus.ai.daily_cost_usd`.
- **Alerte** : coût journalier > seuil configuré.
- **Insight** : taux d'acceptation (formateur publie le brouillon sans modifs) — cible > 40 %.

## Considérations sécurité

- RBAC : seuls formateurs et admins peuvent générer.
- Quota strict server-side (pas seulement UI).
- Pas de PII utilisateur dans le prompt (juste ressources pédagogiques).
- Clé LLM en secret env, pas commitée.
- Validation Zod stricte de la sortie (pas d'XSS via la generation).
- Rate limit réseau (Inngest concurrency = 5 par org).

## Considérations accessibilité

- Loading state annoncé (`aria-live="polite"` : "Génération en cours, \~10 secondes…").
- Quiz généré navigable au clavier en mode édition.
- Toast d'erreur accessible.
- Skeleton durant le loading (pas juste spinner).

## Considérations performance

- Job async : pas de blocage HTTP > 30 s.
- Polling 1 s ou SSE pour status.
- Cap tokens : input 8k max, output 4k max (équivalent ± 15 questions).
- Latence p95 cible : 15 s pour un quiz 10 questions.
- Cache : pas de cache (chaque génération doit être fresh).
