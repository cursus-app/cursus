## Contexte business

L'IA, c'est un budget. Sans tableau de bord clair ("on consomme X dollars / mois", "feature Y coûte 60 % du total"), on dérive sans s'en rendre compte jusqu'à la facture surprise. Cette story livre la visibilité opérationnelle : usage, coût, taux d'erreur, latence par feature IA, accessible aux admins.

Business value : pilotage économique du tier Differentiator. Permet de décider quelle feature continuer / arrêter / optimiser.

## Description fonctionnelle

- Table `ai_call_log` (créée en ST-21.5) alimentée par chaque call IA.
- Page admin `/admin/ai-usage` avec :

  - KPI cards : coût total mois en cours, coût jour, nb calls jour, taux d'erreur.
  - Graphique line chart : coût $ / jour sur 30 jours.
  - Graphique stacked bar : répartition coût par feature (quiz_gen, summary, suggest, insights).
  - Tableau top users / orgs par consommation.
  - Alertes config : seuil $ / jour, seuil erreur.

- Export CSV pour comptabilité.
- Pas de PII (juste user_id, feature_id, métriques techniques).

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Admin voit le coût du mois
Given l'admin ouvre /admin/ai-usage
When la page charge
Then le KPI "Coût mois en cours" affiche le total (ex: $42.30)
And un line chart montre l'évolution sur 30 jours
And un stacked bar montre la répartition par feature
```

```gherkin
Scénario 2 — Alerte dépassement
Given le seuil $5 / jour est configuré
When le coût journalier atteint $5
Then une notification Slack admin (ou email) est envoyée
And un badge alerte apparaît sur la page
```

```gherkin
Scénario 3 — Export CSV
Given l'admin clique "Exporter CSV"
When un range de dates est sélectionné
Then un CSV est téléchargé avec date, feature, calls, tokens_in, tokens_out, cost_usd
```

## Cas limites à traiter

- **Coût = 0 (mock LLM en dev)** : afficher "Aucune donnée" + lien doc.
- **Dépassement seuil pendant la nuit** : alerte arrive le matin (acceptable).
- **Taux d'erreur élevé mais isolé** (LLM down 1h) : pas d'alerte si retour rapide — alerte si > 5 min consécutives.
- **Données manquantes** (call sans cost faute de provider price) : marquer estimated.
- **Beaucoup de calls** : graphes agrégés par jour, drill-down sur demande.
- **Permissions** : page accessible aux admins uniquement.

## Non-goals

- Pas de prédiction de coût (forecasting) au MVP.
- Pas de billing par client (single-tenant).
- Pas de comparaison cross-providers (Claude vs OpenAI) automatisée.
- Pas de A/B sur prompt versions.

## Tests à écrire

- **Unit Vitest** : agrégations SQL retournent bonnes valeurs sur fixture.
- **Integration** : `GET /api/admin/ai-usage` filtré par date.
- **RBAC** : non-admin reçoit 403.
- **Alertes** : seuil $5 / jour atteint → Slack envoyé (mock).
- **CSV export** : format correct, encoding UTF-8 BOM.
- **E2E** : admin ouvre dashboard, switch range, exporte CSV.

## Observabilité

- **Logs** : event `ai.cost_alert.triggered` (warn) avec `{ threshold, actual, period }`.
- **Métriques** : c'est cette story qui les expose.
- **Dashboard** : c'est cette story qui le livre.

## Considérations sécurité

- RBAC admin strict.
- Pas de PII dans les logs / exports (user_id hashé optionnel).
- CSV export rate-limité (1 / heure / admin).

## Considérations accessibilité

- Charts : table équivalente accessible (cf. ST-23.\*).
- Couleurs charts respectent contraste + symboles pour daltoniens.
- KPI cards : structure semantique (`<dl>` ou `<section aria-label>`).
- Annonce des mises à jour (`aria-live` sur compteurs).

## Considérations performance

- Agrégations pré-calculées la nuit (cron) pour rapidité.
- Cache côté client (vue-query) avec stale 1 min.
- Charts virtualisés si > 1 an de données.
- Cible : page interactive < 500 ms post-load.
