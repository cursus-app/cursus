## Contexte business

Quand une cohorte fait un quiz, les résultats bruts (note moyenne, distribution) sont une vue de surface. La vraie valeur pédagogique : identifier quelles **notions** sont mal comprises (ex: "60 % des stagiaires ratent les questions sur les hooks useEffect"). Cette story livre l'analyse IA des patterns d'erreurs pour donner au formateur un signal actionnable.

Business value : transforme les quiz d'un outil d'évaluation en outil de diagnostic pédagogique. Marqueur fort Differentiator ("AI insight cohort-level").

## Description fonctionnelle

- Déclenché après la fin d'une session quiz par cohorte (job Inngest `quiz.analyze.cohort`).
- Input : agrégation des réponses par question avec stats (taux de bonne réponse, distractor le plus choisi en cas d'erreur).
- Le LLM analyse les patterns : groupes de questions sur le même concept avec taux d'erreur > 40 %.
- Output : `{ insights: [{ topic, error_rate, supporting_questions[], suggestion_for_teacher }] }`.
- Affichage dans le dashboard formateur : encadré "Notions à retravailler" avec liste prioritaire.
- Action proposée : envoyer un message à la cohorte / recommander une ressource complémentaire.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Analyse après quiz
Given la cohorte X a terminé le quiz "React Hooks" (15 questions, 12 stagiaires)
When le job quiz.analyze.cohort s'exécute (après last attempt)
Then le dashboard formateur affiche 2-5 insights (ex: "useEffect dependencies" 70 % d'erreur)
And chaque insight liste les questions concernées
And chaque insight propose une suggestion d'action
```

```gherkin
Scénario 2 — Cohorte trop petite
Given seulement 2 stagiaires ont fait le quiz
When le job s'exécute
Then aucun insight n'est généré (échantillon insuffisant)
And le dashboard affiche "Échantillon trop petit pour analyse fiable (min 5 stagiaires)"
```

```gherkin
Scénario 3 — Suggestion d'action
Given un insight "useEffect dependencies 70 % erreur"
When le formateur clique "Envoyer une ressource complémentaire à la cohorte"
Then une modal pré-remplie avec le message + ressource suggérée apparait
And il peut l'envoyer en 1 clic
```

## Cas limites à traiter

- **Cohorte < 5 stagiaires** : skip analyse (échantillon insuffisant).
- **Quiz à 1 question** : skip (pas de pattern détectable).
- **Résultats unanimes (100 % bons)** : pas d'insights, message "Félicitations !".
- **Résultats unanimes (100 % échec)** : insight global "réviser tout".
- **Quiz ré-fait par stagiaire** : prendre dernière tentative seulement.
- **Hallucination IA** (regroupement non pertinent) : garde-fou ST-21.5 (confidence < 0.6 → skip).

## Non-goals

- Pas d'analyse par stagiaire individuel (cohort-level only au MVP).
- Pas de génération automatique de quiz de remédiation.
- Pas de comparaison cross-cohortes.
- Pas de feedback continu pendant le quiz (post-completion only).

## Tests à écrire

- **Unit Vitest** : agrégation stats par question correcte, parser confidence threshold.
- **Integration** : 1 cohorte avec data fixture → mock LLM → insights persistes.
- **Cohorte < 5** : test que job skip et message UI affiché.
- **A11y** : encadré navigable, suggestions cliquables.
- **E2E** : parcours formateur voit insights après quiz complet de la cohorte.
- **Security** : pas de PII stagiaire dans le prompt (juste IDs anonymisés + texte questions).

## Observabilité

- **Logs** : event `ai.quiz_insights.generated` (info) avec `{ cohort_id, quiz_id, n_insights, latency_ms, tokens, cost_usd }`.
- **Métriques** : compteur `cursus.ai.quiz_insights.total`, gauge `cursus.ai.quiz_insights.cost_usd`.
- **Insight** : taux d'action sur insight (formateur envoie ressource) > 30 % cible.

## Considérations sécurité

- RLS : seul le formateur de la cohorte voit les insights.
- Pas de noms stagiaires envoyés au LLM (juste texte questions + stats anonymisées).
- Quotas par cohorte.

## Considérations accessibilité

- Encadré insights : `<section aria-label="Notions à retravailler">`.
- Chaque insight : `<article>` avec headline.
- Liens vers questions et suggestions navigables.
- Couleur seule jamais utilisée pour priorité (texte + icône).

## Considérations performance

- Job async (pas blocage).
- Cap input : 4k tokens, output : 1k tokens.
- Latence cible : génération en < 15 s.
- Stockage en DB évite re-call à chaque visite dashboard.
