## Contexte business

L'IA apparaît dans plusieurs stories (génération quiz, résumé rapport, suggestion ressources, analyse patterns). Sans garde-fous systématiques, on s'expose à : hallucinations, output XSS, coûts incontrôlés, prompt injection, leak de PII. Cette story est l'infrastructure transversale qui durcit toutes les autres stories AI.

Business value : sans garde-fous, on ne peut pas livrer le tier Differentiator AI en prod. C'est l'assurance qualité et conformité de la roadmap IA.

## Description fonctionnelle

- **Validation Zod stricte** des outputs LLM (schemas réutilisés).
- **Hallucination detection** : confidence threshold (skip si `confidence < 0.6`), URL HEAD-check, faits vérifiés.
- **Prompt injection defense** : separator structuré, escape des inputs user, pas d'instruction "override previous".
- **PII redaction** avant envoi LLM : helper `redactPii(text)` qui masque emails, noms, tokens.
- **Cost guard** : quotas par feature, par user, par org, par jour. Fallback gracieux si quota atteint.
- **Output sanitization** : pas d'HTML brut dans les outputs (Vue échappe par défaut, on confirme).
- **Audit trail** : chaque call LLM est tracé (model, prompt hash, tokens, cost, user_id).
- **Kill switch** : feature flag global pour désactiver toutes les features IA.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Output invalid rejeté
Given le LLM retourne un JSON non-conforme au schema
When la validation Zod s'exécute
Then une erreur est levée
And un retry avec prompt plus strict est tenté
And un log d'erreur Sentry est créé
```

```gherkin
Scénario 2 — Quota atteint
Given l'organisation a déjà consommé le quota IA du jour
When un user tente une génération
Then le serveur renvoie 429 avec message clair "Quota IA atteint, réessayer demain"
And la métrique cursus.ai.quota_exceeded.total est incrémentée
```

```gherkin
Scénario 3 — Kill switch
Given AI_GLOBAL_KILLSWITCH=true en env
When n'importe quelle feature IA est appelée
Then elle renvoie immédiatement 503 avec message "Fonctionnalité IA temporairement désactivée"
And aucun appel LLM n'est fait
```

## Cas limites à traiter

- **Prompt injection** ("ignore previous instructions and…") : separator structuré + tagging clair des inputs.
- **PII dans input** : `redactPii()` systématique (regex emails, names DB-lookup, tokens).
- **Output XSS** : pas de `v-html`, Vue échappe par défaut.
- **Coût dérive** : monitoring quotidien + alerte si > seuil.
- **LLM provider down** : circuit breaker après 5 échecs consécutifs.
- **Réponse vide ou trop courte** : retry avec prompt enrichi.
- **Dépassement context window** : tronquer input + warn.

## Non-goals

- Pas d'output rewriting (juste validation strict + retry).
- Pas de moderation content provider (Anthropic / OpenAI ont leurs propres).
- Pas de detection de jailbreaking sophistiqué au MVP.
- Pas de RAG ou citation des sources au MVP.

## Tests à écrire

- **Unit Vitest** : `redactPii` masque les emails, `safePrompt` escape `<` et `>`.
- **Unit** : Zod validation rejette outputs malformés.
- **Integration** : circuit breaker s'ouvre après 5 échecs.
- **Quota** : 10 calls → 11e bloquée ; reset après 24h.
- **Kill switch** : env=true → 503 immédiat.
- **Security pen-test** : prompt injection ("ignore previous") ne change pas le format de sortie.
- **A11y** : messages d'erreur quota / killswitch accessibles.

## Observabilité

- **Logs** : event `ai.call.started` / `ai.call.completed` / `ai.call.failed` avec `{ feature, model, tokens, cost, latency_ms, status }`.
- **Métriques** : `cursus.ai.calls.total` (par feature, par status), `cursus.ai.cost_usd_total` (par feature, par day).
- **Alerte** : cost / day > seuil, taux d'erreur > 5 %, circuit breaker ouvert.
- **Dashboard** : page admin `/admin/ai-usage` avec graphes coût + appels.

## Considérations sécurité

- Clé LLM en secret env, rotation régulière.
- PII redaction obligatoire (audit du code en CI).
- Quota strict server-side.
- Rate limit réseau (Inngest concurrency).
- Logging sans secrets (clés, prompts redactés).
- Audit trail tamper-evident (insertion-only).

## Considérations accessibilité

- Messages d'erreur (quota, killswitch) accessibles via `aria-live`.
- Texte explicite (pas seulement code).

## Considérations performance

- Helpers de safety doivent ajouter < 5 ms par call.
- Circuit breaker en mémoire (pas de DB round-trip).
- Quota check : caché par 1 min (acceptable, max 1 over-quota par minute).
