## Contexte business

Badges = reconnaissance de comportements positifs au-delà des XP. Engagement émotionnel. Feature premium qui structure l'apprentissage.

Valeur livrée : 5 badges initiaux (premier deploy, commit signé, cursus complet, capstone en avance, mentor du jour), déblocage automatique via engine de règles JSON, attribution manuelle possible par formateur.

## Description fonctionnelle

Engine de règles JSON (`badgeRules.ts`). Event handlers écoutent les métriers et attribuent automatiquement. Animation card-flip au déblocage. Attribution manuelle formateur avec mention.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Déblocage automatique**

```gherkin
Given un événement métier survient (ex: url_responds validé)
When il matche une règle de badge
Then le badge est attribué au stagiaire avec timestamp
And une notification "Tu as débloqué : <badge>" s'affiche
And le badge apparaît sur le profil
```

**Scénario 2 — Attribution manuelle**

```gherkin
Given un formateur attribue manuellement "Mentor du jour"
Then le badge est ajouté avec mention "Attribué par <formateur>"
```

**Scénario 3 — Badge déjà obtenu**

```gherkin
Given un badge est déjà débloqué
When le même événement se reproduit
Then le badge n'est pas attribué à nouveau (idémpotent)
```

## Badges initiaux

1. **"Premier deploy"** — `url_responds` validé
2. **"Commit signé"** — `commits_signed` validé
3. **"Cursus complet"** — 100% modules validés
4. **"Capstone en avance"** — capstone >= 3j avant échéance
5. **"Mentor du jour"** — manuel

## Cas limites à traiter

- **Règle qui matche plusieurs fois** : idémpotent (badge attribué 1 fois).
- **Badge attribué par erreur** : possibilité de révocation par formateur.
- **Règles modifiées après déblocage** : badges existants conservés.
