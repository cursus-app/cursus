## Contexte business

Un stagiaire prêt à soumettre dans le train sans wifi : aujourd'hui, il doit attendre. Avec la soumission différée, sa soumission est queuée localement et automatiquement envoyée dès que la connexion revient. Zéro effort utilisateur.

Business value : feature engagement forte ("ça marche même en métro"). Marqueur premium d'app pro mobile-first.

## Description fonctionnelle

- L'user clique "Soumettre" → si offline, le payload est stocké dans IndexedDB + Background Sync est enregistré.
- UI : toast persistant "Soumission en attente — sera envoyée dès connexion". Liste des pending submissions visible dans le profil.
- Au retour en ligne (ou via SW Background Sync), Cursus tente le POST. Si succès : toast "Soumission envoyée". Si échec : retry exponentiel.
- Annulation possible avant envoi.
- Idempotence : `idempotency_key` généré client-side pour éviter doublons si retry.

```gherkin
And le toast disparaît
```

## Cas limites à traiter

- **App fermée** : Background Sync API déclenche même app fermée (sur navigateurs supportés).
- **Serveur 400** (validation échouée) : pas de retry, notifier user de l'erreur.
- **Serveur 500** : retry exponentiel (5 fois).
- **Browser sans Background Sync** (Safari) : fallback sur Online event listener.
- **Idempotency** : si serveur reçoit 2× même `idempotency_key`, retourner la réponse stockée initiale.
- **Queue volumineuse** : limite 10 pending par user, message clair.

## Sous-tâches techniques

- **TT-24.4.1** — IndexedDB store `pending_submissions { id, idempotency_key, payload, created_at, last_attempt_at, attempts }`.
- **TT-24.4.2** — Helper `queueSubmission(payload)` : enregistre + register Background Sync.
- **TT-24.4.3** — SW handler `sync` : flush queue, POST chaque pending.
- **TT-24.4.4** — Endpoint `POST /api/submissions` accepte `Idempotency-Key` header.
- **TT-24.4.5** — Composant `<PendingSubmissionsList />` (dans profil ou drawer).
- **TT-24.4.6** — Toast persistant `<PendingSubmissionToast />`.
- **TT-24.4.7** — Fallback sans Background Sync : `window.addEventListener('online', flushQueue)`.
- **TT-24.4.8** — Tests offline simulator.

## Non-goals

- Pas de queueing pour autres POST (juste soumissions).
- Pas de conflict resolution complexe (si entre temps cohorte clôturée, le serveur refuse, l'user notifié).
- Pas de chiffrement du payload local (RGPD : juste obfuscation, IndexedDB origin-locked).
- Pas de partage cross-device.

## Tests à écrire

- **Unit Vitest** : `queueSubmission` stocke, `flushQueue` POST chaque pending.
- **Integration** : POST avec `Idempotency-Key` dupé → même réponse.
- **E2E Playwright** : offline, soumettre, online, vérifier auto-envoi.
- **Idempotency** : test que retry n'envoie pas 2 soumissions distinctes.
- **A11y** : toast persistant accessible.
- **Security** : RLS à l'envoi (le serveur revalide).

## Observabilité

- **Logs** : `submission.queued` (info), `submission.flushed.success` (info), `submission.flushed.failed` (warn).
- **Métriques** : compteur queued, taux de flush success, latence moyenne entre queue et flush.
- **Insight** : nb users avec pending > 1 (indicateur connexion difficile).

## Considérations sécurité

- Payload chiffré / token au server side renvoyé.
- `Idempotency-Key` validé côté serveur (max age 24 h).
- RLS revalidée à chaque flush.
- Audit log à chaque envoi.
- Suppression queue au logout.

## Considérations accessibilité

- Toast `aria-live="polite"`.
- Liste pending navigable.
- Confirmation modal pour annulation.
- Annonce auto-envoi.

## Considérations performance

- Background Sync : aucune charge UI pendant l'envoi.
- Limite queue 10 (prévient les fuites).
- Cible : flush en < 5 s post-online.
- IndexedDB compact (cible < 1 MB total).
