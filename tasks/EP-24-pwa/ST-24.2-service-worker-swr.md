## Contexte business

Le service worker est ce qui rend une PWA "vraie" : interception des requêtes, cache stratégique, offline. Cette story livre la couche SW de base avec stratégie `stale-while-revalidate` pour les assets et `network-first` pour les API critiques. Prépare le terrain pour les modes offline dédiés (ST-24.3, ST-24.4).

Business value : ressenti app native (pages instantanées au re-visit), résilience réseau (connexion intermittente métro).

## Description fonctionnelle

- SW généré via `@vite-pwa/nuxt` (workbox sous le capôt), compatible Nuxt 4.4.
- Stratégies :

  - `stale-while-revalidate` : assets statiques (`/_nuxt/`, `/fonts/`, `/icons/`, `/illustrations/`).
  - `network-first` (timeout 3 s) : API non-mutantes (`GET /api/cursus`, etc.).
  - `network-only` : POST / PUT / DELETE.
  - `cache-first` : images, fonts hashées.

- Background sync pour les soumissions queuées (ST-24.4).
- Update flow : nouveau SW détecté → toast "Nouvelle version, recharger ?".
- Cache versionné par release.
- Skip-waiting + clients-claim après confirmation user.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Assets cachés
Given l'user a visité la home en ligne
When il revient hors-ligne (DevTools offline)
Then la home se charge depuis le cache SW (200 ms)
And les assets _nuxt sont servis stale
And une nouvelle requête réseau est faite en background
```

```gherkin
Scénario 2 — API GET en cache
Given le user a chargé /cursus en ligne
When il y revient avec réseau lent (timeout > 3 s)
Then la réponse cachée est servie (network-first fallback)
And un indicateur visuel "Données du cache" est affiché
```

```gherkin
Scénario 3 — Nouvelle version déployée
Given un user a la version v1.0 chargée
When une v1.1 est déployée
Then un toast apparaît : "Nouvelle version disponible. Recharger ?"
And cliquer recharge avec skip-waiting
```

## Cas limites à traiter

- **Cache débordé** : workbox LRU eviction (max 50 MB).
- **Update silencieux risqué** (in-flight form data perdue) : demander confirmation.
- **Browser sans SW** (vieux Safari) : graceful fallback, pas de crash.
- **CORS sur assets externes** : pas cache.
- **API avec param d'auth** : key cache par URL complet pour éviter cross-user leak.
- **POST en offline** : queué (ST-24.4), pas servi par cache.

## Non-goals

- Pas de cache pour POST/PUT/DELETE.
- Pas de full app shell architecture (le rendu Nuxt 4 SSR suffit, le SW vient en complément pour le re-visit).
- Pas de push notifications via SW au MVP.
- Pas de pre-cache exhaustif (lazy par utilisation).

## Tests à écrire

- **Unit Vitest** : composable update flow.
- **Integration** : DevTools offline, charger page → SW cache hit.
- **E2E Playwright** : SW register, update flow.
- **Lighthouse PWA** : score augmente avec SW actif.
- **Security** : pas de cache cross-origin sensible.
- **Performance** : SW ne ralentit pas le boot (cible LCP ≤ pre-SW).

## Observabilité

- **Logs** : `sw.installed` (info), `sw.activated`, `sw.update.available`, `sw.update.accepted`.
- **Métriques** : compteur `cursus.sw.cache.hit_ratio`, gauge `cursus.sw.cache.size_mb`.
- **Alerte** : ratio cache hit < 50 % après 1 semaine (problème de stratégie).

## Considérations sécurité

- Pas de cache d'API authentifiées cross-user (clef cache inclut session token hash si besoin).
- CSP doit autoriser SW.
- HTTPS only.
- Update consensuel (pas de skip-waiting silencieux).

## Considérations accessibilité

- Toast update : `role="alert"` (urgent attention).
- Boutons annoncés.
- Pas de jank perceptible au switch.

## Considérations performance

- SW ajoute < 50 ms au boot la 1ère fois.
- Subsequent navigation : -80 % temps de chargement.
- Cache size capé 50 MB.
- Cible Lighthouse perf score ≥ 90 avec SW.
