## Contexte business

Le ressenti mobile : un utilisateur en réseau instable a besoin de savoir clairement quel est son état (en ligne, hors ligne, connexion faible). Sans cet indicateur, il croit que l'app est cassée. Un simple badge dans la barre de navigation transforme l'incertitude en confiance.

Business value : ressenti app pro premium. Clé pour assumer le statut PWA en mode réel.

## Description fonctionnelle

- Composant `<NetworkStatusIndicator />` dans la nav globale.
- 4 états : Online (vert subtil, icône wifi), Connexion lente (ambre, icône signal faible — détecte via `navigator.connection.downlink`), Offline (rouge subtil, icône wifi barré), Reconnect en cours (anim).
- Tooltip explicatif au hover.
- Banner offline persistant si reste > 5 s.
- Métriques : usage Network Information API si dispo, fallback `online/offline` events.
- Test ping `/healthz` toutes les 30 s pour confirmer la connexion réelle.

## Critères d'acceptation (Gherkin)

```gherkin
Scénario 1 — Indicateur en ligne
Given l'user est en ligne avec bonne connexion
When il regarde la nav
Then l'indicateur est vert avec icône wifi
And hover affiche "En ligne (4G)"
```

```gherkin
Scénario 2 — Passage offline
Given l'user est en ligne
When il perd la connexion
Then l'indicateur passe rouge avec wifi barré
And un banner top "Mode hors-ligne" apparaît après 5 s
And hover affiche "Hors ligne — lecture seule"
```

```gherkin
Scénario 3 — Connexion lente
Given la connexion est réelle mais downlink < 0.5 Mbps
When le composable détecte
Then l'indicateur passe ambre
And hover affiche "Connexion lente, certaines actions peuvent prendre du temps"
```

## Cas limites à traiter

- **Connection API non supportée** (Safari) : juste online/offline events.
- **Network apparait online mais API down** : ping `/healthz` confirme.
- **Faux positif offline** (court burst) : débouncer 2 s avant montrer banner.
- **prefers-reduced-motion** : pas d'anim de reconnect.
- **Mobile data saving** : respecter `connection.saveData` (afficher warning).
- **Localisation** : tooltip et banner traduits.

## Non-goals

- Pas de mesure de latence précise.
- Pas de switch automatique entre cache/réseau au-delà de ST-24.3.
- Pas de notification push si offline.
- Pas d'historique de connexions.

## Tests à écrire

- **E2E Playwright** : offline → banner après 5 s, online → banner disparaît.
- **A11y axe-core** : indicateur navigable, tooltip annoncé.
- **i18n** : FR + EN.

## Observabilité

- **Logs** : `network.status.changed` (info) avec `{ from, to, downlink, effective_type }`.
- **Métriques** : compteur transitions online/offline, gauge `cursus.users.offline.total`.
- **Insight** : % de sessions ayant connu offline > 5 s (engagement mobile).

## Considérations sécurité

- `/healthz` ne leak aucune donnée (juste 200/503).
- Pas de PII dans les logs.
- Rate limit du polling /healthz (1/30s/user) côté client (évite spam serveur).

## Considérations accessibilité

- Icône + texte alternatif (jamais couleur seule pour status).
- Banner `role="status"` (non urgent).
- Tooltip accessible (focus visible).
- Annonce du switch online/offline via `aria-live="polite"`.
- Respect prefers-reduced-motion (pas d'anim de spinning).

## Considérations performance

- `/healthz` ping minimal (HEAD, < 1 KB response).
- Composable shared (singleton, pas re-poll par composant).
- Indicateur composant ultra léger (< 2 KB).
- Pas de re-render inutile (Vue reactivity ciblée).
