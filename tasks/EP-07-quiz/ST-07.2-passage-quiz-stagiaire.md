## Contexte business

Le stagiaire passe le quiz, voit son score, revoit les explications pour apprendre de ses erreurs. **Anti-pilier respecté** : pas de timer dur, pas d'anti-triche — c'est un check-in pédagogique pas un examen.

Valeur livrée : passage fluide, sauvegarde auto en cas de perte de connexion, randomisation des questions ET des options, 3 tentatives max avec meilleur score conservé.

## Description fonctionnelle

Page dédiée au quiz. Ordre des questions et options randomisé. Pas de timer dur, mais affichage du temps écoulé. Sauvegarde auto des réponses en cours. 3 tentatives max. Résultat avec explications.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Passage standard**

```gherkin
Given un stagiaire démarre un quiz
When il passe les 10 questions
Then chaque question affichée a ses options dans un ordre randomisé
And il peut revenir en arrière
```

**Scénario 2 — Soumission et révisions**

```gherkin
Given il soumet
Then son score s'affiche (X/10)
And il peut revoir question par question avec les bonnes réponses + explications
```

**Scénario 3 — Tentatives multiples**

```gherkin
Given il a 3 tentatives max
When il rate la 1ʳᵉ
Then il peut retenter
And la meilleure tentative est conservée
```

## Cas limites à traiter

- **Connexion perdue pendant le quiz** : reprendre où il en était (sauvegarde auto).
- **Toutes tentatives consommées** : message + bouton "Demander une re-tentative au formateur".
- **Multi-tab** : avertir si le même quiz est ouvert ailleurs.
- **Page reload** : reprendre où il en était.

## Tests à écrire

- **E2E (Playwright)** : passage complet d'un quiz.
- **Unit** : tirage aléatoire reproductible (seed).
- **Resilience** : perte de connexion → reprise.
- **Security** : un stagiaire ne peut soumettre que pour lui.

## Observabilité

- **Métriques** : `quiz_attempts.total`, score moyen, taux de reprise.

## Considérations sécurité

- RLS : stagiaire soumet pour lui uniquement.
- Pas de leak des bonnes réponses dans le payload côté client (validation serveur).
- Tentatives comptées DB (pas localStorage).

## Considérations accessibilité

- Form quiz : labels, navigation clavier.
- Bonnes réponses post-soumission : icône + texte.
- Pas de timer stress (anti-pilier).

## Considérations performance

- Page quiz : LCP < 1.5s.
- Sauvegarde auto : debounce 2s.

## Definition of Done spécifique

- E2E passage complet.
- Resilience perte connexion testée.
