## Contexte business

Le `contains` (case-insensitive) du MVP (ST-20.2) marche pour < 5000 records mais dégrade rapidement, et la qualité de relevance est faible (pas de stemming, pas de pertinence). Passer au full-text search Postgres (`tsvector` + `ts_rank`) est l'évolution naturelle pour atteindre une expérience search premium (typo tolerance via pg_trgm, ranking par pertinence).

Business value : recherche rapide et pertinente sur 100k+ records, latence p95 < 100 ms. Le pilier Premium "navigation rapide" devient crédible à l'échelle.

## Description fonctionnelle

- Créer des colonnes `search_tsv` (`tsvector`) sur les tables `cursus`, `cohortes`, `users`, `modules`, `alertes`. Générées par triggers (concaténation de title + description + tags pondérée setweight 'A' pour title, 'B' pour description).
- Index GIN sur `search_tsv` (rapide recherche).
- Endpoint `GET /api/search` mis à jour pour utiliser `plainto_tsquery` puis ranking via `ts_rank` (via Prisma `$queryRaw` avec preview feature `fullTextSearchPostgres`). Fallback `pg_trgm` similarity pour typo tolerance.
- Configuration French par défaut (`'french'::regconfig`), bascule `'english'` selon locale du user.
- Métriques de p95 latency + couverture.

## Critères d'acceptation (Gherkin)

```

```

```gherkin

Given un user EN cherche "developing"
When le tsquery utilise 'english' config
Then les contenus EN avec "develop", "developed" remontent
And les contenus FR ne sont pas perturbés
```

## Cas limites à traiter

- **Query multi-mots** : `plainto_tsquery` gere AND par défaut, on peut basculer sur `websearch_to_tsquery` pour OR + guillemets exacts.
- **Caractères spéciaux** (`@`, `#`) : strip avant tsquery, conservés pour recherche exacte si entre guillemets.
- **Documents très longs** : tronquer le tsvector (max 1 MB par row), prioriser title + première phrase.
- **Index obsolète** après update : trigger garantit la cohérence immédiate.
- **DB en cours de migration** : si `search_tsv` colonne pas encore peuplée, fallback `contains`.
- **Query vide ou < 2 chars** : retourner \[\], pas de scan complet.

## Non-goals

- Pas d'indexation de PDFs uploadés.
- Pas de fuzzy synonyms ("développeur" ↔ "dev") au MVP (custom dict possible si besoin).
- Pas de recherche cross-language sans switch explicite.
- Pas de moteur de recherche externe (ElasticSearch / Meilisearch) au MVP.
- Pas d'highlighting des matches dans les snippets (UX, story dédiée si besoin).

## Tests à écrire

- **Unit Vitest 4.1** : helper `buildSearchQuery(q, locale)` génère la SQL attendue.
- **Integration** : peupler 1000 cursus, requêter "dév" → ranking attendu en tête.
- **Migration** : trigger insère et update tsvector correctement.
- **Backfill** : script idempotent (re-runnable sans casse).
- **Performance** : 100k records, latence p95 < 100 ms (test bench).
- **Locale switch** : FR vs EN, mêmes documents, rankings cohérents.
- **Security** : injection dans le tsquery proprement échappée (Prisma `$queryRaw` tagged template).

## Observabilité

- **Logs** : event `search.fts.query` (info) avec `{ q_length, regconfig, results_count, latency_ms, used_fallback }`.
- **Métriques** : histogram `cursus.search.fts.latency_ms` p50 / p95 / p99 ; compteur `cursus.search.fts.fallback_used.total`.
- **Alerte** : p95 > 200 ms 5 min consécutives → enquête.
- **Insight** : taux de fallback > 5 % → réviser la config FTS ou ajouter synonyms.

## Considérations sécurité

- RLS toujours active (FTS ne contourne pas les policies ; rôle `cursus_app` non-superuser).
- Param Zod, pas d'injection (`$queryRaw` tagged template paramétré).
- Pas de leak de données privées via FTS (RLS appliqué avant ranking).

## Considérations accessibilité

- Page de recherche complète `/recherche` : empty state, loading state, error state, tous annoncés.
- Snippets : pas d'HTML échappé dans les résultats.
- Compteur de résultats annoncé.

## Considérations performance

- Index GIN : cible insert/update < 50 ms (overhead trigger).
- Search query p95 < 100 ms sur 100k records.
- Pas d'expansion combinatoire (`prefix:*` agressif évité).
- Cache de query côté serveur (Redis) si charges élevées (post-MVP).
