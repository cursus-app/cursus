## Contexte business

Un spinner global "loading" est un anti-pattern : il ne communique pas ce qui charge ni la structure attendue. Les skeleton loaders (carrelage qui préfigure le contenu) sont l'équivalent moderne. Cette story livre les skeletons sur 10 écrans clés via le composant `USkeleton` de @nuxt/ui 4.8. C'est un marqueur Premium (axe 1, design-system §3).

## Description fonctionnelle

Utiliser le composant `USkeleton` de @nuxt/ui v4 (shimmer natif, dark/light compatible). Compositions par écran via des helpers `Skeleton<Page>.vue` qui agencent plusieurs `USkeleton` pour refléter la structure réelle (évite CLS lors du remplacement).

Application sur 10 écrans :

1. Dashboard stagiaire (timeline + carte semaine)
2. Dashboard formateur (table cohorte)
3. Page module (ressources + livrable)
4. Rapport harnais (checks list)
5. Portfolio (livrables)
6. Profil public (header + grid)
7. Cohorte detail (membres)
8. Cursus list (cartes)
9. Audit log explorer (table rows)
10. Page certif (PDF preview)

Chaque skeleton reflete la structure réelle (même layout) pour éviter CLS lors du remplacement.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Dashboard stagiaire loading**

```gherkin
Given le dashboard stagiaire charge
When on attend la donnée
Then le squelette structurel s'affiche (compositions de USkeleton : timeline, cards)
And les contours respectent les dimensions réelles
And le shimmer USkeleton anime subtil
```

**Scénario 2 — Pas de CLS**

```gherkin
Given le skeleton est affiché
When le contenu réel arrive
Then pas de saut de mise en page (CLS = 0)
And le passage est fade in (200ms via motion-v / @vueuse/motion)
```

**Scénario 3 — prefers-reduced-motion**

```gherkin
Given prefers-reduced-motion = reduce
When un USkeleton est affiché
Then pas de shimmer animation
And background simplement gris (token semantic --color-muted)
```

## Cas limites à traiter

- **Loading > 5s** : afficher un "prend du temps" message en plus du skeleton.
- **Loading lightning fast** : pas afficher skeleton si < 200ms (min display 300ms).
- **Skeleton trop long** : section par section (progressive loading).
- **Reuse skeleton vs cache** : préférer cache si data disponible.
- **Erreur de loading** : skeleton remplacé par error state.

## Non-goals

- Pas de skeleton sur toutes les pages (10 clés suffisent).
- Pas de skeleton dynamique (auto-généré depuis HTML).
- Pas de progressive image loading (story v1.x).

## Tests à écrire

- **Unit** : composition skeleton renders correctement avec props.
- **E2E** : vérifier skeleton apparaît sur slow network.
- **Visual Chromatic** : snapshots des skeletons (light + dark).
- **CLS** : Lighthouse CLS = 0 pages avec skeleton.

## Considérations accessibilité

- Skeleton container : `aria-busy="true"`, `aria-live="polite"` sur loaded.
- Shimmer respect prefers-reduced-motion (géré nativement par USkeleton).
- Pas de contenu mock textuel (just shapes).
- Annonce "Loading..." pour lecteurs d'écran.

## Considérations performance

- USkeleton : CSS pur, no JS.
- CLS = 0 (structure identique).
- LCP non dégradé (skeleton compte comme paint).
