## Contexte business

Sans CGU (Conditions Générales d'Utilisation) et Politique de Confidentialité publiées et acceptées explicitement par les utilisateurs, Cursus n'est pas légalement déployable. Ces documents sont la base contractuelle de l'usage et la base légale du traitement des données (RGPD article 6 : consentement, exécution d'un contrat). Sans cette story, on ne peut pas lancer le pilote.

C'est aussi un signal de sérieux pour les organisations clientes (le DPO ou le juridique vérifie systématiquement avant d'autoriser un déploiement).

## Description fonctionnelle

Deux pages publiques accessibles depuis le footer : `/legal/cgu` et `/legal/privacy`. Contenu rédigé en français, structuré en sections numérotées, avec date de dernière mise à jour. Le texte est versionné dans le repo (`content/legal/cgu-vX.Y.md`).

À l'inscription, une case à cocher obligatoire "J'ai lu et j'accepte les CGU et la Politique de Confidentialité" avec liens. L'acceptation est tracée en DB (`user_consents` : user_id, document_version, accepted_at).

Quand une nouvelle version majeure des CGU sort, les utilisateurs existants voient un modal au login leur demandant de re-accepter. Pas de re-acceptation pour les versions mineures (typos).

## Critères d'acceptation (Gherkin)

**Scénario 1 — Inscription avec acceptation**

```gherkin
Given un nouveau visiteur sur la page d'inscription
When il remplit le formulaire et coche "J'accepte les CGU et la Politique de Confidentialité"
And il soumet le formulaire
Then un compte est créé
And un record est inséré dans `user_consents` avec la version actuelle des CGU et des PC
And l'événement "consent.given" est tracé
```

**Scénario 2 — Refus d'acceptation**

```gherkin
Given un visiteur sur la page d'inscription
When il remplit le formulaire mais ne coche pas la case CGU
And il tente de soumettre
Then le bouton "Créer mon compte" est désactivé
And un helper text rouge indique "Tu dois accepter les CGU pour créer un compte"
```

**Scénario 3 — Re-acceptation suite à mise à jour majeure**

```gherkin
Given un utilisateur existant
And une nouvelle version majeure des CGU est publiée (v2.0)
When il se connecte
Then un modal s'affiche "Nos CGU ont été mises à jour"
And il doit cocher "J'accepte" pour continuer
And un nouveau record user_consents v2.0 est créé
And s'il refuse, son compte est verrouillé (statut "pending_consent")
```

## Cas limites à traiter

- **Modification mineure (typo)** : pas de re-acceptation, version mineure incrementée silencieusement.
- **Modification majeure (clause nouvelle)** : re-acceptation forcée, ancien consentement archivé.
- **Utilisateur en pending_consent qui refuse** : compte verrouillé, support contact requis.
- **Versionning** : table `legal_documents_versions` (kind, version_major, version_minor, published_at, content_md).
- **Lien direct vers une ancienne version** : accessible via `/legal/cgu?v=1.0`, mais l'utilisateur est averti "Tu consultes une version archivée".
- **i18n** : versions EN dès le MVP via fichiers `content/legal/cgu-vX.Y.en.md`.

## Non-goals

- Pas d'éditeur WYSIWYG des CGU dans l'app (édition par PR markdown).
- Pas de génération automatique de versions PDF des documents (Nuxt rend HTML, suffisant).
- Pas d'historique consultable par l'utilisateur de ses consentements (audit log admin suffit).
- Pas de support multi-juridictions (FR + EN uniquement au MVP).

## Tests à écrire

- **Unit** :

  - Détection version majeure vs mineure : `1.0 → 1.1` mineure, `1.0 → 2.0` majeure
  - Composant `LegalConsentCheckbox` : disabled si non coché

- **Integration** :

  - `POST /api/me/consents` insère un record et incrémente compteur
  - Middleware `requireFreshConsent` : redirige si version majeure non acceptée

- **E2E (Playwright)** :

  - Inscription : sans cocher → bouton désactivé. Avec cocher → compte créé.
  - Re-acceptation : publier v2.0, se reconnecter, modal apparaît
  - Refus de re-acceptation → compte verrouillé

- **A11y (axe-core)** :

  - Checkbox avec label cliquable, lien CGU accessible clavier
  - Modal de re-acceptation : focus piégé, Esc bloqué (acceptation obligatoire)
  - Pages CGU/Privacy : structure heading hiérarchique correcte

- **Security** :

  - Tentative de bypass du consentement via appel direct register → refus
  - Impossible de falsifier la version acceptée (server-side verify)

- **Legal** :

  - Review par counsel de la complétude RGPD (article 13/14)

## Observabilité

- **Logs** :

  - `consent.given` (info, payload : `user_id`, `kind`, `version`)
  - `consent.refused` (warn, payload : `user_id`, `kind`)
  - `legal.document.published` (info, payload : `kind`, `from_version`, `to_version`)

- **Métriques** :

  - Compteur `cursus.consents.given.total` labellisé par `kind`
  - Compteur `cursus.consents.refused.total`

- **Alertes** : pic de refus de re-acceptation > 30 % → enquête.

## Considérations sécurité

- Acceptation stockée avec hash de la version acceptée (preuve de quoi exactement a été accepté).
- Pas de modification rétroactive du contenu d'une version publiée (immuabilité).
- Audit log de chaque consentement et refus.
- Pas d'inscription possible sans consentement (côté serveur, pas seulement client).

## Considérations accessibilité

- Pages légales : structure sémantique (h1/h2/h3, listes, sections), table des matières.
- Liens CGU/PC : focus visible, ouverts dans le même onglet (sinon avertir).
- Checkbox de consentement : label cliquable, état announced via `aria-checked`.
- Texte : taille minimum 16px, ligne 1.5, contraste AAA.
- Modal de re-acceptation : pas de close button (acceptation requise), mais lien "Quitter sans accepter" visible et accessible.

## Considérations performance

- Pages légales rendues SSR depuis Markdown (cache HTTP long).
- Lookup de la version courante des CGU caché en mémoire app (invalidation lors de publish).
- Cible : page CGU TTFB < 200 ms.
