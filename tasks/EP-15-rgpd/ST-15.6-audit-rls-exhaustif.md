## Contexte business

Le playbook (§6.2) impose RLS sur toutes les tables sensibles. Au-delà du "on a une RLS policy", il faut **prouver** que la RLS fonctionne dans tous les cas, notamment les cas négatifs : un utilisateur ne peut pas lire les données d'une autre cohorte, un formateur ne peut pas éditer une cohorte qui ne lui est pas assignée, etc. Cette story est un audit exhaustif via tests d'intégration de toutes les RLS policies.

Une RLS manquante ou trop permissive = fuite de données. C'est le bug n°1 à éviter dans une app multi-utilisateurs.

## Description fonctionnelle

Livrable : un dossier `tests/rls/` contenant un test par table sensible, chacun couvrant 3-5 cas négatifs ("un user X ne peut PAS lire/écrire/supprimer une row appartenant à un user Y") + cas positifs ("un user X peut bien lire/écrire les siennes"). Tests exécutés contre une instance Supabase de test avec seed dataset multi-tenant simulé.

Accompagné d'une **matrice RLS** dans `docs/security/rls-matrix.md` : tableau qui mappe pour chaque table, par rôle (stagiaire, formateur_principal, co_formateur, admin), ce qui est autorisé en SELECT/INSERT/UPDATE/DELETE.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Test négatif sur submissions**

```gherkin
Given 2 stagiaires "alice" et "bob" dans des cohortes différentes
When "alice" tente de SELECT les submissions de "bob"
Then 0 row est retournée (RLS filtre)
And aucune erreur explicite n'est levée (comportement Postgres normal)
```

**Scénario 2 — Test positif sur submissions**

```gherkin
Given le stagiaire "alice" a 3 submissions
When "alice" SELECT * FROM submissions
Then elle voit exactement ses 3 submissions
```

**Scénario 3 — Formateur cross-cohorte**

```gherkin
Given un formateur principal "mohamed" de la cohorte A
And une autre cohorte B avec un autre formateur "sara"
When "mohamed" tente de UPDATE une submission de la cohorte B
Then l'opération retourne 0 rows updated (RLS)
And un test assère explicitement le 0
```

**Scénario 4 — Couverture exhaustive**

```gherkin
Given la matrice docs/security/rls-matrix.md liste 14 tables sensibles
When pnpm test:rls est exécuté
Then tous les tests passent (positifs + négatifs)
And le coverage script vérifie qu'au moins 1 test négatif existe par couple (table, rôle)
```

## Cas limites à traiter

- **Service role bypass** : tests qui vérifient que la RLS est bien appliquée quand on utilise l'`auth.uid()` context (pas le service_role).
- **JWT expiré** : tentative avec un token expiré → 401 (pas une fuite).
- **Rôle anonyme** : vérifier qu'un user non auth ne peut accéder à rien (sauf profils publics et vérif certificat).
- **Co-formateurs assignés à un module précis** : RLS plus fine, test du scope correct.
- **Override admin** : admin a tous droits, test que sa "power" est complète mais auditée.
- **Transition de cohorte (stagiaire archivé)** : l'ancien stagiaire peut toujours lire son propre profil ; le formateur ne peut plus écrire.

## Non-goals

- Pas de fuzz testing RLS (sur-dimensionné au MVP).
- Pas de tests RLS sur les vues matérialisées (story v1.x si on en utilise).
- Pas de tests RLS sur les Edge Functions (couverts dans story dédiée si ajoutées).

## Tests à écrire

- **Unit** : non applicable (cette story EST une suite de tests).
- **Integration RLS** : au moins 50 tests au total (positifs + négatifs).
- **Coverage check** : script qui parcourt les fichiers tests et vérifie qu'au moins 1 test négatif existe par couple (table_sensible, rôle_app).
- **E2E (Playwright)** : un parcours "un user fait UI action sur cohorte d'un autre" → erreur attendue.
- **Security** : red team manuelle (sur staging) : tenter d'accéder à des données non liées via URL crafting.

## Observabilité

- **Logs** :

  - `rls.test.failed` (warn en CI, pas en prod) si un test fait remonter une fuite

- **Métriques** :

  - Compteur en CI : `rls_tests_total`, `rls_negative_tests_total`

- **Alertes** : aucune en prod (audit one-shot).

## Considérations sécurité

- Tests s'exécutent contre un Supabase de test (jamais prod).
- Seed dataset reset entre tests pour reproductibilité.
- Pas de service role dans les tests RLS (sinon bypass).
- En cas de RLS manquante : la PR est bloquée en CI.
- Audit de revue périodique (trimestriel) des policies par security champion.

## Considérations accessibilité

- Non applicable (tests serveur).
- La matrice docs/security/rls-matrix.md est rédigée accessible (tableau sémantique).

## Considérations performance

- Suite de tests RLS : cible < 60 s en CI.
- Seed dataset minimal pour rapidité (10-15 rows par table).
- Parallélisation Vitest activée.
