## Contexte business

Un compte qui émet des certificats vérifiables ne peut pas accepter de mots de passe faibles ni laisser un attaquant tenter 1000 combinaisons. Sans politique de mot de passe forte et sans verrouillage anti-brute-force, Cursus s'expose au compromise de comptes formateur (avec impacts énormes : émission frauduleuse de certificat, accès aux données stagiaires). Cette story est un pilier de la sécurité "par défaut" du playbook.

La politique doit être pragmatique : suffisamment forte pour bloquer les attaques de masse, suffisamment humaine pour ne pas pousser les utilisateurs à écrire leurs mots de passe sur des post-it.

## Description fonctionnelle

**Politique de mot de passe** (à la création et au reset) :

- Minimum 12 caractères
- Pas de pattern trivial : pas d'enchaînement clavier ("qwerty", "azerty"), pas de séquences ("12345")
- Pas dans la liste des 100 000 mots de passe les plus compromis (vérification via API HaveIBeenPwned k-anonymity ou liste embarquée)
- Pas l'email, le prénom ou le nom
- Force estimée ≥ "Strong" via `zxcvbn` (score ≥ 3)
- Feedback live dans l'UI : barre de force + raison du refus

**Verrouillage de compte** :

- 5 tentatives de login échouées en 15 min → compte temporairement verrouillé pour 15 min
- 10 tentatives en 1h → verrouillage 1h + email à l'utilisateur "Tentatives de connexion suspectes"
- 20 tentatives en 24h → verrouillage 24h + email + alerte admin
- Reset par lien magic-link possible à tout moment (pas de blocage des resets)

## Critères d'acceptation (Gherkin)

**Scénario 1 — Création de mot de passe faible refusée**

```gherkin
Given un visiteur sur la page d'inscription
When il saisit le mot de passe "password123"
Then l'UI affiche une barre rouge "Trop faible"
And un message "Mot de passe trop commun"
And le bouton "Créer" est désactivé
```

**Scénario 2 — Verrouillage après 5 tentatives**

```gherkin
Given un utilisateur tente de se connecter
When il échoue 5 fois en 15 minutes
Then la 6ᵉ tentative renvoie "Compte temporairement verrouillé. Réessaie dans 15 minutes ou reset ton mot de passe."
And l'événement "auth.account.locked" est tracé
And les tentatives suivantes renvoient le même message (pas de leak sur la vraie raison)
```

**Scénario 3 — Reset par magic-link en cas de verrouillage**

```gherkin
Given un compte verrouillé
When l'utilisateur demande un reset par magic-link
Then il reçoit l'email normalement (pas de blocage du reset)
And quand il clique, il peut définir un nouveau mot de passe
And le verrouillage est levé automatiquement
```

## Cas limites à traiter

- **Attaquant distribué (botnet)** : rate limit par IP en plus du compteur par compte (5/min/IP sur `/auth/login`).
- **Utilisateur qui oublie son mot de passe et tente 4 fois** : pas encore verrouillé, hint "Reset ton mot de passe" affiché dès la 3ᵉ tentative.
- **Compte verrouillé après absence longue** : le compteur de tentatives a une fenêtre glissante (pas une journée fixe).
- **Réutilisation d'un ancien mot de passe** : interdit (vérification contre les 5 derniers hashes en DB).
- **Mots de passe pasteurs (Unicode bizarre)** : NFC normalization avant hash pour stabilité.
- **Username enumeration via reset password** : message générique "Si ce compte existe, un email a été envoyé".

## Non-goals

- Pas de 2FA dans cette story (story dédiée v1.1, mais cohérent avec axe Premium 5).
- Pas de password manager intégration (autocomplete="current-password" suffit côté navigateur).
- Pas d'expiration forcée des mots de passe (anti-pattern moderne — NIST SP 800-63B).
- Pas de CAPTCHA au MVP (rate limit suffit ; ajout si abus constaté).

## Tests à écrire

- **Unit** :

  - `validatePasswordStrength()` : matrice de mots de passe (faibles refusés, forts acceptés)
  - Détection contextuelle : un mot de passe = email du user → refus
  - HaveIBeenPwned k-anonymity : hash partiel envoyé, jamais le complet

- **Integration** :

  - 5 tentatives → 6ᵉ retourne lockout
  - Reset password lève le lockout
  - Réutilisation d'un ancien mot de passe → refus

- **E2E (Playwright)** :

  - Inscription avec mot de passe faible → refus client + server
  - Brute force simulé → lockout après 5 tentatives → reset par magic-link déverrouille
  - Email "tentatives suspectes" reçu après 10 tentatives

- **A11y (axe-core)** :

  - Barre de force : `aria-valuenow`, texte alternatif (jamais que couleur)
  - Messages d'erreur annoncés via `aria-live`

- **Security** :

  - Timing attack : login réussi vs échec → même temps de réponse (constant time compare)
  - Username enumeration : reset password sur email inexistant → même UX qu'existant
  - Brute force distribué via VPN : rate limit IP-based ne suffit pas seul, on combine email

- **Performance** :

  - Validation mot de passe < 100 ms (zxcvbn local)

## Observabilité

- **Logs** :

  - `auth.login.success` (info, payload : `user_id_hash`, `ip_hash`)
  - `auth.login.failure` (info, payload : `email_hash`, `ip_hash`, `reason`)
  - `auth.account.locked` (warn, payload : `user_id_hash`, `lockout_duration`)
  - `auth.password.weak_rejected` (warn)

- **Métriques** :

  - Compteur `cursus.auth.login.total` labellisé par `result`
  - Compteur `cursus.auth.lockouts.total`
  - Gauge `cursus.auth.locked_accounts.current`

- **Alertes** :

  - Pic de `auth.login.failure` > 100/min → possible brute force.
  - `auth.account.locked` > 10/heure → investigation.

## Considérations sécurité

- Constant-time compare pour les comparaisons de hash (timing attacks).
- Hash des emails dans les logs (pas en clair, pour conformité RGPD).
- HaveIBeenPwned k-anonymity (pas de fuite du hash complet).
- Rate limit IP+email combiné (Upstash Redis).
- Sessions invalidées au reset password (force re-login).
- Bcrypt cost factor ≥ 12.

## Considérations accessibilité

- Barre de force : icône + texte + couleur (jamais couleur seule).
- Feedback live : annoncé via `aria-live="polite"` (pas spam).
- Bouton "Afficher le mot de passe" : toggle accessible clavier.
- Messages d'erreur : explicites et actionnables ("Ton mot de passe est trop commun. Essaie une phrase de 4 mots.").

## Considérations performance

- zxcvbn lazy-chargé (gros bundle \~700kb) uniquement sur pages auth.
- API HaveIBeenPwned async, timeout 1.5s, fallback liste locale.
- Cible : validation client + server < 300ms total.
- Index DB : `idx_auth_attempts_email_attempted_at`.
