## Contexte business

Malgré les meilleurs efforts préventifs, un incident sécurité peut survenir : compte compromis, fuite d'une clé API, attaque DDoS, bug critique exploité. Sans un **runbook** écrit à froid, l'équipe improvisera à chaud — c'est la garantie d'oubli d'étape critique (notification CNIL sous 72h, par exemple). Cette story livre la **procédure de réponse à incident** : chronologie type, rôles, communication interne/externe, obligations légales.

C'est aussi une exigence implicite des conformités ISO 27001 / SOC 2 (controls IR-1 à IR-8).

## Description fonctionnelle

Livrable principal : un document `docs/security/incident-response-runbook.md` structuré en 6 phases :

1. **Détection** : sources possibles (alerte Sentry, signalement utilisateur, audit log anomalie, bug bounty), criticité (P0 à P3).
2. **Containment** : actions immédiates pour limiter la fuite (rotation clés, suspension comptes, kill-switch via feature flag, rollback).
3. **Éradication** : correctif technique (patch, hotfix, mise à jour dépendances).
4. **Recovery** : remise en service, monitoring renforcé.
5. **Notification** : interne (Slack #security, hiérarchie), externe (utilisateurs impactés, CNIL sous 72h si données personnelles fuites, partenaires).
6. **Postmortem blameless** sous 48h, plan d'action publié.

Document complété par un **template de postmortem**, une **liste de contacts** (DPO, counsel, ops, fournisseurs critiques), et une **checklist criticité P0** imprimable.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Compromise d'un compte formateur**

```gherkin
Given le runbook est en place
When un compte formateur est suspecté compromis (login depuis IP inconnue + actions anormales)
Then un on-call applique la phase "Containment" :
  - révoque toutes les sessions du compte
  - force reset mot de passe
  - rotation des clés d'intégration impactées (GitHub App, etc.)
And la phase "Notification" précise : DPO + admin + utilisateur ciblé
```

**Scénario 2 — Fuite clé API (commit accidentel)**

```gherkin
Given un secret est commité dans le repo
When la détection se fait (gitleaks pre-push, Trivy CI ou alerte GitHub Secret Scanning)
Then le runbook impose immédiatement :
  - rotation clé dans le provider concerné (< 15 min)
  - retrait commit + force-push history (filter-repo)
  - audit des logs pour usage suspect
And un postmortem est planifié sous 48h
```

**Scénario 3 — Fuite de données personnelles**

```gherkin
Given une fuite confirmée affecte > 1 utilisateur
When la phase Notification démarre
Then le DPO est informé sous 1h
And une déclaration CNIL est préparée en parallèle (modeling 72h)
And les utilisateurs impactés sont notifiés individuellement par email
And un modèle de communication est dans le runbook
```

## Cas limites à traiter

- **Faux positif d'alerte** : runbook prévoit une étape "Vérifier la nature de l'alerte" avant escalade.
- **Incident hors-heures** : rotation on-call avec PagerDuty (ou équivalent), 1 personne joignable 24/7 minimum.
- **Indisponibilité du DPO** : adjoint désigné + matrix RACI à jour.
- **Chaîne de dépendances tierces compromise** (Supabase, Vercel) : escalation au support tiers + plan de continuité (export DB, fallback static).
- **Couverture médiatique potentielle** : préparer un statement de crise et identifier un porte-parole.

## Non-goals

- Pas de SOAR (Security Orchestration) automatisé — sur-dimensionné.
- Pas de bug bounty au MVP (ouverture v1.x).
- Pas d'équipe SOC dédiée (équipe Cursus assume rôle).
- Pas de SIEM enterprise — logs structurés + Sentry suffisent au MVP.

## Tests à écrire

- **Tabletop exercise** : 1 scénario fictif par trimestre (compte compromis, fuite clé, DDoS).
- **Test des kill-switch flags** : vérifier qu'activer le flag coupe bien la fonctionnalité < 30 s.
- **Test de rotation de clé** : un dry-run de rotation des clés GitHub App + GH webhook secret.
- **Test de communication** : envoyer un email de test "violation" sur compte de test, vérifier rendu.

## Observabilité

- **Logs** :

  - `incident.detected` (warn, payload : `source`, `severity`)
  - `incident.contained` (info)
  - `incident.resolved` (info, payload : `duration_minutes`, `users_impacted`)

- **Métriques** :

  - Compteur `cursus.incidents.total` labellisé par `severity`
  - Gauge `cursus.incidents.active.current`

- **Alertes** : auto-création d'un incident en cas de Sentry burst (≥ 50 erreurs / 5 min).

## Considérations sécurité

- Runbook stocké en clair dans le repo (pas de secret dedans).
- Liste de contacts : versionnée dans `docs/` mais avec uniquement rôles / aliases (pas d'emails personnels).
- Tabletop exercises ne dévoilent pas de vraies infos (utilisation d'un environnement de staging).
- Secrets prêts pour rotation rapide (clés GitHub App, HMAC harnais, signature certif).

## Considérations accessibilité

- Runbook structuré (h2/h3, listes), index, table des matières.
- Checklist P0 imprimable accessible (texte simple, no-image).
- Modèles emails utilisateurs : ton clair, action requise mise en évidence.

## Considérations performance

- Activation kill-switch < 30 s (feature flag store caché mais TTL court).
- Rotation clé GitHub App testable en < 5 min sans downtime utilisateur.
- Sentry burst alerte < 1 min après seuil.
