## Contexte business

Catalogue de checks paramétrables, **base du harnais**. Sans ces actions, le harnais ne sait rien vérifier. C'est l'investissement le plus important d'EP-06 (8 pts).

Valeur livrée : 9 checks au MVP couvrant les cas essentiels (repo, branches, PR, fichiers, tests, linter, URL, Lighthouse, commits signés), chacun comme une action GitHub composable réutilisable.

## Description fonctionnelle

Bibliothèque de 9+ checks au MVP, chacun comme une action GitHub composable, recevant ses paramètres en input et produisant un résultat structuré `{check_id, status, message, details?}`.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Exécution cohérente**

```gherkin
Given une liste de checks est définie pour un module
When le harnais s'exécute
Then chaque check produit { check_id, status, message, details? }
And les résultats sont agrégés et postés au webhook
And l'ordre des checks dans la spec est respecté
```

**Scénario 2 — Check timeout individuel**

```gherkin
Given un check timeout (>2 min)
When le timeout est atteint
Then le check est marqué en erreur "timeout"
And la suite des checks continue
```

**Scénario 3 — Repo n'existe plus**

```gherkin
Given le repo n'existe plus pendant le run
When le 1er check détecte 404
Then tous les checks suivants short-circuitent en "skipped"
And le rapport indique la cause racine
```

## Checks au MVP

1. `repo_exists_public` (params : url_pattern) — Vérifie que le repo existe et est public
2. `branch_exists` (params : branch_names\[\]) — Vérifie la présence de branches spécifiques
3. `pr_merged_with_strategy` (params : source, target, strategy) — Vérifie un merge via PR
4. `file_exists` (params : path, content_regex?) — Vérifie l'existence d'un fichier
5. `tests_pass` (params : directory, command) — Exécute les tests
6. `linter_pass` (params : type=eslint|ruff|...) — Lint le code
7. `url_responds` (params : url, expected_status, timeout) — Vérifie qu'une URL répond
8. `lighthouse_min` (params : url, perf, a11y, best-practices, seo thresholds) — Mesure Lighthouse
9. `commits_signed` (params : min_count) — Vérifie les commits signés GPG

## Cas limites à traiter

- **Check qui timeout individuellement** (>2 min) : marqué en erreur, suite des checks continue.
- **Repo qui n'existe plus pendant le run** : tous les checks suivants short-circuitent en "skipped".
- **Param invalide** : check validé côté cursus builder (ST-03.4).
- **Network flaky** : retry intern 2× par check.

## Non-goals (hors scope)

- Pas de check custom GitHub Actions au MVP (v1.x).
- Pas de checks de performance avancés (load test, etc.).
- Pas de check d'écritures Git (uniquement read).
- Pas de checks d'accessibilité du code (autre que via Lighthouse).

## Tests à écrire

- **Pour chaque check** : test happy + test failure + test edge (9 × 3 = 27 tests min).
- **Test sur le catalogue complet** en parallèle (9 checks ensemble).
- **Tests de timeout** sur chaque check.
- **Property-based** : configurations aléatoires.

## Observabilité

- **Métriques par check** : `harness.check.<id>.duration.seconds`, `harness.check.<id>.result`.
- **Logs** : chaque check produit un log structuré.
- **Alertes** : si un check a un taux d'échec > 30% sustained, investiguer (bug ou trop strict).

## Considérations sécurité

- Pas d'exécution de code stagiaire (uniquement read).
- Sanitize params (pas de chemin `/etc/passwd`).
- Tests `tests_pass` et `linter_pass` : exécutés en sandbox (Docker GitHub Actions).
- Pas de leak de secrets dans les logs des checks.

## Considérations performance

- **Parallélisation** : les checks indépendants run en parallèle.
- **Timeout strict** : 2 min par check, 10 min total.
- **Cache GitHub Actions** : checkout cache pour accélérer.

## Definition of Done spécifique

- 9 checks implémentés avec tests happy + failure + edge.
- Documentation par check dans `docs/harness/checks/`.
- Test catalogue complet sur 4 repos fixtures.
