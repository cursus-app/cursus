## Contexte business

À chaque soumission, déclencher un workflow GitHub Actions qui analyse le repo du stagiaire et poste le résultat. C'est l'**hypothèse #1 risquée** du produit (cf ST-01.6). Sans ce service, aucun harnais automatique.

Valeur livrée : trigger automatique du harnais via GitHub App, queue avec retry + DLQ, webhook sécurisé, persistance fiable des résultats.

## Description fonctionnelle

GitHub App "Cursus Harness" + repo central `cursus-harness-runner` sur l'org GitHub. À chaque submission : appel `workflow_dispatch` avec payload (submission_id, repo_url, deploy_url, criteria_json). Le workflow termine en posting webhook vers Cursus. Service `harness.ts` orchestrant. Queue Inngest avec retry exponentiel + DLQ.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Déclenchement workflow**

```gherkin
Given une submission est créée
When le service Harness est appelé
Then un workflow_dispatch est déclenché sur le repo cursus-harness-runner
And le payload contient submission_id, repo_url, criteria_json
And un HarnessRun est créé en DB en statut "queued"
```

**Scénario 2 — Réception webhook**

```gherkin
Given le workflow termine
When le POST webhook arrive avec signature HMAC valide
Then le HarnessRun passe à "completed" avec les résultats stockés
And la Submission est mise à jour
And un événement Supabase Realtime est émis
```

**Scénario 3 — Timeout du workflow**

```gherkin
Given le workflow timeout (10 min)
When le timeout GH Actions est atteint
Then le HarnessRun passe à "timeout"
And la Submission reste en "pending" pour pouvoir resoumettre
```

**Scénario 4 — Signature HMAC invalide**

```gherkin
Given un webhook arrive avec signature HMAC invalide
When le serveur le reçoit
Then il répond 401
And un log security est émis
And l'événement n'est PAS persisté
```

## Cas limites à traiter

- **GitHub Actions down** : workflow ne démarre pas → statut "queued" trop longtemps → job de surveillance détecte et alerte après 5 min.
- **Webhook arrive en double** : idempotence par run_id.
- **Signature HMAC invalide** : 401 + log security critique.
- **GitHub API rate limit** : retry exponentiel via Inngest.
- **Repo qui n'existe plus** : workflow échoue proprement.

## Non-goals (hors scope)

- Pas de runners self-hosted au MVP.
- Pas de parallélisation extrême (sera évalué v1.x).
- Pas de retry automatique d'un harnais failed (le stagiaire resoumet).
- Pas de cache des résultats entre runs.

## Tests à écrire

- **Integration** : déclenchement → webhook → DB.
- **Security** : webhook avec mauvaise signature → 401.
- **Security** : tentative de replay webhook → détectée.
- **Tests sur repos fixtures** (voir ST-16.5).
- **Resilience** : Inngest retry sur échec.

## Observabilité

- **Métriques** : `harness.runs.total`, `harness.duration.seconds` (histogram).
- **Logs structurés** : run_id, submission_id, duration, status.
- **Alertes** : si `harness.duration.p95` > 7 min sustained.
- **Alertes** : si webhook signature invalid > 5 en 10 min = potential attack.

## Considérations sécurité

- **HMAC** sur tous les webhooks (SHA256, secret rotatable).
- **GitHub App private key** : stockée chiffrée dans Vault, jamais logée.
- **Webhook URL** : non-publique (préférer endpoint sur sous-domaine).
- **Idempotence par run_id** : pas de double execution.
- **Pas d'exécution de code arbitraire** : workflow YAML statique, pas injecté.
- **Audit log** : tous les triggers de harnais.

## Considérations performance

- **P95 cible** : 5 min entre submission et résultat affiché.
- **Throughput** : 100 harnais/h soutenu.
- **Webhook handler** : < 100ms (juste persister, traitement async).

## Definition of Done spécifique

- E2E : submission → harnais → webhook → DB → UI update.
- Test résilience : tuer le worker mid-run, retry succeeds.
- HMAC validation vérifiée.
