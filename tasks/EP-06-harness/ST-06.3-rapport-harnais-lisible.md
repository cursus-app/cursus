## Contexte business

Le rapport est ce que le stagiaire **lit en premier**. Doit être actionnable, pas un dump de log. Un rapport illisible = stagiaire qui abandonne. C'est la matérialisation visible du pilier "feedback rapide et utile".

Valeur livrée : rapport en cartes par check, icônes claires, messages humains, accordion vers détails techniques. Lisible par n'importe quel stagiaire débutant.

## Description fonctionnelle

Composant `HarnessReport` qui affiche 1 carte par check avec : icône (✅/❌/⚠️/⏳), titre humain (pas "check_branch_exists" mais "Les branches sont-elles présentes ?"), message d'aide, accordion vers les détails techniques bruts. Dictionnaire de messages par check_id.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Affichage standard**

```gherkin
Given un HarnessRun completed
When le composant rend
Then chaque check affiche : icône, titre humain, message lisible
And si failed : un message expliquant ce qu'il manque (ex : "La branche feature/login n'existe pas dans ton repo")
And un bouton "Voir les détails techniques" déploie le log brut
```

**Scénario 2 — Check en erreur (crash interne)**

```gherkin
Given un check est en erreur (pas failed mais crash interne)
When il est affiché
Then une icône d'avertissement + message "Vérification impossible, contacte ton formateur"
And les détails techniques sont accessibles pour debug
```

**Scénario 3 — Respect reduced motion**

```gherkin
Given prefers-reduced-motion est actif
When le rapport apparaît
Then pas d'animation cascade
And les cartes apparaîssent immédiatement
```

## Cas limites à traiter

- **Check non documenté** : titre = id technique avec warning + lien doc.
- **Très long log brut** : pagination ou truncate avec "voir plus".
- **Tous les checks ✅** : message global "Bien joué !".
- **Tous ❌** : message global empathique (pas de honte).
- **Mix** : message neutre "X/Y checks validés".

## Considérations sécurité

- Sanitize les messages renderés (DOMPurify si HTML).
- Pas d'exécution de code dans les logs bruts (juste affichage).
- Pas de PII dans les messages humains.

## Considérations accessibilité

- **Statut icône + texte** (pas seulement couleur — daltoniens).
- **Structure semantique** : `<section>`, headings.
- **Accordion** : ARIA expanded/collapsed, navigation clavier.
- **Annonce des statuts** au lecteur d'écran.
- Contraste 4.5:1 sur tous les statuts.
- Respect `prefers-reduced-motion`.

## Considérations performance

- Rapport : virtualization si > 30 checks.
- Lazy load des détails techniques (uniquement à l'expand).

## Definition of Done spécifique

- Storybook avec 5 états (pending/success/failure/error/skipped).
- A11y axe-core : 0 erreur critique.
- Dictionnaire FR + EN pour les 9 checks.
