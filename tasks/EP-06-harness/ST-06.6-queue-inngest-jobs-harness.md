## Contexte business

Les jobs harnais (déclenchement workflow, attente résultat) doivent être **robustes** : retry, DLQ, observabilité. Sans queue, un échec réseau perd la submission et le stagiaire est bloqué.

Valeur livrée : orchestration fiable des jobs harnais via Inngest (free tier généreux), retry exponentiel automatique, dead letter queue pour les échecs persistants, dashboard de debug.

## Description fonctionnelle

Utiliser Inngest (free tier généreux) pour orchestrer les jobs : `triggerHarness`, `handleWebhook`, `cleanupStaleRuns`. Retry exponentiel (3 tentatives). DLQ. Dashboard Inngest visible pour debug. Métriques exposées.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Retry exponentiel**

```gherkin
Given une submission est créée
When le job `trigger-harness` est enqueué
Then Inngest l'exécute avec retry exponentiel en cas d'échec (3 tentatives : 1s, 5s, 25s)
```

**Scénario 2 — Dead Letter Queue**

```gherkin
Given le job échoue 3 fois
Then il est placé en Dead Letter Queue
And une alerte est levée (Sentry critical)
And la submission reste en "queued" en attendant intervention
```

**Scénario 3 — Succès**

```gherkin
Given le job réussit (éventuellement après retry)
Then il met à jour le HarnessRun en DB
And l'événement Realtime est émis
```

## Cas limites à traiter

- **Inngest down** : fallback synchrone (basique, dégradé) avec retry simple.
- **Job qui prend plus que le timeout Inngest** : split en sous-jobs.
- **Race condition** : webhook arrive avant la fin du job trigger → idempotence par run_id.

- Pas de fanout massif au MVP.

## Tests à écrire

- **Test** : retry sur échec (mock Inngest).
- **Test** : timeout déclenche le bon comportement (split).
- **Integration** : DLQ reçoit bien les échecs persistants.
- **Resilience** : tuer le worker mid-run, retry succeeds.

## Observabilité

- **Dashboard Inngest** = source de vérité pour debug.
- **Métriques propres** : `harness.jobs.queued.gauge`, `harness.jobs.dlq.total`.
- **Logs** : `inngest.job.started`, `inngest.job.failed`, `inngest.job.dlq`.
- **Alertes** : si DLQ > 5 en 1h = investigation urgente.

## Considérations sécurité

- Inngest signing key stockée dans Vault.
- Webhook Inngest → Cursus avec signature verification.
- Pas de PII dans les payloads de jobs.

## Considérations performance

- Capacité : 100 jobs/min sustained (free tier OK pour MVP).
- Latence enqueue : < 100ms.
- Latence démarrage job : < 5s.
