## Contexte business

Pour gérer les cas exceptionnels où le harnais rate à tort (faux négatif), le formateur doit pouvoir valider manuellement. Sans ça, le stagiaire est bloqué et le formateur impuissant.

Valeur livrée : override 1-clic par formateur avec motif obligatoire archivé en audit, transparence côté stagiaire ("Validé manuellement").

## Description fonctionnelle

Bouton "Valider manuellement" sur la fiche submission côté formateur. Modal avec motif obligatoire (texte libre, min 20 chars). Statut passe à "validated_override". Audit log avec before/after. Badge "Validé manuellement" visible.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Override avec motif**

```gherkin
Given un formateur est sur la fiche d'une submission failed
When il clique "Valider manuellement"
Then une modal demande un motif (texte libre, min 20 chars)
And il valide
Then le statut passe à "validated_override"
And une entrée audit est créée avec user, motif, before/after
And le portfolio + XP s'appliquent comme pour une validation normale
And un tag "Validé manuellement" est visible (transparence)
```

**Scénario 2 — Submission déjà validée**

```gherkin
Given une submission est déjà validée
When le formateur tente un override
Then l'action est refusée (pas nécessaire)
```

**Scénario 3 — Override d'un override**

```gherkin
Given une submission est en "validated_override"
When le formateur tente un nouvel override
Then refus (déjà override)
```

## Cas limites à traiter

- **Submission déjà validée** : pas d'override possible (refus).
- **Override d'un override** : refus (déjà override).
- **Motif < 20 chars** : refus côté client + serveur.
- **Révocation d'override** : possible via flow séparé admin (hors scope MVP).

## Sous-tâches techniques

- **TT-06.5.1** — Endpoint `POST /api/submissions/:id/override`

- **TT-06.5.3** — Audit log entry obligatoire
- **TT-06.5.4** — Badge "Validé manuellement" sur la submission (visible stagiaire)

## Dépendances

- **Bloqué par** : ST-02.4 (CUR-35), ST-08.4 (CUR-67) audit log.

## Non-goals (hors scope)

- Pas d'override en masse au MVP.
- Pas de révocation d'override automatique.
- Pas d'override par co-formateur sans permission.

## Tests à écrire

- **E2E (Playwright)** : override happy path.
- **Security** : non-formateur ne peut pas override.
- **Integration** : audit log créé correctement.
- **Unit** : transitions de statut.

## Observabilité

- **Logs** : `submission.overridden` (info, avec user, motif).
- **Métriques** : `submissions.overridden.total`.
- **Alertes** : si taux d'override > 30% sustained = indicateur que le harnais est trop strict.

## Considérations sécurité

- RLS : formateur principal ou co-formateur du module uniquement.
- Audit log obligatoire (immuable).
- Motif sanitisé (pas d'XSS).
- Pas de PII du stagiaire dans le motif (formateur sensibilisé).

## Considérations accessibilité

- Modal : `role="dialog"`, focus piege.
- Motif : label clair, compteur de caractères, `aria-describedby`.
- Badge "Validé manuellement" : visible et annoncé.

## Considérations performance

- Override : < 500ms p95.
- Pas de chemin critique.

## Definition of Done spécifique

- E2E override formateur → statut + badge + audit log.
- RLS test : co-formateur non assigné ne peut pas override.
- Audit log contient bien before/after.
