## Contexte business

Le stagiaire ne doit pas rafraîchir manuellement pour voir le résultat du harnais. Expérience premium = push automatique. Sans Realtime, le stagiaire F5 toutes les 30s pendant 5 min — frustration garantie.

Valeur livrée : résultat pushé en temps réel, UI auto-mise à jour, reconnexion automatique, fallback polling pour les firewalls bloquants.

## Description fonctionnelle

Subscription Supabase Realtime sur la table `harness_runs` filtrée par run_id de la submission en cours. Push de l'événement à l'UI. Reconnexion automatique. Fallback polling 10s si WebSocket bloqué.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Push temps réel**

```gherkin
Given un stagiaire soumet un livrable
When le HarnessRun est créé
Then une subscription est ouverte côté client
And quand le statut change (queued → running → completed/failed), l'UI se met à jour en temps réel
```

**Scénario 2 — Persistance du résultat**

```gherkin
Given le stagiaire ferme l'onglet pendant l'analyse
When il revient plus tard
Then le rapport est déjà disponible (le résultat a été persisté)
And une notif in-app + email (si configuré) l'informe
```

**Scénario 3 — Reconnexion WS**

```gherkin
Given la connexion WebSocket est perdue
When elle se rétablit
Then la subscription se rétablit automatiquement (Supabase Realtime gère le reconnect)
And l'UI se met à jour avec l'état actuel (catch-up)
```

**Scénario 4 — Fallback polling**

```gherkin
Given la connexion WS est bloquée par firewall entreprise
When WS échoue 3 fois
Then un fallback polling toutes les 10s prend le relais
And l'utilisateur ne voit pas la différence
```

## Cas limites à traiter

- **Connexion WebSocket bloquée par firewall** (entreprise) : fallback polling 10s.
- **Multi-tabs** : chaque tab a sa propre subscription (overhead minimal).
- **Subscription leak** : cleanup au unmount du composant.
- **Réseau lent** : timeout WS élevé (60s).

## Non-goals (hors scope)

- Pas de Realtime sur d'autres entités au MVP (sauf si vraiment utile).
- Pas de presence (qui regarde quoi).
- Pas de typing indicators.
- Pas de broadcasting custom (juste DB changes).

## Tests à écrire

- **E2E (Playwright)** : soumission + observer le push.
- **Test de résilience** : couper WS, vérifier reconnect.
- **Unit** : composable cleanup au unmount.
- **Integration** : polling fallback se déclenche après 3 échecs WS.

## Observabilité

- **Métriques** : `realtime.subscriptions.active.gauge`, taux de reconnect, taux de fallback polling.
- **Logs** : `realtime.connected`, `realtime.disconnected`, `realtime.fallback_polling`.
- **Alertes** : si > 10% fallback polling = problème infra.

## Considérations sécurité

- RLS sur `harness_runs` enforced sur Realtime (Supabase vérifie).
- Pas de souscription cross-stagiaire.
- Token JWT vérifié sur chaque message WS.

## Considérations accessibilité

- Indicateur "live" : `aria-live="polite"`, pas `assertive` (pas urgent).
- Mises à jour de statut annoncées.
- Pas d'animation excessive (respect reduced motion).

## Considérations performance

- Throughput WS : 1000 subscriptions actives simultanées (capacité Supabase free tier).
- Cleanup au unmount obligatoire (évite leak mémoire).

## Definition of Done spécifique

- E2E : push reçu en < 1s après update DB.
- Test résilience : WS coupé → reconnect + catch-up.
- Fallback polling fonctionne sur browser bloquant WS.
