## Contexte business

Le cursus pilote est le **livrable pédagogique** de Mohamed : un cursus complet (10-12 semaines, \~12 modules, 1 capstone) sur "Développement web full-stack" (techno phare). Sans contenu de qualité, même l'outil parfait ne sert à rien. Cette story délivre le contenu structuré utilisable par les stagiaires.

## Description fonctionnelle

Un cursus "Web Dev Full-Stack" créé via le Cursus Builder (EP-03) : 12 modules hebdomadaires + 1 capstone. Chaque module a : titre, objectifs (3-5), ressources (4-8 liens curés : doc, tuto vidéo, article), livrable défini avec spécifications (repo, branches attendues), 5-10 questions de quiz, badge associé. Plan pédagogique : HTML/CSS → JS → Git → Node → Express → BDD → React → Auth → Deploy → Testing → Capstone.

Livrables : contenu DB (via builder + scripts seed), bibliographie publique (`docs/pilot/curriculum.md`).

## Critères d'acceptation (Gherkin)

**Scénario 1 — 12 modules complets**

```gherkin
Given le Cursus Builder est utilisé par Mohamed
When il finalise le cursus "Web Dev Full-Stack v1.0"
Then les 12 modules sont définis avec : titre, objectifs, ressources, livrable, quiz, badge
And au moins 60 % des modules ont un quiz de 5-10 questions
And chaque livrable a au moins 4 checks de harnais activés
```

**Scénario 2 — Capstone défini**

```gherkin
Given le cursus est en "Brouillon"
When Mohamed complète le capstone
Then spécifications du capstone existent : description, durée, livrables, grille évaluation
And la grille a 5 critères /4 chacun
```

**Scénario 3 — Publication**

```gherkin
Given le cursus est complet
When Mohamed clique "Publier"
Then le cursus passe en statut "Publié v1.0"
And il est disponible pour création de cohorte pilote
```

## Cas limites à traiter

- **Modules trop denses** : revoir avec design instructionnel (cap 8h/semaine).
- **Ressources périmées** : check de validité des URLs au moment publish.
- **Quiz incomplet** : warning UI + permission d'enregistrer brouillon.
- **Capstone trop ambitieux** : revue par 1 personne extérieure.
- **Surplus de contenu** : split en cursus avancé v2.

## Non-goals

- Pas de cursus pour 4 domaines au MVP — 1 seul suffit.
- Pas de localisation EN du cursus au pilote.
- Pas d'export PDF du cursus complet.

## Tests à écrire

- Smoke : tous les modules ont les champs obligatoires (DB constraint).
- Quiz : aucune question avec 0 ou plusieurs bonnes réponses incohérentes.
- Liens : check 200 sur les URLs ressources (script).

## Observabilité

- **Métriques produit** : nb modules, nb quiz questions, nb badges.
- Document live des updates dans `docs/pilot/curriculum-changelog.md`.

## Considérations sécurité

- Ressources externes : vérifier HTTPS et réputation.
- Pas de PII dans le contenu.
- Respect licences (citation, attribution).

## Considérations accessibilité

- Contenu rédigé lisible (B1).
- Ressources : préférer celles avec transcriptions/sous-titres.
- Quiz : pas de question ambigüe, formulation directe.
