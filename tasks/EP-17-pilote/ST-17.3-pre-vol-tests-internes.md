## Contexte business

Avant le pilote réel, on doit exécuter l'app en conditions « presque réelles » pendant 2 jours, avec une fausse cohorte (1-2 internes) qui simule un parcours hebdomadaire compressé sur 2 jours. Cette étape révèle les bugs UX et intégration qui n'apparaissent qu'avec un usage continu.

## Description fonctionnelle

Mohamed + 1 dev internal jouent les rôles stagiaire(s) + formateur. Création cohorte de test, invitations, semaine 1 jouée en 1 journée (soumission, harnais, blocage, alerte). Semaine 2 et capstone sur jour 2 (compressé). Liste de bugs / friction identifiées à corriger avant lancement.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Tests internes exécutés**

```gherkin
Given l'app est sur staging
When les 2 jours de tests internes se déroulent
Then le parcours stagiaire est testé en bout-en-bout
And le parcours formateur est testé (revue, override, notif)
And au moins 1 cycle complet capstone est testé
```

**Scénario 2 — Bugs documentés et traités**

```gherkin
Given la liste de bugs identifiés
When on en triage
Then les P0 (bloquants pour pilote) sont résolus avant J0
And les P1/P2 sont planifiés pendant le pilote
```

**Scénario 3 — Decision go**

```gherkin
Given aucun bug P0 restant
When Mohamed valide en équipe
Then le pilote peut démarrer (ST-17.4)
```

## Cas limites à traiter

- **Bug bloquant découvert tardivement** : prolongation pré-vol +1 jour autorisée.
- **Test pas concluant sur une feature non critique (ex : portfolio public)** : MEP avec degradation graceful + fix post-pilote.
- **Crash data accidentel** : reset complet de staging documententé.
- **Email Resend bloqué par SPF/DKIM en test** : vérifier configuration avant le test.
- **GitHub Actions rate limit** : tester sur un repo perso avec quota généreux.
