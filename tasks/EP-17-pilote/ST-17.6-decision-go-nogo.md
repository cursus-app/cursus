## Contexte business

Après le pilote, il faut prendre une **décision structurée** : on lance v1.0 ? on pivote ? on arrête ? Sans cette décision formelle, on risque la dérive (rester en "pilote étendu" indéfiniment). Cette story rend la décision objective et trace son plan d'ajustement.

## Description fonctionnelle

Meeting de décision avec stakeholders après lecture du postmortem. Trois options :

- **Go v1.0** : MVP validé, on planifie le lancement officiel (ST-17.7) avec backlog d'améliorations.
- **Go v0.5** : on continue le pilote sur 2-3 nouvelles cohortes pour confirmer (écarts détectés).
- **No-Go** : pivot ou arrêt.

Critères de Go : NPS > 30, taux validation 1ᵉ essai > 40 %, taux capstone > 70 %, satisfaction Mohamed sur outil quotidien.

Livrable : `docs/pilot/go-nogo-decision.md` avec décision, raisons, plan d'ajustement Q1.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Meeting décision**

```gherkin
Given le rapport postmortem est partagé
When le meeting de décision a lieu
Then toutes les parties prenantes ont parlé < 30 min
And la décision est documentée dans go-nogo-decision.md
```

**Scénario 2 — Plan d'ajustement**

```gherkin
Given le Go est validé
When on élabore le plan v1.0
Then on a : top 10 backlog actions P0, sprint roadmap 2 mois, ressources requises
```

**Scénario 3 — No-Go ou pivot**

```gherkin
Given les critères ne sont pas atteints
When on décide pivot
Then une analyse de cause + alternatives est documentée
And un nouveau pilote ou pivot est planifié
```

## Cas limites à traiter

- **Décision difficile** : facilitateur ext (1h consulting).
- **Critères partiels** : Go conditionnel à résolution X.
- **Stakeholder désaccord** : escalade arbitrage.
- **Échantillon trop petit pour conclure** : extension obligatoire.
- **Burn-out de Mohamed** : pause obligatoire avant décision.
