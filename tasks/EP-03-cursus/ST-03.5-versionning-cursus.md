## Contexte business

Quand un formateur modifie un cursus déjà utilisé, on ne doit pas casser les cohortes en cours. Solution : snapshot par version. La cohorte référence une `cursus_version` figée. C'est la garantie de stabilité pédagogique : un stagiaire suit le cursus tel qu'au démarrage.

Valeur livrée : un formateur peut évoluer son cursus en permanence sans craindre de casser les cohortes en cours. Diff visuel entre versions.

## Description fonctionnelle

Toute modification d'un cursus publié → création d'une nouvelle `cursus_version` (snapshot JSON). La version active est celle utilisée pour les nouvelles cohortes. Les anciennes cohortes restent sur leur version d'origine. Page d'historique des versions avec diff visuel.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Nouvelle version sur édition**

```gherkin
Given un cursus en v1 est utilisé par 2 cohortes
When le formateur édite le cursus et publie une nouvelle version
Then v2 est créée comme nouveau snapshot
And les 2 cohortes restent sur v1
And les nouvelles cohortes prennent v2 par défaut
```

**Scénario 2 — Diff visuel**

```gherkin
Given le formateur ouvre l'historique des versions
When il sélectionne v1 et v2
Then un diff visuel des modifications s'affiche (texte ajouté/supprimé, modules ajoutés/supprimés)
```

**Scénario 3 — Isolation entre cohortes**

```gherkin
Given un module est ajouté dans v2
When une cohorte v1 est en cours
Then les stagiaires de cette cohorte ne voient pas le nouveau module
```

## Cas limites à traiter

- **Rollback** : possibilité de "republier v1" si v2 a un bug critique (crée v3 = copie de v1).
- **Limite** : 50 versions max par cursus (warning au-delà, optimisation v1.x).
- **Modification mineure** (typo) : optionnellement non-versionnée (toggle "modification éditoriale").
- **Cohorte vide v1** : peut migrer manuellement vers v2.

## Non-goals (hors scope)

- Pas de merge entre versions (one-way only).
- Pas de branches de cursus.
- Pas de migration automatique d'une cohorte v∧→v∨.

## Tests à écrire

- **Integration** : modification + publish → nouvelle version créée en DB.
- **E2E (Playwright)** : diff de versions affiche les bons changements.
- **Unit** : algo de diff de modules.
- **Edge** : 50 versions, performance acceptable.

## Observabilité

- **Logs** : `cursus.version.created`.
- **Métriques** : nombre moyen de versions par cursus, distribution.

## Considérations sécurité

- Versions = immuables (RLS policy `false` pour UPDATE/DELETE).
- Snapshot contient pas de PII (juste données pédagogiques).

## Considérations accessibilité

- Diff visuel : couleurs ET symboles (+/-) pour daltoniens.
- Navigation clavier dans le diff.

## Considérations performance

- Stockage : snapshot compressé (gzip JSONB Postgres natif).
- Diff : calculé à la demande (pas stocké).

## Definition of Done spécifique

- Modifier cursus + publish → v2 visible.
- Diff v1/v2 montre les changements correctement.
- Cohorte v1 ne voit pas les modifs v2.
