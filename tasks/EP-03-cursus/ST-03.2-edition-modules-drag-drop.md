## Contexte business

Le formateur ajoute des modules (généralement 1 par semaine) au cursus. Chaque module : titre, semaine N, objectifs en markdown, ressources, livrable, quiz, badge. **L'ordre est important** — déplacer un module change la cadence pédagogique. Drag-and-drop est l'UX standard et accélère la création.

Valeur livrée : assemblage rapide d'un cursus (un module édité en moins de 2 min), réordonnancement fluide, navigation clavier pour a11y.

## Description fonctionnelle

Vue d'édition liste avec drag-and-drop (vue-draggable-next) pour réordonner. Chaque ligne expandable pour éditer les détails du module : titre, semaine, objectifs (markdown avec preview), ressources, livrable, quiz, badge. Boutons "monter" / "descendre" en complément pour a11y. Auto-save toutes les N secondes.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Ajout d'un module**

```gherkin
Given un formateur édite un cursus
When il clique "Ajouter un module"
Then une ligne est ajoutée à la fin de la liste
And la semaine est auto-incrémentée
And le focus passe sur le champ titre
```

**Scénario 2 — Réordonnancement drag-and-drop**

```gherkin
Given des modules existent
When le formateur drag-and-drop pour réordonner
Then l'ordre est persisté (PATCH bulk vers l'API)
And les numéros de semaine sont recalculés cohérement (option : "renuméroter" ou "déplacer en gardant la semaine")
```

**Scénario 3 — Édition détaillée**

```gherkin
Given le formateur clique sur l'icône d'expansion d'un module
When la vue détaillée s'ouvre
Then il peut éditer titre, objectifs (markdown avec preview), ressources, livrable, quiz, badge
And les modifs sont auto-sauvegardées toutes les N secondes
```

## Cas limites à traiter

- **50+ modules** : performance virtualization sur la liste (vue-virtual-scroller).
- **Drag-and-drop accessible au clavier** : boutons "monter" / "descendre" en alternative.
- **Modifications non sauvegardées** : indicateur visuel + warning au quit (`beforeunload`).
- **Conflit d'édition multi-utilisateurs** (deux formateurs en même temps) : last-write-wins + warning (lock pessimiste évalué en v1.x).
- **Suppression de module** : modules suivants gardent leur numéro (sauf option "compacter").

## Tests à écrire

- **E2E (Playwright)** : ajouter, éditer, réordonner, supprimer modules.
- **Unit** : algorithme de renumérotation des semaines.
- **A11y (axe-core)** : navigation clavier complète (Tab + Alt+↑/↓).
- **Integration** : auto-save persiste correctement (debounce + rollback en erreur).
- **Stress** : 50 modules en édition simultanée.

## Observabilité

- **Logs** : `module.created`, `module.reordered`, `module.deleted`.
- **Métriques** : `modules.created.total`, temps moyen pour assembler 1 module (cible < 2 min).

## Considérations sécurité

- RLS : formateur uniquement peut modifier ses modules.
- Sanitize markdown des objectifs (DOMPurify).
- Validation Zod des payloads (titre 1-100 char, semaine 1-52).
- Audit log sur drag/drop pour traceability.

## Considérations accessibilité

- **Drag-and-drop ARIA** : `aria-grabbed`, `aria-dropeffect`, announcements live.
- **Alternative clavier** : Alt+↑/↓ + boutons "monter"/"descendre" visibles.
- Focus management : focus reste sur le module déplacé après reorder.
- Prévisualisation markdown : contraste OK.

## Considérations performance

- Liste : virtualization si > 20 modules.
- Drag : debounce 100ms sur la prévisualisation visuelle.
- Auto-save : batch les modifs (1 call HTTP / s max).

## Definition of Done spécifique

- E2E parcours complet sur 10 modules.
- A11y axe-core sur ModuleList : 0 erreur critique.
- Test stress avec 50 modules : pas de lag perceptible.
