## Contexte business

Pour chaque module, le formateur définit ce que le stagiaire doit livrer (texte Markdown) **ET** la liste de checks automatiques que le harnais effectuera. C'est le **contrat pédagogique** : sans cette spécification, le harnais n'a rien à vérifier et le système s'effondre.

Valeur livrée : assemblage rapide d'un livrable (description + checks) avec prévisualisation du YAML GitHub Actions généré. Le formateur a la maitrise complète du contrat.

## Description fonctionnelle

Sous-section du `ModuleEditor`. Description markdown du livrable avec preview live. Liste de checks paramétrables (cases à cocher avec champs de config). Bouton "Prévisualiser le workflow" affiche le YAML GitHub Actions généré. Sauvegarde en JSONB dans `modules.deliverable_spec`.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Édition livrable + checks**

```gherkin
Given un formateur édite un module
When il atteint la section "Livrable"
Then il voit un éditeur markdown avec preview live
And en dessous, une liste de checks à activer
And chaque check expand pour montrer ses paramètres
```

**Scénario 2 — Configuration d'un check paramétré**

```gherkin
Given le formateur active le check "Branches"
When il configure les branches attendues : "main, feature/login, feature/footer"
Then le check est listé avec ses paramètres
And la validation Zod confirme que les params sont valides
```

**Scénario 3 — Prévisualisation YAML**

```gherkin
Given tous les critères sont définis
When le formateur clique sur "Prévisualiser le workflow"
Then le YAML GitHub Actions généré s'affiche en lecture seule (highlight)
And des warnings éventuels s'affichent ("Pas de check de tests, OK ?")
```

**Scénario 4 — Déclenchement effectif**

```gherkin
Given le module est sauvegardé
When une cohorte démarre dessus
Then le workflow GH Actions sera déclenché à chaque soumission avec ces checks
```

## Cas limites à traiter

- **Aucun check activé** : warning fort "Le harnais ne pourra rien valider, soumissions auto-validées".
- **Check avec paramètres invalides** : message d'erreur précis (Zod validation).
- **Conflit entre checks** (ex : `linter_pass` mais pas de fichier ESLint configuré) : ignoré au MVP (à investiguer en v1.x via pre-flight check).
- **Trop de checks (>15)** : warning sur le temps de run.
- **Markdown malicieux** : sanitize pour preview (DOMPurify).

## Non-goals (hors scope)

- Pas de check custom (script GitHub Actions ad-hoc) au MVP — vient en v1.1.
- Pas de simulation / dry-run au MVP.
- Pas d'import d'un fichier YAML existant.
- Pas de templates de livrables prêrêrêts.

## Tests à écrire

- **Unit** : génération YAML à partir d'une config (cas 0 check, 1 check, N checks).
- **Property-based** : générer aléatoirement des configs et vérifier que le YAML produit est valide YAML.
- **E2E (Playwright)** : configurer un livrable, le sauver, vérifier que le YAML est cohérent.
- **Unit** : validation Zod sur chaque check paramétré.
- **Integration** : sauvegarde JSONB et lecture cohérente.

## Observabilité

- **Logs** : `deliverable_spec.saved`, `yaml.preview_generated`.
- **Métriques** : nombre moyen de checks activés par module (cible : 5-9), distribution des checks utilisés.

## Considérations sécurité

- Markdown sanitize avant preview et avant stockage.
- Pas d'injection YAML : génération via une bibliothèque (js-yaml) avec quote strict.
- Pas de variable d'env du formateur dans le YAML (pas de leak de secrets).
- Validation stricte des params (Zod : pas de chemin `/etc/passwd`).

## Considérations accessibilité

- Markdown editor : labels et toolbar accessibles clavier.
- Preview YAML : `role="region"`, `aria-label`.
- Warnings : `aria-live="polite"`.

## Considérations performance

- Prévisualisation YAML : génération < 100ms.
- Markdown preview : debounce 200ms.

## Definition of Done spécifique

- E2E : configurer 5 checks, voir YAML valide.
- Property test : 100 configs aléatoires → 100 YAML valides.
- Audit a11y axe-core : 0 erreur critique.
