## Contexte business

Avant de publier un cursus, le formateur veut voir ce que verra le stagiaire pour valider l'expérience. Sans prévisualisation, il publie à l'aveugle et découvre les problèmes via les stagiaires.

Valeur livrée : confiance du formateur avant publication, réduction du délai entre création et publication.

## Description fonctionnelle

Bouton "Aperçu stagiaire" qui ouvre une vue lecture seule reproduisant l'UI stagiaire, avec possibilité de simuler "Semaine N". Bandeau d'avertissement "Mode aperçu". Aucune écriture en DB.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Accès à l'aperçu**

```gherkin
Given un formateur est sur un cursus
When il clique "Aperçu stagiaire"
Then une nouvelle vue s'ouvre avec un bandeau "Mode aperçu — vous n'êtes pas connecté en stagiaire"
And il peut naviguer comme un stagiaire (sans soumettre)
And il peut sélectionner "Voir semaine X" pour simuler la progression
```

**Scénario 2 — Actions désactivées**

```gherkin
Given le formateur est en mode aperçu
When il essaie de cliquer sur "Soumettre" un livrable
Then le bouton est désactivé ou affiche "Action non disponible en mode aperçu"
And aucune donnée n'est créée en DB
```

**Scénario 3 — Quiz visible mais non validable**

```gherkin
Given un module contient un quiz
When le formateur accède au quiz en mode aperçu
Then il peut le voir mais pas le soumettre
And les bonnes réponses sont visibles (vue formateur)
```

## Cas limites à traiter

- **Module avec livrable obligatoire** : afficher l'UI sans bouton soumettre.
- **Quiz** : affichable mais pas validable (overlay "Mode aperçu").
- **Semaine au-delà du nombre de modules** : message "Plus de modules".
- **Sortie de l'aperçu** : retour direct à l'édition.

## Dépendances

- **Bloqué par** : ST-03.1, ST-03.2, ST-05.1 (la page "Cette semaine" doit exister).
- **Bloque** : Aucune.

## Non-goals (hors scope)

- Pas de simulation d'un stagiaire complet (avec état "déjà validé").
- Pas de simulation de soumission ni de quiz (juste affichage).
- Pas de partage de l'aperçu via URL publique.

## Tests à écrire

- **E2E (Playwright)** : entrée + sortie de l'aperçu.
- **Security** : aperçu ne crée AUCUNE donnée en DB (tests avec spy).
- **Unit** : composant wrapper désactive bien les boutons.

## Observabilité

- **Logs** : `cursus.preview_opened`.
- **Métriques** : `cursus.preview_used.total`, durée moyenne en aperçu.

## Considérations sécurité

- Endpoint preview = pas d'UPDATE/INSERT possible (read-only DB session).
- Aperçu réservé au formateur du cursus (RLS).
- Pas de fuite de données entre cohortes (aucune donnée stagiaire affichée).

## Considérations accessibilité

- Bandeau "Mode aperçu" : `role="alert"`, visible et annoncé par lecteur d'écran.
- Boutons désactivés : `aria-disabled="true"` + tooltip explicite.

## Considérations performance

- Même perf que vue stagiaire (LCP < 1.5s).
- Pas de re-render entier sur changement de semaine.

## Definition of Done spécifique

- E2E : entrée + navigation 3 semaines + sortie.
- Test négatif : aucune row créée en DB pendant aperçu.
- Bandeau bien visible et accessible.
