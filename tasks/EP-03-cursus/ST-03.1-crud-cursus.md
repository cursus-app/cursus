## Contexte business

Un cursus est l'unité pédagogique racine du produit : il contient des modules hebdomadaires, des livrables, des quiz, des badges. C'est l'investissement initial du formateur — si la création est trop lourde, l'app ne sera pas adoptée. Sans cursus, pas de cohorte ; sans cohorte, pas de stagiaire.

Valeur livrée : un formateur crée un cursus en brouillon, le travaille tranquillement, le publie quand prêt. Cycle de vie clair : brouillon → publié → archivé, sans casser les cohortes en cours.

## Description fonctionnelle

Un cursus est défini par : nom, domaine (Dev Web, Ingénierie Web, IA, Cybersec, autre), niveau (L1/L2/L3/M1/M2/Pro), durée prévue en semaines, prérequis (markdown). Statuts : `brouillon`, `publié`, `archivé`. Page `/cursus` avec liste et filtres. Pages `/cursus/new`, `/cursus/:id/edit`, `/cursus/:id`. Suppression refusée si utilisé par une cohorte active.

## Critères d'acceptation (Gherkin)

**Scénario 1 — Création d'un brouillon**

```gherkin
Given un formateur est sur /cursus
When il clique sur "Nouveau cursus"
Then un brouillon est créé avec valeurs par défaut
And il est redirigé sur la page d'édition
```

**Scénario 2 — Publication nécessite ≥ 1 module**

```gherkin
Given le formateur clique "Publier"
When le cursus a au moins 1 module et la description est remplie
Then le cursus passe en statut "publié"
And une cursus_version v1 est figée (snapshot)
Else un message indique les pré-requis manquants
```

**Scénario 3 — Suppression bloquée si cohortes actives**

```gherkin
Given un cursus publié est utilisé par 2 cohortes
When le formateur essaie de le supprimer
Then la suppression est refusée avec un message "Ce cursus est utilisé par 2 cohortes actives"
And l'option "Archiver" est proposée à la place
```

## Cas limites à traiter

- **Tentative de publication sans module** : refus + message clair.
- **Nom de cursus dupliqué** dans l'organisation : autorisé mais warning visuel.
- **Caractères spéciaux dans le nom** : autorisés (slug calculé séparément, sanitisé).
- **Édition d'un cursus publié** : avertissement "Une nouvelle version sera créée" (lié à ST-03.5).
- **Archivage** : les cohortes en cours gardent l'accès lecture seule.

## Tests à écrire

- **Unit** : transitions de statut (brouillon → publié → archivé, refus invalides).
- **Unit** : validation Zod du payload cursus.
- **E2E (Playwright)** : créer, éditer, publier, archiver.
- **Security** : un stagiaire ne peut pas créer/éditer un cursus (RLS test).
- **Integration** : tentative de delete sur cursus utilisé → 409 Conflict.

## Observabilité

- **Logs** : `cursus.created`, `cursus.published`, `cursus.archived`, `cursus.delete_blocked`.
- **Métriques** : `cursus.created.total`, `cursus.published.total`, temps moyen création (brouillon → publish).
- **Alertes** : aucune spécifique.

## Considérations sécurité

- RLS : formateur crée/édite/supprime ses cursus uniquement.
- Slug : sanitize XSS (regex strict alphanum + tirets).
- Markdown des prérequis : rendering safe (markdown-it + DOMPurify).
- Audit log sur publish / archive / delete.

## Considérations accessibilité

- Filtres : labels associés, focus visible, navigation clavier.
- Page edit : autosave avec annonce `aria-live`.
- Liste : pagination ou virtualization si > 100 cursus.

## Considérations performance

- Liste `/cursus` : LCP < 1.5s (pré-rendu SSR + cache).
- API list : pagination 20 items, index sur `status`, `domain`, `level`.

## Definition of Done spécifique

- Cycle complet brouillon → publié → archivé fonctionne.
- Test RLS négatif : stagiaire bloqué.
- E2E parcours formateur complet.
